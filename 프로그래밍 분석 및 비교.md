# 1. 아키텍처 구조

**무엇을 알아야 하는가**

- 내부 동작 모델(스레드 기반, 이벤트 기반, 워커 모델, 프로세스 모델)
    
- 요청 처리 흐름(입력 → 필터 → 처리 → 출력)
    
- 구성요소(프록시, 라우터, 필터, 모듈, 플러그인 등)
    

**왜 중요한가**

- 구조가 곧 성능이고,
    
- 구조가 곧 선택 기준이다.  
    예: Nginx(이벤트 루프) vs HAProxy(멀티스레드 기반) vs Traefik(Go 기반 이벤트 루프)
    

**여기서 공부해야 할 키워드**

- Event loop, thread-per-core, worker process
    
- pipeline 구조
    
- 비동기/동기 모델
    

---

# 2. 기능 범위(Feature Scope)

**도구가 어떤 문제를 해결하려고 하는가?**

- L4/L7 지원 여부
    
- 라우팅 규칙 지원 범위
    
- 인증/보안 기능
    
- 로깅/모니터링 내장 여부
    
- 프로토콜 지원(HTTP/2, gRPC, WebSocket 등)
    

**필터링/라우팅 도구면:**

- 라우팅 규칙
    
- 헬스체크 방식
    
- SSL/TLS termination
    
- Sticky session 여부
    

**메시지 브로커라면:**

- QoS
    
- Consumer model
    
- Partition/rebalance 방식
    

이미 이 레벨에서 대부분의 도구 차이가 드러난다.

---

# 3. 구성 난이도 & 운영 편의성

**반드시 공부할 포인트**

- 설정 파일 구조(Nginx: 블록 기반, HAProxy: 단일 파일)
    
- Hot reload 가능 여부
    
- 변경 시 downtime 발생 여부
    
- 장애 원인 추적 난이도
    
- 디버깅 난이도
    

해석하자면,  
“실서비스에서 운영 가능하냐?”를 판단하는 기준.

---

# 4. 성능 특성

**이건 반드시 공통적으로 체크해야 한다.**

- 동시성 처리 방식
    
- Throughput/TPS 성향
    
- Latency 특성
    
- CPU bound인지 I/O bound인지
    
- Zero-copy / Buffering 방식
    
- 스케줄링 전략
    

예:

- Nginx는 단일 이벤트 루프라 CPU 하나 꽉 차면 끝
    
- HAProxy는 스레드 기반이라 코어 확장이 유리함
    

성능 벤치마크를 하려면 이 부분을 먼저 알아야 한다.

---

# 5. 확장성 & 생태계

**확장성 체크 항목**

- 플러그인/모듈 구조
    
- 클러스터 구성 가능 여부
    
- 서비스 디스커버리 지원 여부
    
- Config as Code 지원
    
- Kubernetes 통합성
    

여기서 도구의 수명과 미래가 갈린다.

예:  
Traefik → K8s 네이티브  
Envoy → 서비스메시 기반  
Nginx → 전통적 구조 + 모듈 생태계 강함

---

# 6. 운영 안정성과 장애 대응 모델

**공부해야 할 부분**

- 헬스체크(Active/Passive)
    
- 장애 전파 모델
    
- Failover 방식
    
- Rolling update 지원 여부
    
- 재시도 정책
    
- Circuit breaker 유무
    

서버가 늘어나면 이게 핵심이 된다.

---

# 7. 도입 목적과 가장 잘 맞는 문제 영역

도구 비교의 핵심은 “무엇이 더 좋은가?”가 아니라  
“**무엇이 어떤 문제에서 더 최적인가?**”

따라서 아래를 반드시 공부해야 한다:

- 이 도구가 설계될 때 처음 의도
    
- 강점이 드러나는 영역
    
- 약한 부분 or 희생한 부분
    
- 대체 가능한 경쟁 제품과 차이
    

예:  
    HAProxy → 고성능 L4/L7 로드밸런싱에 필살기  
    Nginx → 정적 파일, Reverse Proxy, 모듈형 라우팅  
    Traefik → 동적 구성 + K8s + 마이크로서비스  
    ALB → AWS 인프라와의 완벽한 통합  
    Envoy → 서비스메시 시대에 최적화

# 결론: 도구가 20~30개여도 비교 분석 체계를 단 6가지로 통일하라

도구 개수가 아무리 많아도 다음 6가지를 공부해두면 **비교가 자동화**된다.

1. 아키텍처 구조
    
2. 기능 범위
    
3. 구성 난이도 / 운영 편의성
    
4. 성능 특성
    
5. 확장성 / 생태계
    
6. 운영 안정성 / 장애 대응
    
7. 도입 시 가장 잘 맞는 문제 영역(부가)
    

이 7가지를 공통 축으로 잡으면  
어떤 도구(Nginx, HAProxy, Traefik, Envoy, Kafka, Redis, Keycloak, Elasticsearch…)가 와도 **비교·선택 기준이 일관적으로 유지된다.**
