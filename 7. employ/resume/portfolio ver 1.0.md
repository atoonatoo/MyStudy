## 1. 부하 테스트 설계와 도구 선정

### 1.1 version 1
프로젝트에서 제가 담당한 서비스의 안정적 동작을 검증하기 위해 부하 테스트의 필요성을 인식했습니다. 이를 위해 `k6`, `Apache JMeter`, `nGrinder`, `Locust` 등 대표적인 부하 테스트 도구를 후보로 선정하고 각 도구의 특징을 비교 및 분석 하였습니다.

처음에는 각 도구의 기능이 비슷해 보여 단순 비교로는 차이를 느끼기 어려웠습니다.

하지만 문서와 사례를 분석하며 각 도구가 지향하는 목적과 강점이 다르다는 점을 파악했습니다. 예를 들어 `k6`는 코드 기반의 시나리오 작성과 자동화에 강점을 보였고, `Locust`는 Python 언어를 활용한 유연한 부하 제어가 가능했습니다.

이러한 비교 과정을 통해 단순한 기능 나열보다 프로젝트의 부하 형태와 검증 목표에 적합한 도구를 선택해야 한다는 기준을 세울 수 있었습니다.

부하 테스트 도구들을 비교 및 분석한 끝에, 저는 최종적으로 `Apache JMeter`를 선택했습니다. 가장 큰 이유는 JMeter가 GUI에 특화된 구조를 갖추고 있으며, 풍부한 플러그인과 다양한 프로토콜을 지원하여 복잡한 시나리오도 시각적으로 쉽게 구성할 수 있기 때문입니다.

JMeter는 Java 기반 특성상 메모리 소비량이 높은 편이었는데, 이에 대한 해결책으로 공식 문서에서도 권장하는 방식인 GUI로 시나리오를 설계하고 CLI 모드로 테스트를 실행하는 방법을 적용하여 성능 문제를 완화할 수 있었습니다.

### 1.2 version 2
프로젝트에서 담당한 서비스의 안정적 동작을 검증하기 위해 부하 테스트의 필요성을 인식했습니다. 이를 위해 `k6`, `Apache JMeter`, `nGrinder`, `Locust` 등 주요 도구를 후보로 선정해 특징을 비교·분석했습니다. 처음에는 각 도구들의 기능이 비슷해 보였지만 문서와 사례를 검토하며 각 도구의 목적과 강점이 다르다는 점을 파악했습니다.

예를 들어 `k6`는 코드 기반 자동화에, `Locust`는 Python을 통한 유연한 부하 제어에 강점을 보였습니다. 이러한 과정을 통해 프로젝트의 부하 형태와 검증 목표에 적합한 도구를 선택해야 한다는 기준을 세웠습니다.

최종적으로 `Apache JMeter`를 선택했으며, GUI 기반 시나리오 구성과 다양한 프로토콜 지원으로 복잡한 테스트를 직관적으로 설계할 수 있었습니다. 또한 Java 기반 특성으로 인한 메모리 사용 문제는 GUI로 설계 후 CLI 모드 실행 방식을 적용해 완화했습니다.
