# 1. 부하 테스트 설계와 도구 선정

프로젝트에서 제가 담당한 서비스의 안정적 동작을 검증하기 위해 부하 테스트의 필요성을 인식했습니다. 이를 위해 `k6`, `Apache JMeter`, `nGrinder`, `Locust` 등 대표적인 부하 테스트 도구를 후보로 선정하고, 공식 문서를 기반으로 각 도구의 특징을 비교 및 분석 하였습니다.

`k6` 는 개발자 친화적인 테스트 도구로서 가장 효율적이고 높은 성능을 보장하는 도구로 이해했습니다. k6는 Go 기반의 엔진을 통해 대규모 트래픽 시나리오에서 매우 안정적이고 효율적인 처리가 가능합니다. 이를 Apache Jmeter와 비교해보았습니다.

`Apache JMeter`는 Java 기반으로 개발된 도구이기 때문에 JVM 위에서 실행됩니다. 이러한 특성상 소규모 테스트에서는 문제가 없지만, 대규모 부하 테스트에서는 구조적으로 비효율적인 면이 있습니다.

그 이유는 Java가 OS 스레드(약 512KB~1MB의 고정 스택 메모리) 를 직접 사용하는 구조이기 때문입니다. 스레드가 생성될 때마다 일정한 메모리 공간이 예약되며, 요청의 크기와 관계없이 이 스택 크기가 고정되어 있어 메모리 사용이 비효율적입니다.

또한 JMeter는 요청이 완료되어도 스레드를 즉시 반환하지 않고 세션을 유지하는 구조를 가지고 있습니다. 예를 들어 2만 명의 사용자가 5분간 요청을 분산해 보내는 시나리오를 실행하더라도, 각 스레드는 요청이 끝난 뒤에도 해제되지 않고 유지됩니다. 반면 `k6`나 `Locust`와 같은 다른 도구들은 요청이 종료되면 스레드를 반환하기 때문에 메모리 효율이 훨씬 높습니다.

반면 `k6`는 Go 언어로 개발된 엔진을 기반으로 하며, Go의 **Goroutine**이 제공하는 경량 스레드(약 2KB부터 시작)를 활용합니다. 이러한 구조 덕분에 `Apache JMeter`와 달리 최소 수십만에서 최대 수백만 개의 동시 스레드 실행이 가능합니다. 따라서 `k6`는 대규모 부하 테스트에 매우 유리한 성능과 확장성을 갖춘 도구입니다.
