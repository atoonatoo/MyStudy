## 1. 부하 테스트 설계와 도구 선정

프로젝트에서 제가 담당한 서비스의 안정적 동작을 검증하기 위해 부하 테스트의 필요성을 인식했습니다. 이를 위해 `k6`, `Apache JMeter`, `nGrinder`, `Locust` 등 대표적인 부하 테스트 도구를 후보로 선정하고, 공식 문서를 기반으로 각 도구의 특징을 비교 및 분석 하였습니다.

`k6` 는 개발자 친화적인 테스트 도구로서 가장 효율적이고 높은 성능을 보장하는 도구로 이해했습니다. k6는 Go 기반의 엔진을 통해 대규모 트래픽 시나리오에서 매우 안정적이고 효율적인 처리가 가능합니다. 이를 Apache Jmeter와 비교해보았습니다.

`Apache JMeter`는 Java 기반으로 개발된 도구이기 때문에 JVM 위에서 실행됩니다. 이러한 특성상 소규모 테스트에서는 문제가 없지만, 대규모 부하 테스트에서는 구조적으로 비효율적인 면이 있습니다.

그 이유는 Java가 OS 스레드(약 512KB~1MB의 고정 스택 메모리) 를 직접 사용하는 구조이기 때문입니다. 스레드가 생성될 때마다 일정한 메모리 공간이 예약되며, 요청의 크기와 관계없이 이 스택 크기가 고정되어 있어 메모리 사용이 비효율적입니다.

또한 JMeter는 요청이 완료되어도 스레드를 즉시 반환하지 않고 세션을 유지하는 구조를 가지고 있습니다. 예를 들어 2만 명의 사용자가 5분간 요청을 분산해 보내는 시나리오를 실행하더라도, 각 스레드는 요청이 끝난 뒤에도 해제되지 않고 유지됩니다. 반면 `k6`나 `Locust`와 같은 다른 도구들은 요청이 종료되면 스레드를 반환하기 때문에 메모리 효율이 훨씬 높습니다.

반면 `k6`는 Go 언어로 개발된 엔진을 기반으로 하며, Go의 **Goroutine**이 제공하는 경량 스레드(약 2KB부터 시작)를 활용합니다. 이러한 구조 덕분에 `Apache JMeter`와 달리 최소 수십만에서 최대 수백만 개의 동시 스레드 실행이 가능합니다. 따라서 `k6`는 대규모 부하 테스트에 매우 유리한 성능과 확장성을 갖춘 도구입니다.

`Apache JMeter`는 GUI 기반의 테스트 애플리케이션으로, 초보자도 쉽게 접근할 수 있는 친숙한 도구로 이해했습니다.

반면 `k6`는 GUI를 지원하지 않고, 필요한 플러그인을 직접 빌드하는 방식으로 확장합니다. `nGrinder`는 웹 UI 형태의 GUI를 제공하지만, 플러그인 개념이 거의 존재하지 않으며 확장성이 제한적입니다. `Locust`는 플러그인을 지원하지만 GUI는 제공하지 않습니다.

이와 달리 `JMeter`는 다양한 플러그인과 폭넓은 프로토콜을 기본적으로 지원하며, 테스트 완료 후 고품질의 HTML 기반 정적 리포트를 생성할 수 있습니다. 비록 실시간 모니터링 기능은 상대적으로 부족하지만, 결과 보고서의 정밀도와 시각화 완성도 면에서는 매우 우수한 도구라고 판단했습니다.

`nGrinder`는 조직이나 기업 단위의 대규모 부하 테스트를 위해 설계된 엔터프라이즈급 테스트 플랫폼으로 이해했습니다.

모니터링, 메트릭 수집, 시각화, 결과 리포팅 등 통합 기능을 제공하며, 다른 도구들과 달리 대규모 환경과 중앙 관리에 특화된 구조적 성격이 두드러집니다.

또한 스크립트의 버전 관리와 중앙화를 위해 Controller 내부에 내장형 SVN 서버(embedded SVN server) 가 포함되어 있어, 테스트 스크립트를 형상 관리용으로 사용할 수 있습니다.

`nGrinder`는 Controller–Agent 구조를 기반으로 하며, 분산 실행을 주요 설계 목표로 하고 있습니다. Agent는 여러 대의 머신에 설치되며 Controller가 이를 동적으로 할당합니다.

이러한 구조를 통해 단일 머신의 자원 한계를 넘어 부하를 분산 실행할 수 있고, 서로 다른 성격의 테스트를 동시에 수행하더라도 Agent를 다중 테스트에 병렬로 할당하여 동시·병렬 테스트 수행 능력을 확보할 수 있습니다.

`Locust`는 실제 사용자 행위를 정교하게 시뮬레이션할 수 있는 개발자 중심의 테스트 프레임워크형 도구로 이해했습니다.

코드 기반과 관리 UI의 하이브리드형 구조를 채택하여, 개발자는 Python 코드로 시나리오를 자유롭게 정의하면서도 Web UI를 통해 테스트 실행을 실시간 제어 및 모니터링할 수 있습니다.

또한 조건 분기 자유도가 높아 복잡한 사용자 흐름을 세밀하게 구성할 수 있으며, 테스트 흐름의 유연성 면에서도 강점을 보입니다. 실행은 CLI 기반 실행기를 통해 가능하고, Master–Worker 구조를 활용해 여러 머신에 부하를 분산시켜 대규모 테스트를 수행할 수 있습니다.

위와 같은 부하 테스트 도구들을 비교 및 분석한 끝에, 저는 최종적으로 `Apache JMeter`를 선택했습니다.

가장 큰 이유는 JMeter가 GUI에 특화된 구조를 갖추고 있으며, 풍부한 플러그인과 다양한 프로토콜을 지원하여 복잡한 시나리오도 시각적으로 쉽게 구성할 수 있기 때문입니다.

실제 테스트 과정에서 제가 필요하다고 판단한 기능들(`정규식 추출기`, `CSV Data Set Config`, `95% Line Report` 등)을 손쉽게 찾을 수 있었고, 설정 절차 또한 직관적이었습니다.

다만 JMeter는 Java 기반 특성상 메모리 소비량이 높은 편이었는데, 이에 대한 해결책으로 공식 문서에서 권장하는 방식인 GUI로 시나리오를 설계하고 CLI 모드로 테스트를 실행하는 방법을 적용하여 성능 문제를 완화할 수 있었습니다.
