### 2. MVP 출시 타임라인 2026-05-20

#### 1주차 : 설계 및 인프라 구축 (뼈대 잡기)
- **목표** : 데이터베이스 구조를 확정하고, 무료 서버 환경을 세팅한다.
      
- **주요 태스크**
    - **DB 모델링** : 회원(User), 공고(Job), 지원서(Application) 테이블 설계
    - **인프라 구축** : AWS 프리티어 또는 Supabase/Vercel을 활용한 개발 환경 세팅
    - **다국어(i18n) 기획** : 한국어/영어 번역 텍스트를 관리할 구조 설계
    
- **산출물** : DB ERD(다이어그램), API 명세서 초안
      
- **비고**
    - 비자(Visa) 정보는 회원가입 테이블에서 제외하고, 우선 `지원서 작성 단계`에서 텍스트로 입력받도록 DB를 설계한다. 그래야 초기 가입 허들이 낮아진다.
---
#### 2주차 : 백엔드 핵심 기능 개발 (기능 구현)
- **목표** :  회원인증 및 운영자용 공고 관리 API를 완성한다.
      
- **주요 태스크**
    - **인증(Auth) 시스템:** 이메일/비밀번호 또는 카카오/페이스북 간편 로그인 구현
    - **공고(Job) API:** 운영자가 공고를 등록, 수정, 삭제, 조회(CRUD)할 수 있는 기능 개발
    - **이미지 업로드 기능:** AWS S3 프리티어 등을 연동하여 업체 이미지 업로드 API 구현
    
- **산출물** : 백엔드 API 서버 배포 완료
---
#### 3주차 : 프론트엔드 UI/UX 및 반응형 레이아웃 (시각화)
- **목표** : 모바일(페이스북 인앱 브라우저) 화면에 최적화된 반응형 웹 스켈레톤을 만든다.
    
- **주요 태스크:**
    - **공통 컴포넌트:** 네비게이션 바, 푸터, 다국어 전환 버튼 구현
    - **반응형 레이아웃:** 테일윈드(Tailwind CSS) 등을 활용해 모바일 화면 위주로 퍼블리싱
    - **페이지 구현:** 메인 페이지(공고 리스트), 로그인/회원가입 페이지
    
- **산출물** : 프론트엔드 UI 화면 개발 완료
---
#### 4주차 : 데이터 연동 및 기능 결합 (풀스택 완성)
- **목표** : 백엔드 API와 프론트엔드 UI를 연결하여 실제 작동하는 서비스를 만든다.
    
- **주요 태스크**
    - **API 연동** :  프론트엔드에서 공고 리스트 호출 및 회원가입 기능 연결
    - **지원서 제출 시스템** :  사용자가 공고를 보고 '지원하기' 버튼을 눌렀을 때, 필수 정보(이름, 연락처 등)가 운영자 DB로 쌓이도록 연동
    - **다국어 적용** :  버튼 클릭 시 한/영 텍스트가 실시간으로 변환되도록 프론트엔드 로직 완성
    
- **산출물** :  주요 기능이 정상 작동하는 웹 어플리케이션
---
#### 5주차 : 운영자(관리자) 페이지 및 디버깅 (안정화)
- **목표:** 지인(운영자)이 사용할 관리자 화면을 만들고, 전체적인 버그를 잡는다.
    
- **주요 태스크**
    - **어드민 대시보드** : 운영자가 로그인했을 때, 지원자 리스트를 한눈에 볼 수 있는 심플한 페이지 구현
    - **크로스 브라우징 테스트** : 특히 스마트폰 페이스북 앱 내부 브라우저에서 화면이 깨지지 않는지 집중 테스트
    - **디자인 디테일 수정** : 톤앤매너(심플한 블루 컬러) 맞추기
    
- **산출물:** 관리자 기능 포함 통합 테스트 완료 버전
---
#### 6주차 : 최종 배포 및 런칭 (인수인계)
- **목표** : 상용 서버에 도메인을 연결하고 서비스를 공식 오픈한다.
    
- **주요 태스크**
    - **최종 배포** : 실서버 배포 및 도메인 연결
    - **가이드라인 문서 작성** : 운영 방법(공고 올리는 법, 지원자 확인법)을 담은 매뉴얼 작성 및 지인에게 전달
    - **마케팅 시작 연동** : 페이스북 광고/홍보 링크가 정상적으로 작동하는지 최종 확인
    
- **산출물** : 라이브 서비스 URL, 인수인계 가이드 문서

#### 백엔드

##### SpringBoot : 4.0.6
- 버전에 대한 고민은 크게 할 필요가 없었다.
- **3.5**
    - 1개월 뒤 OSS 지원이 끝난다. 따라서 신규 프로젝트에 적합하지 않다.
- 4.0은 향후 확장성과 가상 스레드 등의 성능을 모두 잡을 수 있다.
- 앞으로의 기술 지원(보안 패치 등)을 가장 길게 받을 수 있다.
- Spring Boot는 버전이 올라갈수록 오히려 설정이 간소화되고 자동화가 더 잘 돤다. 
- 3.x 버전에서 사용하던 문법 대부분이 4.0에서도 그대로 사용 가능하며, 오히려 더 깔끔한 방식으로 개선되었다.
##### Java : 4.0
- Spring Boot 4.0과 가장 궁합이 좋은 장기 지원(LTS) 버전이다.
- Java 21은 8과 17이랑 문법적으로는 큰 차이가 없으며, 오히려 `Record`, `Text Blocks`, `Switch Expressions` 같은 현대적인 기능들 덕분에 코딩량이 절반으로 줄어든다.


#### 프론트엔드

##### Thymeleaf + Tailwind
- SSR (모놀리식)
- 매우 빠르게 구현이 가능
- 데이터가 가볍다.
- CORS 에러에 대한 설정이 필요없음.
- Vercel, S3 등 따로 구축할 필요없음.
- 화면 전환이 부자연스러움
- 모바일 앱 전환시 중복 작업 부담

##### React (SPA)
- CSR (클라이언트 렌더링)
- 우수한 UI/UX 페이지
- 역할 분리가 확실하다.
- 크로스 플랫폼 연계
- 인프라 공수 증가
- FB 브라우저 리스크

##### Next.js
- SSR/Hybrid (최신 프레임워크)
- 페이스북 마케팅 최적화
    - 첫 화면은 서버에서 미리 그려서 보내주기 때문에 모바일 인앱 브라우저에서 속도가 엄청나게 빠르며, 페이스북 공유 시 링크 미리보기(OG Tag)가 완벽하게 작동.
- 가장 높은 러닝커브
    - React를 완벽히 이해한 상태에서 서버 사이드 개념까지 알아야 하므로 혼자서 6주 안에 Spring Boot와 병행하기엔 시간적 리스크가 매우 크다.
    - 기술적으로는 이 프로젝트(마케팅 위주 웹)에 가장 이상적이지만, 6주 MVP 타임라인을 맞추기엔 비현실적인 오버 엔지니어링이 될 확률이 높다.


#### DB

##### Supabase (PostgreSQL)
- 무료 티어가 매우 넉넉하고, 무엇보다 관계형 DB(SQL)이다. 나중에 B2B로 확장해도 데이터를 안전하게 지킬 수 있다.
- **데이터 안전성(ACID)** : 외국인 노동자 채용 정보는 사람과 돈이 엮인 중요한 정보이다. 데이터가 꼬이면 안 된다. 관계형 DB는 데이터의 정합성을 철저히 보장한다.
- **Spring Boot 연동** : Spring Data JPA를 사용하면, 데이터베이스 테이블 구조를 자바 코드로 깔끔하게 관리할 수 있다.
- **무료 정책:** 현재 프로젝트 규모에서 데이터가 1만 건이 넘어가도 충분히 무료 범위를 유지할 수 있다.
- **나중을 위한 포석** : 나중에 B2B 시스템으로 넘어가도 이 DB 그대로 확장할 수 있다.
- 이 조합은 실제 대기업에서 사용하는 기술 스택의 핵심이다. Firebase보다 처음 세팅은 1~2일 더 걸리지만, 나중에 운영할 때 발생할 수 있는 데이터 정합성 사고를 완벽히 막아준다.

##### Firebase(Firestore)
- 스프링부트와 조합이 매우 좋지않다.
    - 스프링부트는 데이터의 관계를 엄격히 관리하는 RDB(SQL)와 연동했을 때 가장 강력한 힘을 발휘한다.
    - Firestore의 NoSQL은 SQL의 트랜잭션, 조인(Join) 기능을 지원하지 않는다. 
    - Spring Boot의 강력한 ORM인 JPA/Hibernate를 사실상 사용할 수 없게 되어, 스프링의 큰 장점을 버리고 시작하는 꼴이 된다.

#### 중요 : 연동을 위한 백엔드(Spring Boot) 필수 준비물
- 프론트엔드가 React로 결정되었으므로, 백엔드 설계 시 다음 두 가지는 반드시 구현해야 한다.

- **CORS 설정** 
    - React(3000번 포트)와 Spring Boot(8080번 포트)는 서로 다른 서버로 인식된다. 
    - Spring Security에서 CORS를 열어주지 않으면 통신이 불가능하다.
-  **JWT 인증** 
    - 앱으로의 확장성을 고려할 때 세션(Session) 방식보다는 JWT(JSON Web Token) 방식의 토큰 인증을 구현하는 것이 향후 앱 개발 시 훨씬 유리하다.
