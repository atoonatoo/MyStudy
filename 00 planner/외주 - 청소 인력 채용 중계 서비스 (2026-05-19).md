### 2. MVP 출시 타임라인 2026-05-20

#### 1주차 : 설계 및 인프라 구축 (뼈대 잡기)
- **목표** : 데이터베이스 구조를 확정하고, 무료 서버 환경을 세팅한다.
      
- **주요 태스크**
    - **DB 모델링** : 회원(User), 공고(Job), 지원서(Application) 테이블 설계
    - **인프라 구축** : AWS 프리티어 또는 Supabase/Vercel을 활용한 개발 환경 세팅
    - **다국어(i18n) 기획** : 한국어/영어 번역 텍스트를 관리할 구조 설계
    
- **산출물** : DB ERD(다이어그램), API 명세서 초안
      
- **비고**
    - 비자(Visa) 정보는 회원가입 테이블에서 제외하고, 우선 `지원서 작성 단계`에서 텍스트로 입력받도록 DB를 설계한다. 그래야 초기 가입 허들이 낮아진다.
---
#### 2주차 : 백엔드 핵심 기능 개발 (기능 구현)
- **목표** :  회원인증 및 운영자용 공고 관리 API를 완성한다.
      
- **주요 태스크**
    - **인증(Auth) 시스템:** 이메일/비밀번호 또는 카카오/페이스북 간편 로그인 구현
    - **공고(Job) API:** 운영자가 공고를 등록, 수정, 삭제, 조회(CRUD)할 수 있는 기능 개발
    - **이미지 업로드 기능:** AWS S3 프리티어 등을 연동하여 업체 이미지 업로드 API 구현
    
- **산출물** : 백엔드 API 서버 배포 완료
---
#### 3주차 : 프론트엔드 UI/UX 및 반응형 레이아웃 (시각화)
- **목표** : 모바일(페이스북 인앱 브라우저) 화면에 최적화된 반응형 웹 스켈레톤을 만든다.
    
- **주요 태스크:**
    - **공통 컴포넌트:** 네비게이션 바, 푸터, 다국어 전환 버튼 구현
    - **반응형 레이아웃:** 테일윈드(Tailwind CSS) 등을 활용해 모바일 화면 위주로 퍼블리싱
    - **페이지 구현:** 메인 페이지(공고 리스트), 로그인/회원가입 페이지
    
- **산출물** : 프론트엔드 UI 화면 개발 완료
---
#### 4주차 : 데이터 연동 및 기능 결합 (풀스택 완성)
- **목표** : 백엔드 API와 프론트엔드 UI를 연결하여 실제 작동하는 서비스를 만든다.
    
- **주요 태스크**
    - **API 연동** :  프론트엔드에서 공고 리스트 호출 및 회원가입 기능 연결
    - **지원서 제출 시스템** :  사용자가 공고를 보고 '지원하기' 버튼을 눌렀을 때, 필수 정보(이름, 연락처 등)가 운영자 DB로 쌓이도록 연동
    - **다국어 적용** :  버튼 클릭 시 한/영 텍스트가 실시간으로 변환되도록 프론트엔드 로직 완성
    
- **산출물** :  주요 기능이 정상 작동하는 웹 어플리케이션
---
#### 5주차 : 운영자(관리자) 페이지 및 디버깅 (안정화)
- **목표:** 지인(운영자)이 사용할 관리자 화면을 만들고, 전체적인 버그를 잡는다.
    
- **주요 태스크**
    - **어드민 대시보드** : 운영자가 로그인했을 때, 지원자 리스트를 한눈에 볼 수 있는 심플한 페이지 구현
    - **크로스 브라우징 테스트** : 특히 스마트폰 페이스북 앱 내부 브라우저에서 화면이 깨지지 않는지 집중 테스트
    - **디자인 디테일 수정** : 톤앤매너(심플한 블루 컬러) 맞추기
    
- **산출물:** 관리자 기능 포함 통합 테스트 완료 버전
---
#### 6주차 : 최종 배포 및 런칭 (인수인계)
- **목표** : 상용 서버에 도메인을 연결하고 서비스를 공식 오픈한다.
    
- **주요 태스크**
    - **최종 배포** : 실서버 배포 및 도메인 연결
    - **가이드라인 문서 작성** : 운영 방법(공고 올리는 법, 지원자 확인법)을 담은 매뉴얼 작성 및 지인에게 전달
    - **마케팅 시작 연동** : 페이스북 광고/홍보 링크가 정상적으로 작동하는지 최종 확인
    
- **산출물** : 라이브 서비스 URL, 인수인계 가이드 문서

#### 이후 스케일업 단계 (출시 후 2달간의 무료 유지보수 기간)
이 기간에는 새로운 기능을 무리하게 추가하기보다, **안정성 확보**와 **데이터 수집**에 집중해야 한다.

### 3. 기술 스택 선정 2026-05-21

#### 최종 기술 스택
##### Framework : React (Vite 기반 빌드)

- **Thymeleaf + Tailwind**
    - SSR (모놀리식)
    - 매우 빠르게 구현이 가능
    - 데이터가 가볍다.
    - CORS 에러에 대한 설정이 필요없음.
    - Vercel, S3 등 따로 구축할 필요없음.
    - 화면 전환이 부자연스러움
    - 모바일 앱 전환시 중복 작업 부담
      
- **React (SPA)**
    - CSR (클라이언트 렌더링)
    - 우수한 UI/UX 페이지
    - 역할 분리가 확실하다.
    - 크로스 플랫폼 연계
    - 인프라 공수 증가
    - FB 브라우저 리스크
    
- **Next.js**
    - SSR/Hybrid (최신 프레임워크)
    - **페이스북 마케팅 최적화**
        - 첫 화면은 서버에서 미리 그려서 보내주기 때문에 모바일 인앱 브라우저에서 속도가 엄청나게 빠르며, 페이스북 공유 시 링크 미리보기(OG Tag)가 완벽하게 작동.
    - **가장 높은 러닝커브**
        - React를 완벽히 이해한 상태에서 서버 사이드 개념까지 알아야 하므로 혼자서 6주 안에 Spring Boot와 병행하기엔 시간적 리스크가 매우 크다.
    - 기술적으로는 이 프로젝트(마케팅 위주 웹)에 가장 이상적이지만, 6주 MVP 타임라인을 맞추기엔 비현실적인 오버 엔지니어링이 될 확률이 높다.

---

##### Backend :  Java 21 (LTS), Spring Boot (4.0.6)
- **SpringBoot : 4.0.6**
    - 버전에 대한 고민은 크게 할 필요가 없었다. 
    - 3.5버전은 1개월 뒤 OSS 지원이 끝난다. 따라서 신규 프로젝트에 적합하지 않다.
    - 4.0은 향후 확장성과 가상 스레드 등의 성능을 모두 잡을 수 있다.
    - 앞으로의 기술 지원(보안 패치 등)을 가장 길게 받을 수 있다.
    - Spring Boot는 버전이 올라갈수록 오히려 설정이 간소화되고 자동화가 더 잘 돤다. 
    - 3.x 버전에서 사용하던 문법 대부분이 4.0에서도 그대로 사용 가능하며, 오히려 더 깔끔한 방식으로 개선되었다.
      
- **Java : 4.0**
    - Spring Boot 4.0과 가장 궁합이 좋은 장기 지원(LTS) 버전이다.
    - Java 21은 8과 17이랑 문법적으로는 큰 차이가 없으며, 오히려 `Record`, `Text Blocks`, `Switch Expressions` 같은 현대적인 기능들 덕분에 코딩량이 절반으로 줄어든다.
      
##### State Management : Zustand
##### Communication : Axios
##### Database : Supabase (PostgreSQL)

- **Supabase (PostgreSQL)**
    - 무료 티어가 매우 넉넉하고, 무엇보다 관계형 DB(SQL)이다. 나중에 B2B로 확장해도 데이터를 안전하게 지킬 수 있다.
    - **데이터 안전성(ACID)** : 외국인 노동자 채용 정보는 사람과 돈이 엮인 중요한 정보이다. 데이터가 꼬이면 안 된다. 관계형 DB는 데이터의 정합성을 철저히 보장한다.
    - **Spring Boot 연동** : Spring Data JPA를 사용하면, 데이터베이스 테이블 구조를 자바 코드로 깔끔하게 관리할 수 있다.
    - **무료 정책:** 현재 프로젝트 규모에서 데이터가 1만 건이 넘어가도 충분히 무료 범위를 유지할 수 있다.
    - **나중을 위한 포석** : 나중에 B2B 시스템으로 넘어가도 이 DB 그대로 확장할 수 있다.

### 5. API 명세서 2026-05-22

- 초기 콘셉트(개인용 구인 공고 서비스) **핵심 API 명세서**
- 사용자(구직자)와 운영자(관리자)의 권한을 명확히 분리하고, 직관적인 HTTP 메서드와 URL 구조로 설계했다.
- 보안 요구사항에 따라 일반 사용자는 본인의 프로필 이미지만 조회 가능하며, 타인의 이미지는 볼 수 없다. 오직 운영자(어드민)만 지원자 목록 조회를 통해 지원자의 프로필 이미지를 확인할 수 있도록 권한 제어 로직을 백엔드(Spring Security)단에 적용해야 한다.

#### 인증 및 회원 관리 API (Auth/User)

회원가입 시 프로필 이미지를 필수로 S3에 먼저 업로드한 뒤, 그 URL을 포함하여 가입을 요청한다.

| **기능명**        | **HTTP 메서드** | **엔드포인트 (URI)**       | **권한** | **주요 요청 데이터 (Request)**                       | **주요 응답 데이터 (Response)**                | **설명 및 보안 규칙**                    |
| -------------- | ------------ | --------------------- | ------ | --------------------------------------------- | --------------------------------------- | --------------------------------- |
| **이미지 사전 업로드** | POST         | /api/v1/assets/upload | 전체     | Multipart File (이미지)                          | s3_url                                  | 가입/공고 등록 전 S3에 이미지를 올리는 공통 API    |
| **회원가입**       | POST         | /api/v1/auth/signup   | 전체     | 이메일, 비밀번호, 이름, 연락처, 나이, 성별, profile_image_url | 성공 메시지 또는 가입 정보                         | 유저 회원가입 처리 (비밀번호 암호화 필수)          |
| **로그인**        | POST         | /api/v1/auth/login    | 전체     | 이메일, 비밀번호                                     | AccessToken (JWT)                       | 인증 성공 시 JWT 토큰 발급                 |
| **내 정보 조회**    | GET          | /api/v1/users/me      | 회원     | 없음 (JWT 토큰으로 식별)                              | 이메일, 이름, 연락처, 나이, 성별, profile_image_url | 본인만 호출 가능. 내 프로필 이미지를 화면에 띄울 때 사용 |

#### 구인 공고 API (Jobs)

운영자만 공고를 작성/관리할 수 있으며, 일반 회원은 목록과 상세 내용을 날짜별로 조회만 할 수 있다.

| **기능명**      | **HTTP 메서드** | **엔드포인트 (URI)**             | **권한** | **주요 요청 데이터 (Request)**           | **주요 응답 데이터 (Response)** | **설명 및 보안 규칙**                     |
| ------------ | ------------ | --------------------------- | ------ | --------------------------------- | ------------------------ | ---------------------------------- |
| **공고 등록**    | POST         | /api/v1/jobs                | 운영자    | 제목, 내용, work_date, image_url (선택) | 생성된 공고 정보                | 운영자만 접근 가능 (Spring Security 권한 체크) |
| **공고 목록 조회** | GET          | /api/v1/jobs                | 전체     | 페이징 파라미터, work_date (필터용)         | 공고 리스트 (제목, 날짜, 상태 등)    | 사용자가 메인 화면에서 날짜별 공고를 조회할 때 사용      |
| **공고 상세 조회** | GET          | /api/v1/jobs/{jobId}        | 전체     | 없음                                | 공고 상세 내용, 장소 이미지 URL, 상태 | 특정 공고의 세부 업무 요건을 확인할 때 사용          |
| **공고 마감 처리** | PATCH        | /api/v1/jobs/{jobId}/status | 운영자    | status (CLOSED)                   | 변경된 공고 정보                | 운영자가 해당 날짜의 인력 모집을 수동 마감할 때 사용     |

#### 지원 및 관리자 조회 API (Applications)

가장 중요한 보안 구역이다. 지원자 목록 조회 시 회원 테이블(Users)과 조인하여 이름, 연락처, 비자 정보와 함께 프로필 이미지를 반환하지만, 이 API 자체를 오직 운영자만 호출할 수 있게 잠궈야 한다.

| **기능명**           | **HTTP 메서드** | **엔드포인트 (URI)**                         | **권한** | **주요 요청 데이터 (Request)** | **주요 응답 데이터 (Response)**                                               | **설명 및 보안 규칙**                                              |
| ----------------- | ------------ | --------------------------------------- | ------ | ----------------------- | ---------------------------------------------------------------------- | ----------------------------------------------------------- |
| **공고 지원하기**       | POST         | /api/v1/applications                    | 회원     | job_id, visa_status     | 생성된 지원 번호                                                              | 회원이 특정 공고에 지원(로그인 토큰 기반 회원 번호 추출)                           |
| **공고별 지원자 목록 조회** | GET          | /api/v1/admin/jobs/{jobId}/applications | 운영자    | 없음                      | 지원서 ID, 지원 일시, visa_status, 지원자 정보(이름, 연락처, 나이, 성별, profile_image_url) | 운영자 전용. 일반 회원이 이 URI를 요청하면 403 Forbidden 에러를 반환하여 이미지 유출 차단 |

#### 백엔드 보안 구현 가이드 (Spring Security)

- 요청하신 프로필 이미지 보안을 완벽하게 지키기 위해, 자바 스프링 부트 개발 시 아래 두 가지 규칙을 명확히 코딩해야 한다.

1. **URL 패턴 기반 권한 분리:**
    
    스프링 시큐리티 설정 클래스(SecurityFilterChain)에서 어드민 전용 경로를 명확하게 제한한다.
    
```java
    http.authorizeHttpRequests(auth -> auth
        .requestMatchers("/api/v1/admin/**").hasRole("ADMIN") // 운영자만 접근 가능
        .requestMatchers("/api/v1/users/me").authenticated()   // 본인만 접근 가능
        .anyRequest().permitAll()
    );
    
```
    
2.  **데이터 노출 최소화:**
    일반 사용자가 조회하는 '공고 상세 조회' 혹은 '공고 목록 조회' 응답 객체(DTO)에는 **지원자들의 ID나 프로필 이미지 경로 컬럼을 아예 포함하지 않도록** 응답 객체를 분리 설계해야 안전하다.

#### 중요 : 연동을 위한 백엔드(Spring Boot) 필수 준비물
- 프론트엔드가 React로 결정되었으므로, 백엔드 설계 시 다음 두 가지는 반드시 구현해야 한다.

- **CORS 설정** 
    - React(3000번 포트)와 Spring Boot(8080번 포트)는 서로 다른 서버로 인식된다. 
    - Spring Security에서 CORS를 열어주지 않으면 통신이 불가능하다.
-  **JWT 인증** 
    - 앱으로의 확장성을 고려할 때 세션(Session) 방식보다는 JWT(JSON Web Token) 방식의 토큰 인증을 구현하는 것이 향후 앱 개발 시 훨씬 유리하다.


- **이미지 처리 최적화**
    - 문제 : 지원자들의 프로필 사진은 용량이 클 수 있다. 그대로 S3에 올리면 비용도 나가고 모바일 로딩 속도가 느려진다.
    - 해결 : 프론트엔드에서 업로드 전 이미지 리사이징(압축) 처리를 하는 라이브러리(예: `browser-image-compression`)를 추가하는 것을 가이드라인에 포함해야 한다.
      
- **환경 변수 관리 (Security)**
    - **문제:** JWT Secret Key, DB 접속 정보, S3 접근 키 등을 소스코드에 그대로 올리면 보안 사고가 발생합니다.
    - **해결:** `.env` 파일을 활용하고, 배포 환경의 'Environment Variables' 설정 기능을 반드시 사용해야 합니다.

### 5. 학습 자료

#### BE과 FE는 왜 따로 배포해야하는가?

- **기술적 성격이 다르다.**
    - React(프론트)와 Spring Boot(백엔드)가 서로 다른 기술적 성격을 가지고 있기 때문이다.
        
    - **프론트엔드 (React):** 사용자의 브라우저에서 실행되는 정적 파일(HTML, CSS, JS)의 집합이다. 서버의 CPU나 메모리 연산이 필요 없고, 그냥 어디든 올려두면 사용자가 다운로드해서 보여주기만 하면 된다. (마치 웹사이트 호스팅과 같다.)
        
    - **백엔드 (Spring Boot):** 서버의 CPU와 메모리를 사용하여 연산을 수행하고, DB와 소통하며, API를 처리하는 실행 중인 애플리케이션(Runtime)이다. (마치 24시간 돌아가는 공장과 같다.)
        
    - **따라서 배포 전략이 완전히 다르다 !**
        
    - 프론트는 CDN(Content Delivery Network)이나 정적 호스팅 서비스에 올려서 전 세계 어디서나 빠르게 파일을 내려받게 해야 한다.
        
    - 백엔드는 애플리케이션 서버에 올려서 자바 가상 머신(JVM)을 띄우고 API 요청을 기다리게 해야 한다.
    
- **배포 도구에 대한 오해 바로잡기**
