

## 데드락 실제 사례 조사

#### 운영체제에서의 데드락
##### 파일과 프린터 자원 충돌
- 프로세스 A는 파일을 점유한 후 프린터를 요청
- 프로세스 B는 프린터를 점유한 후 파일을 요청
- 서로가 필요한 자원을 상대가 점유하고 있어, 무한 대기 상태 발생
##### 시스템 콜 간 커널 락 충돌
- 두 커널 스레드가 read()와 write() 시스템 콜을 처리 중
- 락 획득 순서가 어긋나면서 커널 내부에서 데드락 발생 가능성 존재
---
#### 데이터베이스 트랜잭션 충돌 사례
##### 교차 테이블 접근
- 트랜잭션 A : 테이블 X의 행을 점유한 후 테이블 Y 접근 시도
- 트랜잭션 B : 테이블 Y의 행을 점유한 후 테이블 X 접근 시도
- 교차 잠금 발생 -> 트랜잭션 간 순환 대기 -> 데드락
##### 외부 I/O 대기 포함 트랜잭션 
- 트랜잭션이 자원을 점유한 상태로 외부 API 응답을 기다리는 동안, 다른 트랜잭션이 해당 자원을 요청하며 시스템 전체가 교착 상태에 빠질 수 있음.
##### 참고
- 주요 RDBMS는 데드락을 감지하여 자동으로 트랜잭션을 강제 종료(Rollback)한다.

---
#### 멀티 스레드 환경에서의 데드락
##### 락 순서 충돌
- 스레드 A : 락 X -> 락 Y 순서로 획득
- 스레드 B : 락 Y -> 락 X 순서로 획득
- 서로가 상대방이 점유한 락을 기다리며 무한 대기 상태 -> 데드락
#####  잘못된 재귀 락 사용
- 스레드가 자신이 이미 점유한 락을 중첩 요청하거나, 락 해제를 누락한 경우 교착 상태가 발생할 수 있음

#### 분산 시스템에서의 데드락 (심화)

분산 시스템, 특히 마이크로 서비스 아키텍처에서는 서비스 간의 호출 흐름과 분산된 자원 점유가 맞물리며 복잡한 형태의 데드락이 발생할 수 있다.

##### 서비스 간 순환 호출 + 자원 점유
- 서비스 A는 데이터베이스 락을 점유한 상태로 서비스 B로 호출
- 서비스 B는 Redis 락을 점유한 상태로 서비스 C 호출
- 서비스 C는 다시 A를 호출 -> 순환 호출 구조 + 자원 점유 상태 -> 분산 데드락 발생
##### 증상은 있으나 오류는 없음
- 표면적으로는 단순한 응답 지연이나 타임아웃 현상처럼 보이지만, 실제로는 서비스 간 순환 대기로 인해 전체 처리가 중단되는 상황
- 일반적인 데드락 탐지 도구로는 식별이 어려움 -> 분산 트레이싱 및 호출 경로 분석 필요

## 엔지니어링 사고력 기르기

### 1단계: 현실 세계의 비유로 '본질' 꿰뚫기 (사고의 출발점)

컴퓨터 과학의 모든 개념은 현실 세계의 문제를 해결하기 위해 나왔습니다. 
데드락의 본질은 "서로 양보하지 않는 고집쟁이들의 대치"입니다.

- **상황:** 좁은 외나무다리에서 두 사람이 마주쳤습니다.
- **교착:** 한 사람이 뒤로 물러서지 않으면 둘 다 평생 그 자리에 서 있어야 합니다.
- **엔지니어의 사고:** 
    - “외나무다리(자원)는 한 번에 한 사람만 지나갈 수 있네?” $\rightarrow$ **상호 배제**
    - “내가 이미 다리 위에 올라선 채로(점유), 상대가 비키기를 기다리네?” $\rightarrow$ **점유 대기**
    - “상대방을 강제로 밀쳐낼 수는 없네?” $\rightarrow$ **비선점**
    - “상대도 나를 기다리고, 나도 상대를 기다리는 고리가 생겼네?” $\rightarrow$ **순환 대기**
- **스스로 던져야 할 질문** : "내가 개발하는 시스템에서 '외나무다리' 역할을 하는 자원은 무엇이며, 프로세스들은 왜 서로 물러서지 못하고 있는가?"

### 2단계: 기술 스택으로 확장하기 (시스템 관점의 사고)

이제 이 개념을 우리가 다루는 백엔드 기술 체계로 확장해야 합니다. 
멘토가 원하는 것은 "OS 교과서에 나오는 데드락"이 아니라, "실무 DB나 멀티스레드 환경에서의 데드락"입니다.

####  사례: 데이터베이스(DB) 동시성 제어에서의 데드락

쇼핑몰 서비스에서 아주 흔하게 발생하는 시나리오를 머릿속으로 그려보세요.
- **트랜잭션 A (사용자 1의 주문):** 
    1. 1번 상품의 재고를 줄이기 위해 **1번 상품 로우(Row)에 락(Lock)을 획득**했습니다.
    2. 그 후, 포인트 할인을 적용하기 위해 **사용자 A의 포인트 로우에 락을 획득**하려 합니다.
- **트랜잭션 B (정산 시스템):**
    1. 마침 정산 처리를 위해 **사용자 A의 포인트 로우에 락을 먼저 획득**했습니다.
    2. 그 후, 정산 데이터 검증을 위해 **1번 상품 로우에 락을 획득**하려 합니다.
- **결과:** 
    - 트랜잭션 A는 B가 가진 포인트 락을 기다리고, 트랜잭션 B는 A가 가진 상품 락을 기다립니다. 시스템이 멈춥니다.
- **스스로 던져야 할 질문** : "자바(Java)의 `synchronized` 키워드나 DB의 `SELECT ... FOR UPDATE`를 쓸 때, 스레드들이 자원을 요청하는 '순서'가 꼬이면 어떻게 될까?"

### 3단계: 트레이드오프(Trade-off) 고민하기 (시니어의 사고)

데드락을 해결하는 방법은 교과서에 예방(Prevention), 회피(Avoidance), 탐지/복구(Detection), 무시(Ignore) 등으로 나옵니다. 
이걸 외우지 말고 **회사의 비용과 효율성 관점**에서 손익계산을 해보는 겁니다.

|**해결 전략**|**엔지니어의 비판적 사고 (Trade-off)**|**실무에서의 선택**|
|---|---|---|
|**자원 순서 고정 (예방)**|"모든 개발자가 완벽하게 같은 순서로만 Lock을 잡도록 코딩할 수 있을까? 시스템이 커지면 순서 규칙을 강제하기 너무 힘들 텐데?"|**매우 선호됨.** 개발 단계에서 코드 리뷰와 컨벤션으로 철저히 통제함.|
|**은행원 알고리즘 (회피)**|"데드락을 피하려고 매번 자원 상태를 계산하면 배보다 배꼽이 더 크지 않나? CPU 연산 낭비가 심할 것 같은데?"|**실무에서 안 씀.** 이론적으로만 존재하고 오버헤드가 너무 큼.|
|**타임아웃 & 데드락 감지**|"데드락이 걸리면 시스템이 탐지해서 하나를 강제로 종료(Rollback)시키면 되겠네. 하지만 롤백 비용과 사용자 경험 저하는 어떻게 책임지지?"|**대부분의 modern DB(MySQL, Oracle)가 채택.** 일단 걸리면 빠르게 한 놈을 죽여서 전체를 살림.|
|**타조 알고리즘 (무시)**|"데드락이 1년에 딱 한 번 발생한다면, 그걸 막으려고 수억 원어치 시스템 성능을 낭비하는 게 맞나? 그냥 재부팅하는 게 싸게 먹히지 않을까?"|**OS(Windows, Linux 등)가 채택.** 아주 드문 버그 때문에 OS 전체가 무거워지는 것을 방지함.|

- **스스로 던져야 할 질문** : "내가 설계하는 이 서비스는 데드락을 '절대 안 걸리게 완벽 방어'해야 하는 금융 시스템인가? 아니면 성능이 더 중요해서 '가끔 터지면 빠르게 복구'하는 커뮤니티 서비스인가?"

### 과제 제출을 위한 '스스로 사고하기' 가이드라인
이론을 복붙하지 마시고 아래와 같은 **'추론 일지'** 형태로 작성해 보세요.

```
### [데드락에 대한 나의 엔지니어링적 고찰]

1. 내가 정의한 데드락의 본질: 
   - 공유 자원을 두고 프로세스들이 '양보할 수 없는 타임라인'이 꼬여서 발생하는 병목 현상.

2. 실무 환경에서의 구체적 유도:
   - MSA(마이크로서비스) 환경에서 서비스 A가 서비스 B를 호출하고, 동시에 서비스 B가 서비스 A를 호출하는 순환 의존성(Circular Dependency)이 발생할 때도 거대한 아키텍처적 데드락이 발생할 수 있겠다고 생각함.

3. 엔지니어로서의 트레이드오프 선택:
   - 무조건 데드락을 없애려고 모든 곳에 락을 안 걸면 데이터가 깨질 것(Race Condition)이고, 과하게 걸면 데드락이 걸릴 것이다.
   - 따라서 나는 실무에서 [자원 접근 순서 대동소이화(Ordering)] 정책을 코드 컨벤션으로 가져가되, DB의 [Lock Timeout] 설정을 통해 시스템이 영원히 멈추는 최악의 상황을 방어하는 아키텍처를 선택하겠다.
```

단순히 4가지 조건을 외우는 코더(Coder)가 될 것인가, 시스템의 구조를 보고 리스크를 계산하는 엔지니어(Engineer)가 될 것인가는 이 "왜(Why)?"와 "비용(Cost)"을 고민하는 한 끗 차이에서 갈립니다.
