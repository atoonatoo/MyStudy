

# 데드락 실제 사례 조사

#### 운영체제에서의 데드락
##### 파일과 프린터 자원 충돌
- 프로세스 A는 파일을 점유한 후 프린터를 요청
- 프로세스 B는 프린터를 점유한 후 파일을 요청
- 서로가 필요한 자원을 상대가 점유하고 있어, 무한 대기 상태 발생
##### 시스템 콜 간 커널 락 충돌
- 두 커널 스레드가 read()와 write() 시스템 콜을 처리 중
- 락 획득 순서가 어긋나면서 커널 내부에서 데드락 발생 가능성 존재
---
#### 데이터베이스 트랜잭션 충돌 사례
##### 교차 테이블 접근
- 트랜잭션 A : 테이블 X의 행을 점유한 후 테이블 Y 접근 시도
- 트랜잭션 B : 테이블 Y의 행을 점유한 후 테이블 X 접근 시도
- 교차 잠금 발생 -> 트랜잭션 간 순환 대기 -> 데드락
##### 외부 I/O 대기 포함 트랜잭션 
- 트랜잭션이 자원을 점유한 상태로 외부 API 응답을 기다리는 동안, 다른 트랜잭션이 해당 자원을 요청하며 시스템 전체가 교착 상태에 빠질 수 있음.
##### 참고
- 주요 RDBMS는 데드락을 감지하여 자동으로 트랜잭션을 강제 종료(Rollback)한다.

---
#### 멀티 스레드 환경에서의 데드락
##### 락 순서 충돌
- 스레드 A : 락 X -> 락 Y 순서로 획득
- 스레드 B : 락 Y -> 락 X 순서로 획득
- 서로가 상대방이 점유한 락을 기다리며 무한 대기 상태 -> 데드락
#####  잘못된 재귀 락 사용
- 스레드가 자신이 이미 점유한 락을 중첩 요청하거나, 락 해제를 누락한 경우 교착 상태가 발생할 수 있음

#### 분산 시스템에서의 데드락 (심화)

분산 시스템, 특히 마이크로 서비스 아키텍처에서는 서비스 간의 호출 흐름과 분산된 자원 점유가 맞물리며 복잡한 형태의 데드락이 발생할 수 있다.

##### 서비스 간 순환 호출 + 자원 점유
- 서비스 A는 데이터베이스 락을 점유한 상태로 서비스 B로 호출
- 서비스 B는 Redis 락을 점유한 상태로 서비스 C 호출
- 서비스 C는 다시 A를 호출 -> 순환 호출 구조 + 자원 점유 상태 -> 분산 데드락 발생
##### 증상은 있으나 오류는 없음
- 표면적으로는 단순한 응답 지연이나 타임아웃 현상처럼 보이지만, 실제로는 서비스 간 순환 대기로 인해 전체 처리가 중단되는 상황
- 일반적인 데드락 탐지 도구로는 식별이 어려움 -> 분산 트레이싱 및 호출 경로 분석 필요

# 엔지니어링 사고력 기르기

### 1단계: 현실 세계의 비유로 '본질' 꿰뚫기 (사고의 출발점)

컴퓨터 과학의 모든 개념은 현실 세계의 문제를 해결하기 위해 나왔습니다. 
데드락의 본질은 "서로 양보하지 않는 고집쟁이들의 대치"입니다.

- **상황:** 좁은 외나무다리에서 두 사람이 마주쳤습니다.
- **교착:** 한 사람이 뒤로 물러서지 않으면 둘 다 평생 그 자리에 서 있어야 합니다.
- **엔지니어의 사고:** 
    - “외나무다리(자원)는 한 번에 한 사람만 지나갈 수 있네?” $\rightarrow$ **상호 배제**
    - “내가 이미 다리 위에 올라선 채로(점유), 상대가 비키기를 기다리네?” $\rightarrow$ **점유 대기**
    - “상대방을 강제로 밀쳐낼 수는 없네?” $\rightarrow$ **비선점**
    - “상대도 나를 기다리고, 나도 상대를 기다리는 고리가 생겼네?” $\rightarrow$ **순환 대기**
- **스스로 던져야 할 질문** : "내가 개발하는 시스템에서 '외나무다리' 역할을 하는 자원은 무엇이며, 프로세스들은 왜 서로 물러서지 못하고 있는가?"

### 2단계: 기술 스택으로 확장하기 (시스템 관점의 사고)

이제 이 개념을 우리가 다루는 백엔드 기술 체계로 확장해야 합니다. 
멘토가 원하는 것은 "OS 교과서에 나오는 데드락"이 아니라, "실무 DB나 멀티스레드 환경에서의 데드락"입니다.

####  사례: 데이터베이스(DB) 동시성 제어에서의 데드락

쇼핑몰 서비스에서 아주 흔하게 발생하는 시나리오를 머릿속으로 그려보세요.
- **트랜잭션 A (사용자 1의 주문):** 
    1. 1번 상품의 재고를 줄이기 위해 **1번 상품 로우(Row)에 락(Lock)을 획득**했습니다.
    2. 그 후, 포인트 할인을 적용하기 위해 **사용자 A의 포인트 로우에 락을 획득**하려 합니다.
- **트랜잭션 B (정산 시스템):**
    1. 마침 정산 처리를 위해 **사용자 A의 포인트 로우에 락을 먼저 획득**했습니다.
    2. 그 후, 정산 데이터 검증을 위해 **1번 상품 로우에 락을 획득**하려 합니다.
- **결과:** 
    - 트랜잭션 A는 B가 가진 포인트 락을 기다리고, 트랜잭션 B는 A가 가진 상품 락을 기다립니다. 시스템이 멈춥니다.
- **스스로 던져야 할 질문** : "자바(Java)의 `synchronized` 키워드나 DB의 `SELECT ... FOR UPDATE`를 쓸 때, 스레드들이 자원을 요청하는 '순서'가 꼬이면 어떻게 될까?"

### 3단계: 트레이드오프(Trade-off) 고민하기 (시니어의 사고)

데드락을 해결하는 방법은 교과서에 예방(Prevention), 회피(Avoidance), 탐지/복구(Detection), 무시(Ignore) 등으로 나옵니다. 
이걸 외우지 말고 **회사의 비용과 효율성 관점**에서 손익계산을 해보는 겁니다.

|**해결 전략**|**엔지니어의 비판적 사고 (Trade-off)**|**실무에서의 선택**|
|---|---|---|
|**자원 순서 고정 (예방)**|"모든 개발자가 완벽하게 같은 순서로만 Lock을 잡도록 코딩할 수 있을까? 시스템이 커지면 순서 규칙을 강제하기 너무 힘들 텐데?"|**매우 선호됨.** 개발 단계에서 코드 리뷰와 컨벤션으로 철저히 통제함.|
|**은행원 알고리즘 (회피)**|"데드락을 피하려고 매번 자원 상태를 계산하면 배보다 배꼽이 더 크지 않나? CPU 연산 낭비가 심할 것 같은데?"|**실무에서 안 씀.** 이론적으로만 존재하고 오버헤드가 너무 큼.|
|**타임아웃 & 데드락 감지**|"데드락이 걸리면 시스템이 탐지해서 하나를 강제로 종료(Rollback)시키면 되겠네. 하지만 롤백 비용과 사용자 경험 저하는 어떻게 책임지지?"|**대부분의 modern DB(MySQL, Oracle)가 채택.** 일단 걸리면 빠르게 한 놈을 죽여서 전체를 살림.|
|**타조 알고리즘 (무시)**|"데드락이 1년에 딱 한 번 발생한다면, 그걸 막으려고 수억 원어치 시스템 성능을 낭비하는 게 맞나? 그냥 재부팅하는 게 싸게 먹히지 않을까?"|**OS(Windows, Linux 등)가 채택.** 아주 드문 버그 때문에 OS 전체가 무거워지는 것을 방지함.|

- **스스로 던져야 할 질문** : "내가 설계하는 이 서비스는 데드락을 '절대 안 걸리게 완벽 방어'해야 하는 금융 시스템인가? 아니면 성능이 더 중요해서 '가끔 터지면 빠르게 복구'하는 커뮤니티 서비스인가?"

### 과제 제출을 위한 '스스로 사고하기' 가이드라인
이론을 복붙하지 마시고 아래와 같은 **'추론 일지'** 형태로 작성해 보세요.

```
### [데드락에 대한 나의 엔지니어링적 고찰]

1. 내가 정의한 데드락의 본질: 
   - 공유 자원을 두고 프로세스들이 '양보할 수 없는 타임라인'이 꼬여서 발생하는 병목 현상.

2. 실무 환경에서의 구체적 유도:
   - MSA(마이크로서비스) 환경에서 서비스 A가 서비스 B를 호출하고, 동시에 서비스 B가 서비스 A를 호출하는 순환 의존성(Circular Dependency)이 발생할 때도 거대한 아키텍처적 데드락이 발생할 수 있겠다고 생각함.

3. 엔지니어로서의 트레이드오프 선택:
   - 무조건 데드락을 없애려고 모든 곳에 락을 안 걸면 데이터가 깨질 것(Race Condition)이고, 과하게 걸면 데드락이 걸릴 것이다.
   - 따라서 나는 실무에서 [자원 접근 순서 대동소이화(Ordering)] 정책을 코드 컨벤션으로 가져가되, DB의 [Lock Timeout] 설정을 통해 시스템이 영원히 멈추는 최악의 상황을 방어하는 아키텍처를 선택하겠다.
```

단순히 4가지 조건을 외우는 코더(Coder)가 될 것인가, 시스템의 구조를 보고 리스크를 계산하는 엔지니어(Engineer)가 될 것인가는 이 "왜(Why)?"와 "비용(Cost)"을 고민하는 한 끗 차이에서 갈립니다.

# 외주 - 몽골 앱 개발 미팅 준비

제시해주신 요구사항을 바탕으로 **데이터베이스(DB) 모델링**, **RESTful API 명세서**, 그리고 기타 시스템 요구사항(기능적/비기능적 요구사항)을 체계적으로 정리한 보고서입니다.

기존에 선호하시는 명확한 구조화 형식에 맞춰 작성되었습니다.

## 1. 데이터베이스(DB) 모델링

Supabase(PostgreSQL) 환경에 최적화된 스키마 구조입니다. 비밀번호 수정 및 회원 탈퇴 시 검증 로직, 지원자 요약 및 상세 정보 조회, 게시글 생성/수정 시간 관리가 가능하도록 설계했습니다.

### ⚠️ 엔티티 관계도 (ERD) 개념 요약

- `users` 1 : N `applications` (한 회원은 여러 공고에 지원 가능)
    
- `posts` 1 : N `applications` (한 공고에는 여러 회원이 지원 가능)
    
- `applications`는 `users`와 `posts` 사이의 다대다(N:M) 관계를 해소하는 교차 테이블입니다.
    

### 📊 테이블 상세 구조

#### 1. `users` (회원 테이블)

- 운영자 계정은 시스템 초기화(Data Seeding) 시 `role`을 `'ADMIN'`으로 설정하여 미리 삽입해 둡니다.
    

| **컬럼명**             | **타입**       | **제약조건**         | **설명**                  |
| ------------------- | ------------ | ---------------- | ----------------------- |
| `id`                | BIGSERIAL    | PRIMARY KEY      | 회원 고유 ID                |
| `email`             | VARCHAR(100) | UNIQUE, NOT NULL | 이메일 (로그인 ID)            |
| `password`          | VARCHAR(255) | NOT NULL         | 암호화된 비밀번호               |
| `name`              | VARCHAR(50)  | NOT NULL         | 이름                      |
| `gender`            | VARCHAR(10)  | NOT NULL         | 성별 (`MALE`, `FEMALE` 등) |
| `birth_date`        | DATE         | NOT NULL         | 생년월일 (나이 계산용)           |
| `nationality`       | VARCHAR(50)  | NOT NULL         | 국적                      |
| `phone`             | VARCHAR(20)  | NOT NULL         | 연락처                     |
| `profile_image_url` | VARCHAR(512) | NOT NULL         | 필수 첨부 얼굴 사진 (S3 URL)    |
| `role`              | VARCHAR(20)  | NOT NULL         | 권한 (`USER`, `ADMIN`)    |
| `created_at`        | TIMESTAMP    | DEFAULT NOW()    | 가입 일시                   |
| `updated_at`        | TIMESTAMP    | DEFAULT NOW()    | 정보 수정 일시                |

#### 2. `posts` (인력 모집 게시판 테이블)

- 운영자만 작성할 수 있으며, 첨부 이미지는 S3에 업로드된 URL을 저장합니다.
    

|**컬럼명**|**타입**|**제약조건**|**설명**|
|---|---|---|---|
|`id`|BIGSERIAL|PRIMARY KEY|게시글 고유 ID|
|`title`|VARCHAR(200)|NOT NULL|모집 제목 (필수)|
|`content`|TEXT|NOT NULL|글 내용 (텍스트)|
|`image_url`|VARCHAR(512)|NULL|첨부 이미지 URL (선택 가능)|
|`author_id`|BIGINT|FOREIGN KEY (users.id)|작성자 ID (운영자)|
|`created_at`|TIMESTAMP|DEFAULT NOW()|글 작성 시간|
|`updated_at`|TIMESTAMP|DEFAULT NOW()|글 수정 시간|

#### 3. `applications` (지원 현황 테이블)

- 특정 사용자가 특정 인력 모집 글에 지원한 이력을 관리합니다. 중복 지원 방지를 위해 복합 유니크 제약조건을 설정합니다.
    

|**컬럼명**|**타입**|**제약조건**|**설명**|
|---|---|---|---|
|`id`|BIGSERIAL|PRIMARY KEY|지원 고유 ID|
|`post_id`|BIGINT|FOREIGN KEY (posts.id)|지원한 게시글 ID|
|`user_id`|BIGINT|FOREIGN KEY (users.id)|지원한 사용자 ID|
|`applied_at`|TIMESTAMP|DEFAULT NOW()|지원 시간|

> **Constraints (제약조건):**
> 
> `UNIQUE (post_id, user_id)` : 동일 사용자가 한 공고에 중복 지원하는 것을 방지합니다.

## 2. RESTful API 명세서

Spring Security와 JWT를 활용한 인증 구조를 반영하였으며, Axios 통신에 적합하도록 설계했습니다.

### 🔐 1. 인증 및 회원 관리 (Auth & User API)

|**기능**|**Method**|**URI**|**권한**|**설명**|
|---|---|---|---|---|
|회원가입|`POST`|`/api/auth/signup`|비인증|필수 정보 수집 (MultipartFile 포함)|
|로그인|`POST`|`/api/auth/login`|비인증|JWT 토큰 발급 (`AccessToken`)|
|내 정보 조회|`GET`|`/api/users/me`|USER, ADMIN|마이페이지 및 상태 관리용|
|회원 정보 수정|`PUT`|`/api/users/me`|USER|국적, 비밀번호, 이미지, 연락처만 수정 가능|
|회원 탈퇴|`POST`|`/api/users/me/withdraw`|USER|현재 비밀번호 검증 후 삭제|

### 📋 2. 인력 모집 게시판 (Recruitment Board API)

|**기능**|**Method**|**URI**|**권한**|**설명**|
|---|---|---|---|---|
|공고 작성|`POST`|`/api/posts`|**ADMIN**|제목(필수), 내용, 이미지(S3 업로드)|
|공고 목록 조회|`GET`|`/api/posts`|전체|전체 인력 모집 글 리스트 조회|
|공고 상세 조회|`GET`|`/api/posts/{postId}`|전체|특정 공고의 상세 내용 조회|
|공고 수정|`PUT`|`/api/posts/{postId}`|**ADMIN**|공고 내용 및 이미지 수정 (`updated_at` 갱신)|
|공고 삭제|`DELETE`|`/api/posts/{postId}`|**ADMIN**|공고 삭제 (연관 지원 내역 Cascade 삭제)|

## 3. 기타 및 비기능 요구사항 (Requirements)

프로젝트 개발 및 아키텍처 구성을 위해 반드시 반영되어야 할 세부 요구사항 목록입니다.

### 🌐 1. 다국어 지원 (Internationalization, i18n)

- **프론트엔드 (React):** `react-i18n` 라이브러리를 활용하여 전역 상태(Zustand)에 사용자가 선택한 언어 코드(`ko`, `en`)를 저장하고 화면 레이블을 동적으로 치환합니다.
    
- **백엔드 (Spring Boot):** 다국어 처리 예외 메시지나 알림이 필요할 경우, `Accept-Language` 헤더를 받아 처리할 수 있도록 기본적인 `LocaleResolver` 환경을 구축합니다.
    

### 📱 2. 반응형 웹 및 UI/UX

- **반응형 디자인:** Mobile First 또는 Desktop First 레이아웃 구조를 적용합니다. Tailwind CSS의 브레이크포인트(`sm`, `md`, `lg`)를 적극 활용하여 모바일 화면에서도 GNB(메뉴바), 테이블, 지원 폼 등이 깨지지 않고 최적화되도록 구성합니다.
    
- **Axios 공통 설정:** JWT 토큰을 LocalStorage 또는 Zustand 상태에 보관하고, Axios 인터셉터(Interceptor)를 통해 모든 요청 헤더에 `Authorization: Bearer {Token}`을 자동으로 첨부합니다.
    

### 🔒 3. 보안 및 비즈니스 검증 로직 (Security & Validation)

- **비밀번호 검증 및 암호화:** * 회원가입 및 탈퇴 시 프론트엔드 단에서 1차로 `Password === PasswordConfirm`을 검증합니다.
    
    - 백엔드에서는 Spring Security의 `BCryptPasswordEncoder`를 통해 비밀번호를 암호화하여 DB에 저장합니다. 탈퇴 시에는 입력된 평문 비밀번호와 DB의 암호화된 비밀번호를 `matches()` 메서드로 검증합니다.
        
- **권한 제어 (RBAC):** Spring Security 설정을 통해 `/api/posts`의 생성/수정/삭제 및 지원 현황 조회 API는 `@PreAuthorize("hasRole('ADMIN')")` 어노테이션으로 접근을 철저히 제한합니다.
    

### 📁 4. 파일 업로드 인프라 (AWS S3)

- **S3 프리티어 활용:** 사용자의 얼굴 사진 및 게시글 첨부 이미지는 Spring Boot 서버를 거쳐 AWS S3 버킷에 저장되며, DB 테이블에는 오직 접근 가능한 **Public URL 텍스트**만 저장됩니다.
    
- **수정/삭제 시 거동:** 회원이 프로필 이미지를 변경하거나 운영자가 게시글을 수정/삭제할 경우, 기존에 S3에 저장되어 있던 구버전 이미지 파일을 S3 버킷에서 지워주는 오브젝트 삭제 로직을 백엔드에 구현하여 스토리지 용량을 관리합니다.
    

### 🚀 5. 초기 데이터 구성 (Data Seeding)
