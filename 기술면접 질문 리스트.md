- 동시성이란?
```
동시성이란 한 번에 여러 가지 작업을 다루는 시스템의 논리적인 설계 능력을 의미합니다. 
이는 물리적으로 동시에 여러 일이 벌어지는 병렬성과는 구분되는 개념입니다. 
동시성은 실제로는 하나의 처리 장치가 여러 작업을 아주 짧은 시간 간격으로 번갈아 가며 수행하여 마치 모든 작업이 동시에 진행되는 것처럼 보이게 만듭니다. 
들어 카페 직원이 한 명일 때 주문을 받고 커피 머신을 작동시킨 뒤 기다리는 동안 다른 손님의 결제를 돕는 상황이 동시성입니다. 
컴퓨터 시스템에서는 운영체제가 CPU 스케줄링을 통해 여러 프로세스나 스레드 사이를 빠르게 오가는 문맥 교환을 수행함으로써 이를 구현합니다. 
동시성의 목적은 단순히 계산 속도를 높이는 것이 아니라 대기 시간이 발생하는 작업들 사이의 공백을 메워 자원 활용도를 극대화하고 사용자가 느끼는 응답성을 개선하는 데 있습니다. 
이를 통해 복잡한 시스템에서도 여러 요청을 지체 없이 처리할 수 있는 유연한 구조를 갖출 수 있게 됩니다.
```

- 동시성 vs 병렬성
```
병렬성이란 여러 개의 작업을 물리적으로 동시에 처리하는 능력을 의미합니다. 
이는 논리적인 순서 교체인 동시성과는 달리 실제로 여러 개의 코어나 처리 장치가 각각 서로 다른 작업을 한꺼번에 수행하는 상태를 말합니다. 
예를 들어 카페에 직원이 두 명 이상 있어서 한 명은 주문을 받고 다른 한 명은 동시에 커피를 내리는 상황이 병렬성입니다. 
컴퓨터 시스템에서는 멀티 코어 프로세서를 활용하여 데이터 연산을 여러 단위로 쪼개어 각 코어에 분배하고 동시에 연산함으로써 작업 완료 시간을 획기적으로 단축합니다. 
병렬성의 핵심은 하드웨어 자원의 병렬적 가동을 통해 전체적인 처리량과 처리 속도를 높이는 데 있습니다. 
동시성이 단일 코어 환경에서도 다중 작업의 진행 흐름을 관리하는 논리적 개념이라면 병렬성은 다중 코어 환경에서 작업의 물리적 중복 처리를 통해 절대적인 성능을 극대화하는 물리적 개념입니다. 
따라서 대규모 데이터를 처리하거나 고성능 연산이 필요한 인공지능 학습 및 그래픽 렌더링 분야에서 병렬성은 필수적인 요소로 작용하게 됩니다. 
또한 분산 컴퓨팅 환경에서 여러 서버가 문제를 나누어 푸는 방식도 병렬성의 한 범주로 볼 수 있습니다.
```

- 경쟁 조건이란?
```
경쟁 조건은 여러 프로세스가 공유 자원에 동시 접근할 때 실행 타이밍이나 순서에 따라 결과값이 달라지는 현상입니다. 
예를 들어 두 스레드가 동일한 변수를 1씩 증가시킬 때 연산 중간에 문맥 교환이 일어나면 각자 읽은 값을 기준으로 덮어쓰게 되어 최종 결과가 의도보다 작게 나타날 수 있습니다. 
이를 방지하기 위해 뮤텍스, 세마포어 등 동기화 매커니즘을 사용하여 임계 영역에 대한 상호 배제를 보장해야 합니다. 
요약하면 동기화되지 않은 자원 접근으로 인해 데이터 무결성이 손상되는 상태를 말합니다.
```

- 경쟁 조건 해결법 : 뮤텍스
```
뮤텍스는 공유 자원을 한 번에 하나의 프로세스만 사용하도록 제한하여 상호 배제를 보장하는 기법입니다. 
예로 화장실 열쇠를 가진 사람만 입장하고 나머지는 줄을 서서 기다리는 것과 같습니다. 
스레드가 임계 영역에 진입하기 전 락을 획득하고 작업이 끝나면 락을 해제하여 다른 스레드의 접근을 제어합니다. 
이를 통해 여러 스레드의 동시 접근에 따른 데이터 오염을 방지합니다. 요약하면 락을 활용해 자원 사용 권한을 순차적으로 부여하는 동기화 도구입니다.
```

- 경쟁 조건 해결법 : 세마포어 & 모니터
```
세마포어는 공유 자원의 개수를 나타내는 변수를 사용하여 여러 프로세스의 접근을 제어합니다. 
여러 칸의 주차 공간처럼 허용된 개수만큼 프로세스가 진입할 수 있으며 자원이 없으면 대기합니다. 
모니터는 객체 지향 프로그래밍 언어 수준에서 제공되는 동기화 도구입니다. 
공유 자원과 이를 조작하는 함수를 하나의 객체로 묶어 보호하며 한 번에 하나의 스레드만 해당 객체의 메서드를 실행하도록 자동으로 보장합니다. 
요약하면 세마포어는 숫자로 개수를 관리하고 모니터는 객체 단위로 안전한 실행을 돕습니다.
```

- 교착 상태란?
```
교착 상태는 두 개 이상의 프로세스가 각자 자원을 점유한 채 서로의 자원을 무한히 기다리며 모든 진행이 멈춘 상태를 뜻합니다. 
예를 들어 프로세스 1은 자원 A를 가진 상태에서 B를 기다리고, 프로세스 2는 자원 B를 가진 상태에서 A를 기다리면 서로 영원히 대기하게 됩니다. 
해결을 위해 상호 배제, 점유 대기, 비선점, 순환 대기 중 하나를 제거하거나 은행원 알고리즘을 사용해 예방 및 회피해야 합니다. 
```

- 교착 상태 성립 & 은행원 알고리즘
```
교착 상태가 성립하기 위해서는 상호 배제, 점유 대기, 비선점, 순환 대기라는 네 가지 조건이 모두 충족되어야 합니다. 
상호 배제는 자원을 한 번에 한 프로세스만 사용함을 의미하며, 점유 대기는 자원을 가진 상태에서 다른 자원을 기다리는 것입니다.
비선점은 할당된 자원을 강제로 뺏을 수 없음을 뜻하고, 순환 대기는 프로세스들이 원형으로 자원을 대기하는 고리가 형성된 상태를 말합니다. 
이를 해결하는 예방 기법은 네 조건 중 하나를 사전에 차단하여 교착 상태를 막지만 자원 이용률이 떨어지는 한계가 있습니다. 
반면 회피 기법의 핵심인 은행원 알고리즘은 자원 할당 전 시스템의 상태를 검사하여 할당 후에도 안전 상태가 유지될 때만 자원을 부여합니다. 
즉 모든 프로세스가 종료될 수 있는 안전 순서열이 존재하는지 미리 계산하여 교착 상태의 가능성 자체를 비껴가는 유연한 전략입니다. 
요약하면 예방은 조건 부정으로 입구를 막는 것이고 회피는 상황을 봐가며 길을 열어주는 지혜로운 관리 방식입니다.
```

- 기아 상태란?
```
기아 상태는 특정 프로세스의 우선순위가 낮아 필요한 자원을 할당받지 못하고 무한히 대기하는 현상입니다. 
예를 들어 우선순위 스케줄링에서 높은 순위의 작업이 계속 들어오면 낮은 순위의 작업은 계속 뒤로 밀려 실행되지 못합니다. 
이를 해결하기 위해 오래 대기한 작업의 우선순위를 점진적으로 높여주는 에이징 기법이나 모든 프로세스에 공평하게 자원을 할당하는 알고리즘을 사용합니다. 
요약하면 자원 할당의 불공정함으로 인해 특정 작업이 무기한 방치되는 상태를 말합니다.
```

- 기아 상태 해결법 : 에이징 기법
```
기아 상태를 해결하는 가장 대표적인 방법은 에이징 기법입니다. 
이는 특정 프로세스가 자원을 할당받지 못하고 오래 대기할수록 해당 프로세스의 우선순위를 점진적으로 높여주는 방식입니다. 
시간이 흐름에 따라 순환적으로 순위가 상승하므로 아무리 우선순위가 낮은 작업이라도 결국에는 자원을 할당받아 실행될 수 있도록 보장합니다. 
또한 모든 프로세스에 공평한 실행 기회를 부여하는 라운드 로빈 스케줄링을 사용하거나 선입선출 방식을 도입하여 특정 작업이 무기한 뒤로 밀리는 현상을 원천적으로 차단할 수 있습니다. 
이외에도 멀티레벨 피드백 큐를 통해 대기 시간이 길어진 작업을 상위 큐로 이동시키는 등 자원 할당 시 대기 시간을 핵심 지표로 반영하는 정책을 수립하는 것이 효과적입니다. 
요약하면 에이징을 통해 대기 시간에 따른 보상을 제공하거나 공정성을 우선시하는 알고리즘을 적용해 시스템 내 특정 작업의 소외를 방지하는 것이 핵심입니다.
```

- 가상화란?
```
가상화는 물리적인 하드웨어 자원을 추상화하여 하나의 물리적 서버에서 여러 개의 가상 시스템을 독립적으로 실행할 수 있게 하는 기술입니다.

하이퍼바이저라는 핵심 소프트웨어 계층을 통해 물리 자원을 논리적으로 분리하여 각 가상 머신에 효율적으로 할당합니다.

이를 통해 하드웨어의 유휴 자원을 최소화하고 자원 활용률을 극대화하여 인프라 운영 비용을 획기적으로 절감할 수 있습니다.

가상화는 물리 장치에 직접 설치되는 베어메탈 방식과 운영체제 위에서 실행되는 호스트 방식으로 나뉘어 다양한 환경에 적용됩니다.

논리적으로 격리된 환경을 제공하기 때문에 하나의 가상 머신에서 발생한 장애가 전체 시스템으로 확산되는 것을 방지합니다.

현대 클라우드 컴퓨팅의 핵심 기반으로서 서비스의 빠른 확장과 복구를 가능하게 하며 서버 관리의 유연성을 극대화합니다.

결국 물리적인 한계를 극복하고 소프트웨어만으로 자원을 제어할 수 있는 환경을 구축하는 것이 가상화의 궁극적인 목표입니다.

가상화의 연장선에 있는 컨테이너 기술이나 도커에 대해서도 이어서 설명해 드릴까요?
```

- 동기 vs 비동기
```
동기 방식은 작업이 순차적으로 실행되는 형태를 말하며 하나의 작업이 완료될 때까지 다음 작업은 대기하게 됩니다.

이는 설계가 매우 간단하고 코드의 가독성이 좋다는 장점이 있지만 앞선 작업이 지연되면 전체 시스템이 멈추는 블로킹 현상이 발생합니다.

반면에 비동기 방식은 요청을 보낸 후 작업의 완료 여부와 관계없이 즉시 다음 명령을 수행하는 효율적인 방식입니다.

시간이 오래 걸리는 네트워크 호출이나 파일 입출력 작업에 주로 사용되며 여러 작업을 동시에 처리하여 시스템 자원을 최적화할 수 있습니다.

다만 비동기 작업은 결과가 돌아오는 시점이 제각각이기 때문에 전체적인 코드의 흐름을 파악하기 어렵고 관리가 복잡해진다는 단점이 있습니다.

현대의 대규모 웹 서비스 개발에서는 사용자 경험과 서버 성능을 극대화하기 위해 비동기 처리 기법이 필수적으로 활용되고 있습니다.

이 두 방식의 차이를 명확히 이해하고 프로젝트의 성격에 맞춰 적절하게 혼합하여 사용하는 것이 중요합니다.

원하신다면 비동기 처리의 구체적인 구현 방식인 프로미스나 에이싱크 어웨이트에 대해서도 더 자세히 설명해 드릴 수 있습니다.
```

- 블로킹 vs 논블로킹
```
블로킹은 작업을 요청받은 쪽에서 제어권을 계속 소유하여 요청한 쪽이 아무것도 하지 못하게 만드는 방식입니다.

작업이 끝날 때까지 호출한 쪽은 실행 흐름을 멈추고 결과가 돌아올 때까지 하염없이 기다려야만 합니다.

반면에 논블로킹은 작업을 요청받은 즉시 제어권을 다시 호출한 쪽으로 넘겨주는 메커니즘을 가지고 있습니다.

이 덕분에 요청을 보낸 쪽은 작업의 완료 여부를 기다리지 않고 곧바로 다른 코드를 실행할 수 있습니다.

블로킹은 흐름이 매우 직관적이고 순서가 보장되지만 자원을 효율적으로 쓰지 못하는 경우가 생깁니다.

논블로킹은 시스템 전체의 응답성을 높이고 자원을 알뜰하게 쓸 수 있지만 프로그래밍 난이도가 높아집니다.

결국 두 개념의 핵심은 제어권이 누구에게 있느냐와 호출된 작업이 완료될 때까지 대기하느냐의 차이입니다.

이 내용과 연계하여 동기 방식과 비동기 방식이 어떻게 다른지도 함께 알아볼까요.
```
