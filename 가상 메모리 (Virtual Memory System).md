
## 정의
`가상 메모리란 컴퓨터가 실제 물리적 메모리(RAM)보다 더 큰 메모리 공간을 가진 것처럼 속여서 사용하는 메모리 관리 기법이다.`

## 메모리의 한계

현대의 많은 컴퓨터를 사용자들은 대부분 컴퓨터를 이용할 때 프로세스를 1,2개만 실행하지 않는다. 그렇기에 실제 각 프로세스 마다 충분한 메모리를 할당하기에는 메모리 크기에 한계가 있다.

## 그렇다면 메모리를 늘리면 성능이 좋아지는가?

가상 메모리를 사용하지 않는 환경에서 RAM을 늘리는 방식으로는 특정 지점까지는 성능이 비약적으로 좋아지지만, 곧 한계에 봉착할 것이다.

- **외부 단편화**
  - 가상 메모리가 없으면 프로세스는 메모리에 연속된 공간으로 할당되어야 한다.
  - 문제
    - 메모리 전체 용량이 16GB이고 남은 공간이 4GB여도, 이 공간이 1GB씩 네 군대로 쪼개져있다면 2GB짜리 프로세스 하나를 실행할 수 없다.
  - 결과
    - RAM을 아무리 늘려도 중간중간 빈틈(GAP)이 생기면 실제 사용할 수 있는 공간보다 훨씬 적은 양만 활용하게 된다.
  - 현실 비유
    - 현실로 비유하자면 가방에 옷을 담을 때 아무렇게나 막 담아서 많은 옷을 못담는 원리이다.
    - 우리는 이러한 문제가 발생하지 않도록, 가방에 옷을 최대한 많이 담기위해 옷을 잘 개어서 차곡차곡 담으며 남은 공간을 최대한 활용하도록 담는 것과 같다.
    
- **메모리 보호 및 격리 문제**
  - 한 프로세스의 버그나 악성 코드가 다른 프로세스의 메모리 영역을 직접 침범하여 시스템 전체를 다운시킬 수 있다.
  - 기업용 시스템에서는 이는 치명적인 안정성 결여로 이어진다.
  
- **주소 공간의 한계**
  - CPU의 비트 수(예:32bit)에 따라 참조할 수 있는 주소 범위가 정해져 있다.
  - 32비트 시스템은 RAM을 100GB로 늘려도 4GB 이상의 영역은 인식조차 할 수 없다.
  - 하드웨어 확장만으로는 아키텍처의 한계를 넘을 수 없다.


## 폰노이만 구조 (컴퓨터 구조)

- `폰 노이만 구조 기반이므로, 코드는 메모리를 반드시 있어야 한다.`
- 우리가 흔히 사용하는 스마트폰, 노트북, 기업용 서버에 이르기까지 거의 모든 현대적 컴퓨터는 폰 노이만(Von Neumann Architecture)를 따르고 있다.

- **폰 노이만 구조의 핵심 : 프로그램 내장 방식**
  - 과거의 초기 컴퓨터는 새로운 계산을 하려면 사람이 직접 전선을 새로 꽂거나 스위치를 조작해야 했다. 즉, 하드웨어를 물리적으로 개조해야 소프트웨어가 바뀌는 구조였다.
  
![](https://velog.velcdn.com/images/atoonatoo/post/ef8c91c9-38bc-4848-ae36-d4c4bd7a730a/image.png)

  - 폰 노이만은 `프로그램(코드)도 숫자 데이터처럼 메모리에 저장해 놓고, CPU가 필요할 때마다 읽어서 실행하게 하자`는 혁신적인 아이디어를 낸다.

  - 이를 프로그램 내장 방식(Stored-program concept)이라고 부른다. 

![](https://velog.velcdn.com/images/atoonatoo/post/b82b411b-513b-491c-ae34-21f0ffba00b5/image.png)


## 코드는 반드시 메모리에 있어야 한다.

- CPU는 컴퓨터의 두뇌이지만, 기억력이 매우 좋지 않으며, 아주 소량의 데이터만 담을 수 있는 `레지스터`라는 공간만 가지고 있다. 따라서 CPU가 작업을 수행하려면 다음의 과정이 필수적이다.

- **저장 장치와 메모리의 차이**
  - 하드디스크는 데이터가 영구히 저장되는 창고지만, 속도가 매우 느리다. 반면 메모리는 속도가 매우 빠른 작업이다.
- **반드시 메모리여야 하는 이유**
  - CPU가 하드디스크에서 직접 코드를 읽어오기에는 속도 차이가 수만배 이상 난다.
  - 따라서 `코드가 실행되려면 반드시 메모리에 있어야 한다.`는 말은 실행 파일 형태(Binary)로 하드디스크에 잠자고 있던 프로그램이 CPU에 의해 처리되기 위해 RAM이라는 작업 공간으로 복사되어 올라와야한다는 뜻이다.
  
### 동작 원리

![](https://velog.velcdn.com/images/atoonatoo/post/860bfe88-4f4f-4743-9ae2-4d9789dd41e3/image.png)

1. 인출(Fetch) : 메모리의 특정 주소에 있는 명령어를 CPU로 가져온다.
2. 해독(Decode) : 가져온 명령어가 무엇(더하기인지, 저장인지 등)인지 해석.
3. 실행(Execude) : 해석된 결과에 따라 연산을 수행.

- 이 과정이 초당 수억번 반복된다.
- 이때 CPU는 메모리 주소만을 바라보며 명령어를 찾기 때문에 메모리에 코드가 없다면 CPU는 아무런 동작을 할 수 없다.


## 가상 메모리가 필요한 이유

- **하나의 프로세스만 실행 가능한 시스템 (배치 처리 시스템 등)**
  - 프로그램을 메모리로 로드 (load)
  - 프로세스 실행
  - 프로세스 종료 (메모리 해제)

- **여러 프로세스 동시 실행 시스템**
  - 메모리 용량 부족 이슈
  - 프로세스 메모리 영역간에 침범 이슈

- **현대의 요구는 다중 프로그래밍이 기본이다.** 
  - 사용자들은 이제 음악을 들으면서, 코딩을 하고, 브라우저를 수십 개 띄워놓기를 원한다. 가상 메모리가 없다면 위에서 이야기한대로 `메모리 용량 부족 이슈`, `프로세스 간 메모리 침범 이슈` 등의 치명적인 문제가 발생하며, 가상 메모리는 이러한 문제들을 해결할 수 있다.

- **예시**
  - 프로세스 하나는 4GB를 할당하지만 개인 PC 메모리는 보통 16GB
    - 프로그램을 두 세개만 실행하는 경우는 없다.
    - 운영체제와 관련된 프로그램만해도 10여개가 넘는다.
    
  - 스마트폰에 딱 하나 앱을 실행하는 경우도 많지 않다.

- **즉, 단순히 메모리가 부족해서를 넘어서 시스템의 효율성과 안정성을 극대화 하기 위해 가상 메모리 구조는 필수이다.**

## 가상 메모리 (Virtual Memory System)

- 다시 한번 정의 하자면 `가상 메모리란 컴퓨터가 실제 물리적 메모리(RAM)보다 더 큰 메모리 공간을 가진 것처럼 속여서 사용하는 메모리 관리 기법이다.`

- **부분적 로딩**
  - 실제 사용하는 메모리는 작다는 점에 착안해서 고안된 기술이다.
  - 프로그램의 전체 용량 중에서 실제로 당장 게산에 쓰이는 부분은 아주 일부분에 불과하다는 뜻이다.
  - 가상 메모리는 이 점을 이용해 꼭 필요한 부분만 RAM에 올리고 나머지는 보조 기억장치에 보관함으로써 메모리 낭비를 극적으로 줄인다.

- **프로세스 격리를 통한 안정성 확보**
  - 프로세스 공간 분리로, 프로세스 이슈가 전체 시스템에 영향을 주지 않을 수 있다.
  - 각 프로세스가 자신만의 가상 공간에서만 움직이기 때문에, 특정 프로세스에서 치명적인 오류가 발생하거나 악성 코드가 작동하더라도, 그 영향이 담장 너머의 다른 프로세스나 운영체제 핵심부로 전달되지 않는다.
  - 이 격리 기법 덕분에 우리는 앱 하나가 멈춘다고 해서 컴퓨터 전체가 강제 종료해야 하는 상황을 피할 수 있다.

## 가상 메모리 기본 아이디어

- 프로세스는 가상 주소를 사용하고, 실제 해당 주소에서 데이터를 읽고/쓸 때만 물리 주소로 변경해야 한다.
- virtual address(가상 주소) : 프로세스가 참조하는 주소
- physical address(물리 주소) : 실제 메모리 주소

## MMU(Memory Management Unit)
![](https://velog.velcdn.com/images/atoonatoo/post/afba9f7f-93a9-4ab2-9893-29ad53bf4310/image.png)

- **정의**
  - MMU는 CPU가 메모리에 접근하려고 할 때, 프로세스가 사용하는 `논리적인 가상 주소(Virtual Address)`를 실제 물리적인 RAM의 `물리 주소(Physical Address)`로 실시간으로 변환해 주는 CPU 내부(또는 근처)의 핵심 하드웨어 장치이다.

- **동작 원리**
  - 비유하자면 가상 주소는 `강남구 테헤란로 123, 이순신 타워 10층 502호` 같은 논리적인 주소이다.
  - 하지만 우체국(CPU/하드웨어)는 이 실제로 편지를 배달하려면 `위도 37.xxx, 경도 127.xxx`라는 물리적인 좌표가 필요하다.
  - MMU는 이 논리적 주소를 물리적 좌표로 즉시 바꿔주는 역할을 한다.
  
  1. CPU 요청 : CPU가 코드를 실행하다가 가상 주소 `0x1000`에 있는 데이터를 가져와` 라고 요청한다.
  2. MMU 개입 : 이 요청은 RAM으로 바로 가지 않고 MMU로 들어간다.
  3. 주소 변환(Mapping) : MMU는 메모리에 저장된 페이지 테이블이라는 지도를 참조하여, 가상 주소 `0x1000`이 실제 RAM의 어디(예:`0x5000`)에 있는지 찾아낸다.

- **메모리에 접근하기 전애 거쳐야하는 이유**

  - 잘못된 접근 차단(보안)
    - 만약 프로세스 A가 허가되지 않은 영역(운영체제 영역 등)의 주소를 요청하면, MMU는 접근을 막고 CPU에 경고(Interrupt)를 보낸다.
  - 데이터 조각 모음(효율)
    - 실제 RAM에는 데이터가 여기저기 흩어져 있어도(단편화), MMU가 이를 논리적으로 이어 붙여서 CPU에게는 연속된 공간인 것처럼 보여주게 한다.
  - 성능 최적화
    - 이 변환 과정은 `O(1)`에 가까운 속도로 이루어져야 한다. 소프트웨어가 아닌 하드웨어(MMU)로 구현된 이유도 바로 이 속도 때문이다.
   

## 가상 메모리와 MMU

- CPU는 가상 메모리를 다루고, 실제 해당 주소 접근 시 MMU 하드웨어 장치를 통해 물리 메모리 접근
- 하드웨어 장치를 이용해야 주소 변환이 빠르기 때문에 별도 장치를 둔다.

- **왜 별도의 하드웨어 장치를 두는가?**  
  - 속도
    - 소프트웨어는 주소 하나 변환할 때마나 추가로 수십 줄 코드가 실행되어야 한다.
    - 컴퓨터 속도가 10~100배 느려지게 된다.
  - 보안
    - 소프트웨어는 해킹이나 버그로 조작 가능성이 있다.
    - 하지만, 주소 변환 권한을 하드웨어가 가지고 있으면 프로세스가 자기 영역 밖을 넘보려 할 때 하드웨어 수준에서 즉각 차단(Segmentation Fault)할 수 있다.

## 페이징 시스템 (Paging system)

- **정의**
  - 페이징 시스템은 메모리를 효율적으로 사용하기 위해 가상 메모리와 물리 메모리를 서로 다른 단위로 나눈다.

- **페이지(Page)**
  - 가상 메모리를 나누는 최소 단위이다. 
  - 프로세스가 바라보는 논리적인 메모리 블록이다.
 
- **프레임(Frame)** 
  - 물리 메모리(RAM)를 나누는 최소 단위이다. 
  - 페이지와 똑같은 크기로 나누어져 있어, 페이지가 딱 들어맞게 들어갈 수 있는 `틀`과 같다.
  
- **페이징 시스템을 사용하는 이유**
  - 과거에는 프로세스를 메모리에 통째로 올리는 연속 할당 방식을 썻다. 
  - 하지만 이 방식은 메모리 중간중간에 쓸모없는 빈 공간이 생기는 외부 단편화(External Fragmentation) 문제를 일으켰다.
  
- **장점**
  - **외부 단편화 해결**
    - 물리 메모리가 조각나 있어도, 빈 프레임만 있다면 페이지를 어디든 배치할 수 있다.

  - **메모리 공유**
    - 여러 프로세스가 동일한 코드(예: 공통 라이브러리)를 사용할 때, 같은 물리 메모리 프레임을 가리키게 하여 메모리를 절약할 수 있다.

  - **보호와 공유**
    - 페이지 단위로 읽기 전용, 실행 전용 등의 권한을 부여할 수 있어 보안에 유리하다.

- **페이징 시스템의 작동 방식 (주소 변환)**
  - CPU가 메모리에 접근하려고 할 때 사용하는 주소는 논리 주소(Logical Address)이다. 
  - 이 주소는 페이징 시스템에 의해 실제 RAM의 물리 주소(Physical Address)로 바뀐다.
  1. 주소 분할 : 논리 주소를 페이지 번호(p)와 변위(offset, d)로 나눕니다.
  2. 테이블 참조 : 페이지 테이블에서 해당 페이지 번호(p)가 몇 번 프레임(f)에 있는지 찾습니다.
  3. 물리 주소 생성 : 찾아낸 프레임 번호(f)에 변위(d)를 더해 실제 데이터를 가져옵니다.

- **Page 또는 Page Frame**
  - 고정된 크기 block(4KB)
  
- **Paging system**
  - 가상 주소 v = (p, d)
  - p : 가상 메모리 페이지 번호
  - d : p 안에서 참조하는 위치

- **페이지 크기가 4KB인 예시**
  - 가상 주소의 0 비트에서 11비트가 변위(d)를 나타내고, 
  - 12비트 이상이 페이지 번호가 될 수 있다.

- **참고 : 모든 것은 결국 bit와 연결**
  - 본래 프로세스가 4GB를 사용했던 이유 - 32bit 시스템에서의 2의 32승이 4GB
  
- **현업, IT 기술과 컴퓨터 공학의 이해**
  - 윈도우즈 64bit/32bit 시스템

## 페이지 테이블(Page table)

- **정의**
  - 페이지 테이블은 가상 메모리 시스템에서 가상 주소를 물리 주소로 매핑하기 위해 사용하는 데이터 구조이다.
  - 프로세스는 각자 독립적인 가상 메모리 공간을 가지고 있다고 착각하지만, 실제 데이터는 물리 메모리의 여기저기 흩어져 저장된다. 이때 `가상 주소 A는 실제 RAM의 B 지점에 있다.`라고 알려주는 지도 역할을 하는 것이 바로 페이지 테이블이다.
 
- **주소 변환**
  - CPU가 특정 메모리 주소에 접근하려고 할 때, 하드웨어 장치인 MMU가 페이지 테이블을 참조하여 실제 물리 주소를 찾아낸다
  - 가상 주소의 구성 : 페이지 번호(p) + 오프셋(d)
  - 물리 주소 계산 : 페이지 테이블에서 p에 해당하는 페이지 프레임번호(PFN)를 찾아 오프셋 d를 더한다.
  
- **메모리 보호**
  - 페이지 테이블의 각 항목(PTE, Page Table Entry)에는 단순 주소뿐만 아니라, 보호비트(Protection Bits)가 포함된다.
    - Read/Write/Execute : 해당 페이지에 쓰기가 가능한지, 실행만 가능한지 권한을 체크한다.
    - Valid/Invalid bit: 현재 해당 페이지가 실제 메모리에 있는지(Present), 아니면 디스크(Swap)에 있는지 나타낸다.
    
- **페이지 테이블 종류** 
  - **단일 단계 (Single-level)**
    - 특징 : 가장 기본적인 형태. 배열 구조.
    - 장점 : 구현이 단순하고 접근 속도가 빠름.

  - **다단계 (Multi-level)**
    - 특징 : 테이블을 쪼개어 계층 구조로 관리.
    - 장점 : 사용하지 않는 페이지의 테이블은 생성하지 않아 메모리 절약 가능. (현대 OS 표준)
    
  - **역 페이지 테이블 (Inverted)**
    - 특징 : 시스템 전체에 딱 하나만 존재.
    - 장점 : 물리 메모리 크기에 비례하여 테이블 크기가 결정됨.
    
- **필요한 이유**
  - **메모리 단편화 해결**
    - 물리 메모리가 조각나 있어도 사용자에게는 연속된 큰 메모리 공간처럼 보이게 해줍니다.
  
  - **프로세스 격리**
    - 각 프로세스는 자신만의 페이지 테이블을 가지므로, 서로의 메모리 영역을 침범할 수 없어 시스템이 안전해집니다.
  
  - **가상 메모리 구현**
    - 실제 RAM보다 더 큰 용량의 프로그램을 실행할 수 있게 해주는 기초가 됩니다.


## 페이징 시스템과 MMU(컴퓨터 구조)

- CPU는 가상 주소 접근시
  - MMU 하드웨어를 통해 물리 메모리 접근
    
- 프로세스 생성시, 페이지 테이블 정보 생성
  - PCB에서 해당 페이지 테이블 접근 가능하고, 관련 정보는 물리 메모리에 적재.
  
- 프로세스 구동 시, 해당 페이지 테이블 base 주소가 별도 레지스터에 저장(CR3)
  
- CPU가 가상 주소 접근시, MMU가 페이지 테이블 base 주소를 접근해서, 물리 주소를 가져옴.
  
- 참고 : 다중 단계 페이징 시스템
  - 미리 페이징 정보를 만들어놓지 않고 특정 페이징 주소 영역(디렉토리)에 대해서만 만들어 놓는다.
  - 32bit 시스템에서 4KB 페이지를 위한 페이징 시스템은
    - 하위 12bit는 오프셋
    - 상위 20bit가 페이징 번호 이므로 2의 20승 (1048576)개의 페이지 정보가 필요.

- 참고 : 다중 단계 페이징 시스템
  - 페이지 번호를 나타내는 bit를 구분해서, 단계를 나눈다. (리눅스 3단계, 최근 4단계)

## TLB(Translation Lookaside Buffer)

- **정의**
  - 주소 변환 전용 캐시 메모리이다.
  - 페이지 테이블을 참조하는 속도를 획기적으로 높이기 위해 MMU 내부에 두는 아주 빠르고 작은 하드웨어 장치이다.
  
- **TLB가 필요한 이유**
  - MMU가 주소 변환을 하려면 RAM에 있는 페이지 테이블을 읽어야한다. 
  - 하지만 CPU 입장에서는 RAM에 접근하는 것은 매우 느린 작업이다.
  1. CPU가 데이터를 요청한다.
  2. MMU가 주소 변환을 위해 RAM에 있는 페이지 테이블에 접근한다. (첫 번째 RAM 접근)
  3. 변환된 물리 주소로 실제 데이터를 가져오기 위해 다시 RAM에 접근한다. (두 번째 RAM 접근)
  
- **TLB의 동작 원리 (Hit와 Miss)**
  - MMU는 주소 변환을 시작할 때 RAM으로 가기 전에 먼저 TLB부터 확인한다.
  1. TLB Hit (적중) : 찾으려는 페이지 번호와 프레임 번호의 쌍이 TLB에 저장되어 있는 경우이다. RAM에 갈 필요 없이 즉시 주소 변환이 완료된다.
  2. TLB Miss (실패) : TLB에 정보가 없는 경우이다. 이때는 어쩔 수 없이 RAM의 페이지 테이블을 확인해야 하며, 찾은 정보를 다음에 또 쓸 수 있도록 TLB에 업데이트한다.

## CR3(Control Register 3)

- **정의**
  - CR3는 현재 CPU에서 실행 중인 프로세스의 페이지 테이블 주소를 저장하는 레지스터이다. 
  - 구체적으로는 가상 주소를 물리 주소로 변환할 때 가장 먼저 참조해야 하는 최상위 페이지 테이블인 페이지 디렉토리(Page Directory) 또는 PML4(4단계 페이징의 경우)의 물리 주소를 보관한다. 
  - 이 레지스터는 페이지 디렉토리 베이스 레지스터(PDBR, Page Directory Base Register)라고도 불린다.
  
- **핵심 역할**
  - **주소 변환의 시작점 제공**
    - MMU가 가상 주소를 물리 주소로 변환하기 위해서는 반드시 페이지 테이블을 읽어야 한다. 
    - 이때 MMU가 메모리의 수많은 데이터 중 어디서부터 페이지 테이블을 읽기 시작해야 할지 알려주는 이정표 역할을 수행한다.
    
  - **프로세스 간 메모리 격리**
    - 운영체제는 각 프로세스마다 독립적인 페이지 테이블을 할당한다. 
    - 프로세스가 교체되는 문맥 교환(Context Switch)이 일어날 때마다 CR3 레지스터의 값을 해당 프로세스의 페이지 테이블 주소로 변경한다. 
    - 이를 통해 프로세스 A는 프로세스 B의 메모리 영역에 접근할 수 없게 되며, 각 프로세스는 자신만의 고유한 가상 주소 공간을 가질 수 있다.

- **동작 원리**
  - **물리 주소 저장**
    - CR3에 저장되는 주소는 가상 주소가 아닌 물리 주소이다. 
    - 주소 변환을 수행하기 위한 기초 정보인 페이지 테이블의 위치가 다시 가상 주소로 되어 있다면, 이를 변환하기 위해 또 다른 페이지 테이블을 참조해야 하는 무한 루프에 빠지게 되기 때문이다. 
    - 따라서 하드웨어인 MMU가 직접 접근할 수 있도록 실제 RAM의 주소를 기록한다.
    
  - **주소 변환 과정**
    1. CPU가 가상 주소에 접근하려고 하면 MMU가 작동한다.
    2. MMU는 CR3 레지스터에 적힌 물리 주소를 확인하여 메모리에 있는 최상위 페이지 테이블로 이동한다.
    3. 가상 주소의 비트 정보를 해석하여 최상위 테이블부터 하위 테이블까지 순차적으로 탐색한다.
    4. 최종적으로 찾아낸 물리 주소의 프레임에 접근하여 데이터를 가져온다.
    
  - **문맥 교환과 TLB의 관계**
    - 프로세스가 바뀌어 CR3의 값이 갱신되면, 이전에 실행되던 프로세스의 주소 변환 정보는 더 이상 유효하지 않게 된다. 
    - 따라서 CR3 레지스터에 새로운 값이 기록되는 순간, CPU 내부에 저장되어 있던 주소 변환 캐시인 TLB(Translation Lookaside Buffer)의 내용들은 일반적으로 모두 삭제(Flush)된다. 
    - 이는 새로운 프로세스가 이전 프로세스의 메모리 주소를 오용하는 것을 방지하기 위한 보안 및 정확성 유지 조치이다.



## 페이징 시스템과 공유 메모리(IPC)

- **공유 메모리 정의**
  - 공유 메모리는 프로세스 간 통신(IPC, Inter-Process Communication) 기법 중 하나로, 말 그대로 둘 이상의 프로세스가 동일한 물리 메모리 영역을 함께 사용하는 방식이다.
  - 원래 가상 메모리 시스템은 프로세스 A의 메모리를 프로세스 B가 절대 볼 수 없게 설계되어 있지만, 운영체제에게 `이 특정 영역만큼은 우리가 같이 쓸게`라고 허락을 받는 것이 핵심이다.
  
- **페이징 시스템을 이용한 동작 원리**
  - 공유 메모리가 마법처럼 갑자기 생겨나는 것은 아니다. 우리가 배운 MMU와 페이지 테이블의 설정을 살짝 바꾸는 것만으로 구현된다.
    1. 운영체제가 물리 메모리(RAM)의 특정 페이지 프레임(Physical Frame) 하나를 선정한다.
    2. 프로세스 A의 페이지 테이블에 해당 물리 주소를 등록한다.
    3. 프로세스 B의 페이지 테이블에도 똑같은 물리 주소를 등록한다.
    
  - 결과적으로 두 프로세스는 서로 다른 가상 주소를 보고 있지만, 실제로 데이터를 읽고 쓰는 곳은 동일한 RAM의 한 지점이 된다.
  
- **공유 메모리를 사용하는 이유**
  - IPC 기법에는 소켓(Socket), 파이프(Pipe), 메시지 큐(Message Queue) 등 여러 가지가 있지만, 공유 메모리는 압도적인 성능 때문에 사용한다.
  
  - **최고의 속도(O(1)에 수렴)**
    - 다른 IPC 기법들은 데이터를 보낼 때 '커널 공간'으로 복사했다가 다시 받는 과정(Copy)이 필요하다. 
    - 하지만 공유 메모리는 내 방에 있는 물건을 옆 사람도 같이 만지는 격이라 데이터 복사가 전혀 일어나지 않는다.
    
  - **대용량 데이터 처리**
    - 수 GB 단위의 데이터를 프로세스 간에 옮겨야 한다면 복사 비용이 너무 크다. 
    - 공유 메모리는 주소만 연결하면 끝이기에 대규모 데이터 처리에 최적이다.
    
- **트레이드 오프**
  - **성능**
    - 장점 : 데이터 복사가 없어 IPC 중 가장 빠름
    - 단점 : 동기화(Synchronization) 메커니즘을 직접 구현해야 함
    
  - **유연성**
    - 장점 : 실시간 데이터 공유 가능
    - 단점 : 한 프로세스의 잘못된 쓰기가 공유 영역 전체를 오염시킴
        
## 요구 페이징(Demand paging 또는 Demanded paging)

- **정의**
  - 요구 페이징은 이름 그대로 실행 중인 프로세스가 특정 페이지를 요구할 때만 해당 페이지를 물리 메모리(RAM)에 올리는 방식이다.
  - 과거에는 프로그램을 실행하려면 실행 파일 전체를 메모리에 다 올려야 했다. 
  - 하지만 요구 페이징 시스템에서는 프로그램의 시작 부분 등 당장 필요한 아주 일부만 메모리에 올리고, 나머지는 하드디스크(Swap 영역)에 그대로 둔다. 
  - 그러다 실행 중에 메모리에 없는 코드가 필요해지는 순간, 운영체제가 개입하여 그 부분만 메모리로 가져온다.
  
- **동작 원리**
  - CPU가 명령어를 실행하다가 메모리에 없는 페이지에 접근하려고 할 때, 시스템은 다음과 같은 정교한 과정을 거친다.
  
  1. 접근 시도 : CPU가 MMU에 가상 주소를 전달한다.
  2. 상태 확인 : MMU가 페이지 테이블을 확인한다. 이때 해당 페이지가 메모리에 없음을 나타내는 유효-무효 비트(Valid-Invalid bit)가 무효(Invalid)로 설정되어 있음을 발견한다.
  3. 페이지 폴트(Page Fault) 발생 : MMU는 CPU에 트랩(Trap)을 걸어 운영체제에 도움을 요청한다.
  4. 디스크 탐색 : 운영체제는 하드디스크(보조 기억장치)의 스왑 영역에서 해당 페이지를 찾는다.
  5. 페이지 로드 : 물리 메모리의 빈 프레임(Free Frame)을 찾아 디스크의 내용을 메모리로 복사한다.
  6. 테이블 갱신 : 페이지 테이블의 비트를 유효(Valid)로 바꾸고, CPU에게 작업을 다시 시작하라고 알려준다.
  
- **요구 페이징을 사용하는 이유**
  - 이 방식은 특히 대규모 엔터프라이즈 환경에서 강력한 힘을 발휘한다.
  
  - **메모리 절약**
    - 프로그램의 모든 기능을 다 쓰지 않는 경우가 많다. 
    - 예를 들어, 워드 프로그램의 인쇄 기능은 인쇄 버튼을 누를 때만 메모리에 올리면 된다. 
    - 이를 통해 더 많은 프로그램을 동시에 띄울 수 있다.
    
  - **빠른 시작 속도**
    - 프로그램 전체를 다 읽어올 때까지 기다리지 않고, 첫 화면에 필요한 페이지만 로드되면 바로 실행할 수 있다. 
    - 사용자 경험(UX) 측면에서 응답성이 비약적으로 좋아진다.
    
  - **물리적 한계 극복**
    - 실제 RAM 용량보다 훨씬 큰 프로그램도 문제없이 실행할 수 있다.
    
- **트레이드 오프**
  - 자원 효율성
    - 장점 : 물리 메모리 사용량 최소화
    - 단점 : 하드디스크 I/O 발생으로 인한 성능 저하
    
  - 시스템 유연성
    - 장점 : 더 많은 프로세스 동시 실행 가능
    - 단점 : 페이지 폴트 처리 시 CPU 시간 소모
    
- 요구 페이징을 사용하면 메모리는 아끼지만, 페이지 폴트가 발생할 때마다 느린 디스크에 다녀와야 하므로 그 순간만큼은 시스템이 멈춘 것처럼 느껴질 수 있다.

- **주의사항: 스래싱(Thrashing) 현상**
  - 물리 메모리가 너무 부족한 상황에서 요구 페이징을 과하게 사용하면, 운영체제는 페이지를 교체하느라 바빠서 실제 연산은 거의 하지 못하는 상태에 빠진다. 
  - 즉, CPU 사용률은 바닥을 치는데 디스크 읽기/쓰기 신호만 가득한 상태가 된다. 
  - 비즈니스 관점에서는 서비스 응답이 완전히 멈춰버리는 대장애로 이어질 수 있다.
  - 이를 방지하기 위해 우리는 프로세스마다 적절한 수의 페이지 프레임을 할당하거나, 페이지 교체 알고리즘(LRU 등)을 최적화하는 작업을 수행한다.
  
## 스왑 영역(Swap Area)

- **정의**
  - 스왑 영역(Swap Area)은 물리적인 메모리(RAM)의 용량이 부족할 때를 대비하여, 하드디스크나 SSD 같은 보조 기억장치의 일부를 마치 RAM처럼 사용하는 공간이다.

- **사용하는 이유**
  - 물리 메모리는 속도가 매우 빠르지만 용량이 제한적이며 가격이 비싸다는 단점이 있다. 반면 보조 기억 장치는 속도는 느리지만 저렴하고 대용량 구현이 용이하다. 
  - 운영체제는 이러한 하드웨어적 특성을 활용하여, 현재 실행 중인 프로세스들이 요구하는 전체 메모리 양이 실제 RAM 용량을 초과할 때, 당장 사용되지 않는 메모리 페이지들을 보조 기억 장치의 특정 구역으로 옮겨 관리한다. 
  - 이 구역이 바로 스왑 영역이다.

- **핵심 동작 원리**
  - **스왑 아웃(Swap-out)**
    - 물리 메모리가 부족해지면 운영체제는 페이지 교체 알고리즘(LRU 등)을 통해 현재 참조 빈도가 낮은 페이지를 선정한다. 
    - 해당 페이지의 데이터를 스왑 영역에 기록한 뒤, RAM에서 해당 프레임을 비운다. 이때 페이지 테이블의 유효 비트는 무효(Invalid)로 수정된다.
    
  - **스왑 인(Swap-in)**
    -  CPU가 가상 주소를 통해 특정 데이터에 접근하려 할 때, 페이지 테이블의 유효 비트가 무효 상태이고 해당 데이터가 스왑 영역에 존재함이 확인되면 페이지 부재(Page Fault)가 발생한다. 
    - 운영체제는 스왑 영역에서 해당 데이터를 읽어 다시 물리 메모리의 빈 프레임에 적재한다.

- **주요 역할**
  - **메모리 과할당(Over-commitment) 허용**
    - 실제 설치된 RAM 용량보다 더 큰 메모리 주소 공간을 프로세스에 제공할 수 있게 한다. 
    - 이를 통해 시스템은 물리적 한계를 넘어 더 많은 프로그램을 동시에 실행할 수 있는 유연성을 확보한다.
    
  - **시스템 안정성 유지**
    - 프로세스가 갑자기 많은 메모리를 요구할 때, 스왑 영역이 존재하지 않으면 운영체제는 메모리 부족(OOM, Out Of Memory) 상태에 빠져 프로세스를 강제로 종료해야 한다. 
    - 스왑 영역은 이러한 상황에서 완충 작용을 하여 시스템의 급격한 붕괴를 방지한다.

## 페이지 폴트(Page fault)

- **정의**
  - 페이지 폴트는 프로그램이 가상 메모리 주소에는 존재하지만, 실제 물리 메모리(RAM)에는 현재 올라와 있지 않은 데이터에 접근하려고 할 때 발생하는 하드웨어 인터럽트(또는 트랩)이다.
  - `Fault(오류)`라는 단어 때문에 프로그램이 잘못된 것처럼 느껴질 수 있지만, 사실 이는 요구 페이징 시스템에서 지극히 정상적이고 의도된 과정이다. 
  - 도서관에서 책을 찾았는데 서가에 없고 '대출 중' 혹은 '창고 보관 중'인 상태를 확인하는 것과 같다.
  
- **페이지 폴트의 동작 원리** 
  - CPU가 메모리에 접근하려다 페이지 폴트가 발생하면, 시스템은 매우 정교한 복구 작업을 시작한다.
  1. **접근 시도**
     - CPU가 특정 가상 주소를 참조한다.
  2. **페이지 테이블 확인**
     - MMU(Memory Management Unit)가 페이지 테이블을 본다. 
     - 해당 페이지의 상태 비트가 무효(Invalid)임을 확인한다.
  3. **트랩(Trap) 발생**
     - MMU가 CPU에 신호를 보내 운영체제(OS)를 호출한다. 
     - 현재 실행 중인 작업을 잠시 멈춘다.
  4. **디스크 검색**
     - 운영체제는 하드디스크의 스왑 영역(Swap Area)에서 요청된 페이지를 찾는다.
  5. **페이지 로드 (I/O)**
     - 물리 메모리의 빈 공간(Free Frame)을 찾아 디스크에 있는 데이터를 RAM으로 복사한다. (이 과정이 가장 오래 걸린다.)
  6. **테이블 갱신 및 재시작**
     - 페이지 테이블을 유효(Valid) 상태로 업데이트하고, 아까 멈췄던 명령어를 다시 실행한다.

- **트레이드 오프**
  - **자원 효율**
    - 장점 : 실제 필요한 데이터만 RAM에 두므로 메모리 공간 극대화
    - 단점 : 페이지 폴트 발생 시 디스크 접근으로 인한 급격한 성능 저하
    
  - **시스템 유연성**
    - 장점 : 물리 메모리 크기보다 훨씬 큰 프로그램 실행 가능
    - 단점 : 폴트가 너무 잦으면 CPU가 연산보다 데이터 이동에 더 많은 시간 소모
    
 - **실무적 주의사항**
   - **참조 국소성 (Locality of Reference) 위반**
     - 프로그램이 메모리의 여기저기를 무작위로 쑤시고 다닌다면 페이지 폴트가 빈번해진다. 
     - 효율적인 코드는 데이터가 모여 있는 곳을 집중적으로 사용하도록 설계되어야 한다.
     
   - **스래싱 (Thrashing)**
     - 메모리가 너무 부족해서 페이지를 가져오자마자 다른 페이지를 내보내야 하는 상황이다. 
     - 페이지 폴트 -> 디스크 읽기 -> 다른 페이지 폴트가 무한 반복되며 CPU 사용률은 높지만 실제 업무 처리량은 0에 수렴하게 된다.
     
- **결론**
  - 페이지 폴트 처리 시간은 메모리 접근 시간(나노초 단위)에 비해 수천 배에서 수만 배(밀리초 단위) 더 느리다.
  - 페이지 폴트 확률 p가 아주 조금만 높아져도 시스템 전체의 응답 속도는 기하급수적으로 느려진다. 
  - 따라서 저희 프로그래머들은 이 p를 최소화하기 위해 알고리즘을 최적화하고 적절한 RAM 용량을 산정합니다.
  
## 페이지 폴트와 인터럽트

![](https://velog.velcdn.com/images/atoonatoo/post/e245b0b2-80c0-47d1-9622-7ac96923d665/image.png)


- **인터럽트(Interrupt) 정의**
  - 인터럽트는 단어 뜻 그대로 '방해하다' 혹은 '중단시키다'라는 의미를 가진다. 
  - CPU가 프로그램을 실행하고 있을 때, 예기치 않은 상황이나 긴급한 요청이 발생하면 현재 작업을 잠시 멈추고 해당 사건을 처리하도록 하는 신호 체계이다.
  - 비유하자면, 요리사(CPU)가 요리를 하다가 벨 소리(인터럽트)가 울리면 가스불을 줄이고 문을 열어주는 것과 같다.
  
- **페이지 폴트와 인터럽트의 연결 고리**
  - 페이지 폴트는 수많은 인터럽트 종류 중에서도 소프트웨어에 의해 발생하는 '내부 인터럽트(Internal Interrupt)', 더 정확하게는 트랩(Trap) 또는 예외(Exception)의 일종이다.
  - CPU가 코드를 실행하다가 메모리에 데이터가 없는 상황을 맞닥뜨리면 스스로 `어? 데이터가 없네!`라고 외치며 운영체제에게 신호를 보낸다. 
  - 이것이 바로 페이지 폴트라는 이름의 인터럽트가 발생하는 순간이다.
  
- **동작 원리와 흐름**
  1. 명령어 실행 중 페이지 폴트 발생 (데이터 부재 확인)
  2. CPU의 현재 상태(레지스터 값 등)를 저장
  3. 인터럽트 벡터를 참조하여 페이지 폴트 핸들러(운영체제의 처리 루틴)로 제어권 전환
  4. 운영체제가 디스크에서 페이지를 가져오는 작업 수행
  5. 작업 완료 후 중단되었던 명령어 지점으로 복귀
  
- **인터럽트의 분류와 페이지 폴트의 위치**

  - **외부 인터럽트 (Hardware)**
    - 발생 원인 : 입출력 장치, 타이머 등 외부 하드웨어
    - 예시 : 키보드 입력, 마우스 클릭, 네트워크 패킷 도착
    
  - **내부 인터럽트 (Software/Exception)**
    - 발생 원인 : 프로그램 실행 중 발생하는 예외 상황
    - 예시 : 0으로 나누기, 권한 없는 메모리 접근, 페이지 폴트
    
 - 페이지 폴트는 프로그램 실행 흐름 속에서 필연적으로 발생하므로 동기적(Synchronous)인 특성을 가진다. 
 - 반면 마우스 클릭 같은 외부 인터럽트는 언제 발생할지 알 수 없는 비동기적(Asynchronous)인 성격을 가진다.
 
- **트레이드 오프**
  - 인터럽트 기반의 페이지 폴트 처리는 자원 효율성을 극대화하지만, 그 이면에는 비용이 존재한다.
  
  - **문맥 교환(Context Switch) 비용**
    - CPU가 하던 일을 멈추고 운영체제의 모드로 전환했다가 다시 돌아오는 과정에서 시간이 소모됩니다.
    
  - **I/O 대기 시간**
    - 이지 폴트 처리의 핵심인 디스크 접근은 CPU 속도에 비해 수백만 배 느리다.
    
- **주의사항**

  - **인터럽트 폭주** 
    - 페이지 폴트가 너무 빈번하면 CPU는 실제 비즈니스 로직을 처리하는 시간보다 인터럽트를 처리하고 문맥을 교환하는 데 더 많은 에너지를 쓰게 된다.
    
  - **실시간성 저하**
    - 인터럽트 처리는 현재 작업을 중단시키므로, 응답 시간이 극도로 중요한 실시간 시스템(Real-time System)에서는 페이지 폴트 발생 확률을 최소화하도록 메모리를 미리 할당(Memory Locking)하기도 한다.

- **요약 및 결론**
  - 페이지 폴트는 운영체제가 메모리를 아끼기 위해 의도적으로 허용한 인터럽트이다.   - 하지만 이 방해가 너무 잦아지면 시스템 전체의 성능이 무너지는 임계점에 도달하게 된다.

## 사용자/커널 모드와 프로세스, 인터럽트

![](https://velog.velcdn.com/images/atoonatoo/post/da28ab25-d81d-4aa4-8a76-b6e38fd8f456/image.png)

- **사용자 모드(User Mode)와 커널 모드(Kernel Mode)**
  - **정의**
    - 현대 CPU는 시스템의 붕괴를 막기 위해 실행 권한을 두 가지 모드로 엄격히 분리하여 운영한다. 이를 이중 모드(Dual-mode) 동작이라고 한다.
  
  - **사용자 모드**
    - 우리가 흔히 사용하는 일반 애플리케이션(브라우저, 게임, 엑셀 등)이 실행되는 모드이다. 시스템 자원에 직접 접근할 수 없으며, 제한된 명령만 수행 가능하다.
  - **커널 모드**
    - 운영체제의 핵심(Kernel)이 실행되는 모드이다. 
    - CPU의 모든 명령어를 실행할 수 있고, 메모리나 하드웨어 같은 물리적 자원을 직접 제어할 수 있는 무소불위의 권한을 가진다.
  
  - **동작 원리 : 시스템 콜(System Call)**
    - 사용자 모드에서 실행 중인 프로그램이 파일을 저장하거나 네트워크 데이터를 보내는 등 하드웨어 자원이 필요하면, 스스로 처리하지 못하고 운영체제에 `대신 해달라고` 요청해야 한다. 
    - 이 요청 통로가 바로 시스템 콜이다. 시스템 콜이 발생하면 CPU는 사용자 모드에서 커널 모드로 전환(Mode Switch)된다.

- **프로세스(Process)**
  - **정의**
    - 프로세스는 단순히 하드디스크에 저장된 프로그램(정적 상태)이 아니라, 메모리에 올라와 실행 중인 상태(동적 상태)를 의미이다.

  - **동작 원리**
    - 운영체제는 프로세스를 관리하기 위해 PCB(Process Control Block)라는 일종의 '인적 사항 보고서'를 만든다.
    - 여기에는 프로세스 ID, 상태, CPU 레지스터 값 등이 기록된다.
  - **프로세스 메모리 구조**
    - Code(Text) : 실행할 코드가 저장된 영역
    - Data : 전역 변수 등이 저장된 영역
    - Heap : 실행 중 동적으로 할당되는 영역
    - Stack : 함수 호출 시 지역 변수 등이 저장되는 영역
    
- **인터럽트(Interrupt)**    

  - **정의**
    - CPU가 프로그램을 실행하고 있을 때, 입출력 완료나 예외 상황 발생 등 긴급한 처리가 필요함을 알리는 신호입이다.
    
  - **동작 원리**
    - 인터럽트가 발생하면 CPU는 하던 일을 멈추고 현재 상태를 PCB에 저장한 뒤, 인터럽트 서비스 루틴(ISR)이라는 특수 코드를 커널 모드에서 실행한다. 
    - 처리가 끝나면 다시 원래의 프로세스로 돌아온다.
    
- **주의 사항**    
  - **시스템 콜 최소화** 
    - 고성능 서버를 설계할 때는 시스템 콜(커널 모드 진입) 횟수를 줄이는 것이 핵심이다. 
    - 예를 들어, 데이터를 1바이트씩 1,000번 보내는 것보다 1,000바이트를 한 번에 보내는 것이 효율적이다.
    
  - **권한 관리**
    - 보안 사고의 대부분은 관리자 권한(커널 수준 권한)이 탈취될 때 발생한다. 
    - 따라서 사용자 모드에서 실행되는 앱이 최소한의 권한만 가지도록 설계하는 것이 '최소 권한의 원칙'이다.

## 인터럽트와 IDT

- **IDT(Interrupt Descriptor Table)정의**
  - 컴퓨터 과학에서 IDT는 인터럽트와 예외 상황을 처리하기 위한 정보를 담고 있는 거대한 안내 지도 또는 인덱스 테이블이라고 이해하시면 된다.
  - CPU는 수많은 인터럽트 요청을 받는다. 키보드 입력, 마우스 클릭, 0으로 나누기 오류, 페이지 폴트 등 종류가 매우 다양하다. 
  - 이때 CPU는 각각의 번호(Vector)를 가지고 있는데, 이 번호를 들고 `이 번호는 어디로 가서 처리해야 하지?`라고 물었을 때 답을 주는 곳이 바로 IDT이다.
  
- **인터럽트와 IDT의 연결 메커니즘 (작동 원리)**
  - 사건이 발생해서 처리가 완료되기까지의 과정을 현업에서는 다음과 같은 흐름으로 파악한다.
  
  1. **인터럽트 발생 및 벡터 번호 생성**
    - 하드웨어나 소프트웨어가 CPU에 신호를 보낸다. 
    - 이때 각 사건에는 고유한 번호가 매겨져 있는데, 이를 인터럽트 벡터(Interrupt Vector)라고 부른다. 
    - 예를 들어, 0으로 나누기 오류는 0번, 페이지 폴트는 14번과 같은 식이다.
    
  2. **IDTR 레지스터 참조**
    - CPU 내부에는 IDTR(IDT Register)이라는 특수한 저장 공간이 있다. 
    - 이곳에는 메모리상에 IDT가 어디에 위치해 있는지(Base Address)와 그 크기가 저장되어 있다. 
    - CPU는 사건이 터지자마자 이 주소를 확인한다.
    
  3. **IDT 인덱싱 및 게이트(Gate) 확인**
    - CPU는 전달받은 벡터 번호를 인덱스로 사용하여 IDT의 특정 항목을 찾아간다.
    - 각 항목은 인터럽트 게이트(Interrupt Gate)라고 불리는 구조체로 되어 있으며, 여기에는 실제 처리 코드인 ISR(Interrupt Service Routine)이 위치한 메모리 주소와 실행 권한 정보가 담겨 있다.
    
  4. **모드 전환 및 ISR 실행**
    - 사용자 모드에서 실행 중이었다면 커널 모드로 전환하고, 찾아낸 주소로 점프하여 실제 처리 로직(ISR)을 실행한다.
  
- **트레이드 오프**
  - **성능**
    - 장점 : 인덱스를 통한 직접 접근이므로 O(1)의 속도로 처리 루틴을 찾음
    - 단점 : 부팅 시 운영체제가 메모리에 IDT를 미리 구성해두어야 하는 초기화 비용 발생
  - **유연성**
    - 장점 : 새로운 하드웨어나 예외 상황이 추가되어도 테이블의 내용만 수정하면 됨
    - 단점 : 잘못된 주소를 적어둘 경우 시스템 전체가 마비되는 치명적 오류 발생 가능성
    
 - **주의 사항**
   - **잘못된 핸들러 설정**
     - 만약 IDT의 특정 항목에 엉뚱한 주소를 넣어두면, 해당 인터럽트가 발생하는 순간 시스템은 갈 길을 잃고 멈춰버린다. 
     - 이를 더블 폴트(Double Fault) 혹은 트리플 폴트(Triple Fault)라고 하며, 이는 곧 서버의 완전한 다운을 의미한다.
     
   - **보안 공격**
     - 과거에는 해커들이 IDT의 주소를 조작하여 악성 코드가 커널 권한으로 실행되게 만드는 공격(IDT Hooking)을 시도하기도 했다. 
     - 현대 운영체제는 이를 막기 위해 IDT 영역을 읽기 전용으로 보호하는 기법을 사용한다.
     
   - **우선순위 관리**
     - 수많은 인터럽트가 동시에 발생할 수 있으므로, 어떤 것을 먼저 처리할지에 대한 우선순위 제어가 IDT 수준이 아닌 인터럽트 컨트롤러(APIC 등)와 연계되어 정교하게 설계되어야 한다.
    
- **요약 및 결론**
  - 인터럽트가 사건이라면, IDT는 그 사건에 대처하는 비상 연락망 명부이다. 
  - CPU는 이 명부를 보고 하던 일을 멈추고 전문가(ISR)에게 일을 맡긴다. 
  - 이 연결 고리가 탄탄해야만 우리가 사용하는 시스템이 끊김 없이 안전하게 동작할 수 있다.
  
## 생각해보기

- **페이지 폴트가 자주 발생되면?**
  - 페이지 폴트가 빈번해지면 시스템은 스래싱(Thrashing)이라는 최악의 상태에 빠진다. 
  - **성능의 수직 낙하**
    - 물리 메모리(RAM) 접근은 나노초(ns) 단위지만, 디스크(SSD/HDD) 접근은 밀리초(ms) 단위다. 
    - 페이지 폴트가 발생할 때마다 CPU는 디스크가 데이터를 가져올 때까지 수백만 사이클을 놀면서 기다려야 한다.
    
  - **CPU 이용률의 역설**
    - 재미있는 현상은, 시스템이 느려지는데 CPU 이용률은 100%에 가깝게 찍힐 수 있다는 점이다. 
    - 하지만 이건 비즈니스 로직을 처리하는 게 아니라, `어떤 페이지를 뺄까? 어떤 걸 넣을까?` 고민하고 인터럽트를 처리하는 운영체제의 오버헤드 때문에 바쁜 것이다. 
    - 정작 우리가 만든 프로그램은 아무것도 못 하고 있다.
    
  - **서비스 중단 (Timeout)**
    - 응답 속도가 기하급수적으로 느려지기 때문에, 웹 서버라면 클라이언트에게 Timeout 에러를 보낸다. 
    - 비즈니스 관점에서는 매출 손실로 직결되는 순간이다.
    
- **페이지 폴트가 발생되지 않게 하려면?**
  - 이 문제를 해결하는 방법은 크게 하드웨어적 접근과 소프트웨어적(공학적) 접근 두 가지로 나뉜다.
  
  - **하드웨어적 해결 : RAM 증설 (Scale-up)**
    - 가장 확실하고 돈이 많이 드는 방법이다. 
    - 프로세스가 실제로 활발하게 사용하는 메모리 덩어리인 워킹 셋(Working Set)이 RAM 크기보다 작아지도록 물리적인 공간을 넓혀주는 것이다.
    
  - **소프트웨어적 해결: 참조 국소성(Locality) 최적화**
    - 숙련된 프로그래머는 코드를 짤 때 메모리 주소를 "가깝게" 사용하도록 설계한다. 
    - 이를 참조 국소성이라고 한다.
    
    - **시간적 국소성(Temporal Locality)**
      - 한 번 쓴 데이터는 조만간 또 쓸 가능성이 높으니 계속 들고 있게 설계한다.
      
    - **공간적 국소성(Spatial Locality)**
      - 어떤 데이터를 읽었다면, 그 바로 옆에 있는 데이터도 곧 읽을 가능성이 높다.
      
    - **실무 예시**
      - 2차원 배열을 처리할 때, 행(Row) 방향으로 읽느냐 열(Column) 방향으로 읽느냐에 따라 페이지 폴트 횟수가 수십 배 차이 날 수 있다. 
      - 현대 CPU와 OS는 옆집 데이터를 미리 가져오는 특징이 있기 때문이다.
      
 - **페이지 교체 알고리즘 최적화**
   - 운영체제 레벨에서 `어떤 페이지를 버릴 것인가`를 잘 결정해야 한다.
   
   - **LRU (Least Recently Used)**
     - 가장 오랫동안 안 쓴 놈을 버린다. 
     - 현업에서 가장 많이 쓰이는 합리적인 전략이다.
     
   - **LFU (Least Frequently Used)**
     - 가장 적게 사용된 놈을 버린다.
    
 - **메모리 락 (Memory Locking / Pinning)**
   - 금융 시스템이나 자율주행처럼 0.1초의 지연도 허용할 수 없는 시스템에서는, 특정 코드나 데이터를 `절대 디스크로 내보내지 말고 RAM에 박아둬(Pinning)`라고 운영체제에 강제로 명령한다.
   
## 페이지 교체 정책(Page Replacement policy)

- **정의**
  - 요구 페이징(Demand Paging) 시스템에서 페이지 폴트가 발생했을 때, 이를 수용할 빈 프레임(Free Frame)이 없다면 현재 물리 메모리에 올라와 있는 페이지 중 하나를 선택해 하드디스크로 쫓아내고(Swap-out), 그 자리에 새로운 페이지를 올리는 전략을 말한다.

- **용도**
  - 물리 메모리의 한계를 극복하고 시스템의 효율성을 극대화하기 위해 사용한다.
  - 어떤 페이지를 내보내느냐에 따라 이후 발생하는 페이지 폴트 횟수가 결정되므로, 시스템 전체의 성능(Throughput)과 직결된다.

- **주요 페이지 교체 알고리즘**
  - **FIFO (First-In, First-Out)**
    - 메모리에 가장 먼저 들어온 페이지를 가장 먼저 내보낸다.
    - 구현이 매우 간단하지만, 오래 있었다고 해서 안 쓰는 페이지는 아니라는 맹점이 있다.
    
  - **OPT (Optimal)**
    - 앞으로 가장 오랫동안 사용되지 않을 페이지를 찾아 내보낸다.
    - 이론상 페이지 폴트가 가장 적은 최적의 방법이지만, 미래를 예측할 수 없기에 구현이 불가능한다. (비교 기준용)
    
  - **LRU (Least Recently Used)**
    - 최근에 가장 오랫동안 사용되지 않은 페이지를 내보낸다.
    - '최근에 썼으면 곧 또 쓰겠지'라는 시간 국소성(Temporal Locality) 원리를 이용하며, 실무에서 가장 널리 쓰인다.
    
  - **LFU (Least Frequently Used)**
    - 참조 횟수가 가장 적은 페이지를 내보낸다.
    - 과거에 많이 썼던 페이지는 앞으로도 쓸 것이라는 가정에 기반한다.
    
- **트레이드 오프**
  - **알고리즘의 정확도 vs 오버헤드**
    - OPT에 가까운 정교한 알고리즘(예: LRU)을 사용할수록 페이지 폴트는 줄어들지만, 이를 관리하기 위해 하드웨어 지원이 필요하거나 CPU 연산 시간이 추가로 소모된다.
    
  - **구현의 단순함 vs 성능
    - FIFO는 구현이 매우 쉽지만, 메모리를 늘려주었는데 오히려 페이지 폴트가 늘어나는 Belady's Anomaly라는 기현상을 겪을 수 있다.  
    
- **주의 사항**    
  - **페이지 교체 비용의 비대칭성**
    - 깨끗한 페이지(수정되지 않은 페이지)를 내보낼 때는 디스크에 쓸 필요가 없어 빠르지만, 수정된(Dirty) 페이지를 내보낼 때는 디스크 쓰기 작업이 동반되어 훨씬 느리다.
    - 따라서 수정 여부를 나타내는 더티 비트(Dirty Bit)를 확인하여 교체 우선순위를 조정하는 것이 효율적이다.
    
  - **스래싱(Thrashing)의 위험**
    - 어떤 알고리즘을 써도 물리 메모리 자체가 절대적으로 부족하면 시스템은 페이지를 교체하느라 바빠서 실제 일을 못 하게 된다.
    - CPU 사용률이 급격히 떨어지면서 I/O만 폭증한다면 알고리즘의 문제가 아니라 RAM 증설이나 프로세스 개수 조절이 필요한 시점이다.
  
- **요약 및 결론**
  - 페이지 교체 정책은 한정된 자원을 누가 더 오래, 더 가치 있게 점유할 것인가를 결정하는 '자원 분배의 지혜'이다. 
  - 실무에서는 대부분 LRU나 그 파생 알고리즘(Clock Algorithm 등)을 사용하여 O(1) 혹은 O(n) 수준의 비용으로 최적의 성능을 낸다.

## 페이지 교체 알고리즘(FIFO)

- **정의**
  - FIFO 알고리즘은 이름 그대로 메모리에 가장 먼저 들어온 페이지를 가장 먼저 내보내는 방식이다. 
  - 페이지가 메모리에 적재된 시간을 기록하여, 가장 오래된 페이지를 교체 대상(Victim Page)으로 선정한다.
  
- **용도**
  - 주로 복잡한 계산이 필요 없는 단순한 운영체제나, 메모리 관리 부하를 최소화해야 하는 소규모 임베디드 시스템에서 사용된다. 
  - 구현이 매우 쉽기 때문에 알고리즘의 기준점(Baseline)으로 자주 인용된다.
  
- **동작 원리**
  - FIFO는 큐(Queue) 자료구조를 사용하여 동작한다.
  1. 새로운 페이지가 메모리에 들어오면 큐의 맨 뒤(Tail)에 삽입한다.
  2. 페이지 폴트가 발생하고 메모리가 가득 찬 상태라면, 큐의 맨 앞(Head)에 있는 페이지를 제거한다.
  3. 제거된 빈 자리에 새로운 페이지를 넣고 다시 큐의 맨 뒤에 추가한다.

- **트레이드 오프**
  - 구현이 매우 간단하고 프로그래밍 오버헤드가 적음.
  - 페이지의 실제 사용 빈도를 전혀 고려하지 않음.
  
- **주의사항 : 벨레이디의 모순 (Belady's Anomaly)**
  - 상식적으로 메모리(Frame)를 늘리면 페이지 폴트가 줄어들어야 할 것 같지만, FIFO에서는 오히려 페이지 폴트가 늘어나는 현상이 발생할 수 있다.
  
  - **원인**
    - 페이지 교체 순서가 실제 사용 패턴과 맞지 않아, 메모리가 늘어났음에도 불구하고 당장 필요한 페이지가 계속 교체 대상이 되기 때문이다.
    
  - **시사점**
    - 단순히 하드웨어(RAM) 사양을 높이는 것만으로는 시스템 성능이 선형적으로 개선되지 않을 수 있음을 시사한다.
    
## 페이지 교체 알고리즘(OPT)

- **정의**
  - OPT 알고리즘은 앞으로 가장 오랫동안 사용되지 않을 페이지를 찾아내어 교체하는 방식이다. 
  - 이론적으로 존재할 수 있는 모든 페이지 교체 알고리즘 중 가장 낮은 페이지 폴트율을 보장힌다.
  
- **용도**
  - 현실적으로 미래를 예측하는 것은 불가능하기 때문에 실제 운영체제에서 구현하여 사용할 수는 없다. 
  - 대신, 다른 알고리즘(LRU, FIFO 등)이 얼마나 효율적인지 측정하기 위한 비교 기준(Benchmark)으로 사용된다.
  - 우리가 만든 알고리즘이 이론적 최적값에 얼마나 근접했는가?를 판단하는 척도가 된다.
  
- **동작 원리**
  - OPT의 핵심은 과거가 아닌 미래를 바라본다는 점이다.
  
  1. 페이지 폴트가 발생하여 메모리에 공간이 필요할 때, 현재 메모리에 있는 페이지들의 미래 참조 계획을 전수 조사한다.
  2. 각 페이지가 앞으로 언제 다시 사용될지 계산한다.
  3. 그중 가장 먼 미래에 사용될 페이지, 혹은 앞으로 영원히 사용되지 않을 페이지를 선택하여 메모리에서 내보낸다.
  
- **트레이드 오프**
  - 장점
    - 이론상 최소의 페이지 폴트 횟수 보장
    - 시스템 성능의 상한선(Upper Bound) 제시
    
  - 단점
    - 미래의 참조 스트링을 미리 알아야 함
    - 실제 시스템 구현이 물리적으로 불가능함
    
- **주의사항**
  - **예측 불가능성**
    - 실제 컴퓨터 시스템에서 프로세스가 미래에 어떤 메모리 주소를 읽을지 미리 아는 것은 불가능하다. 
    - 마치 주식 시장의 내일 종가를 오늘 알 수 없는 것과 같다.
    
  - **벤치마킹의 한계**
    - 특정 프로그램의 실행 기록(Trace)을 미리 추출하여 사후 분석을 할 때만 OPT를 계산할 수 있다. 
    - 따라서 실시간 서비스의 성능 최적화 도구로 제안된다면, 그것은 구현 불가능한 가설임을 인지해야 한다.
    
  - **벨레이디의 모순 해결**
    - FIFO와 달리, OPT는 메모리 프레임 수가 늘어날 때 페이지 폴트가 증가하는 역설적인 현상이 절대 발생하지 않는다.
    
## 페이지 교체 알고리즘(LRU)

- **정의**
  - LRU는 메모리에 올라온 페이지 중 가장 오랫동안 사용되지 않은 페이지를 우선적으로 선택하여 교체하는 알고리즘이다.
  - `최근에 사용되지 않은 것은 앞으로도 사용될 확률이 낮다`는 시간 국소성(Temporal Locality) 원리에 기반한다.

- **용도**
  - 운영체제의 가상 메모리 관리
  - 웹 브라우저의 캐시 관리 (방문한 지 오래된 페이지 삭제)
  - 데이터베이스 버퍼 풀 관리 (자주 안 찾는 데이터 밀어내기)
  - CDN(Content Delivery Network)의 컨텐츠 캐싱

- **동작 원리**
  - LRU는 각 페이지가 마지막으로 사용된 시각을 추적한다. 
  - 새로운 공간이 필요할 때, 이 타임스탬프가 가장 과거인 페이지를 찾아 희생(Eviction)시킨다.
  
  1. **페이지 참조 시**
     - 해당 페이지가 이미 메모리에 있다면, 해당 페이지의 참조 시각을 현재로 갱신합니다. (리스트의 맨 앞으로 이동)
     
  2. **페이지 부재(Fault) 발생 시**
     - 메모리에 빈 공간이 있다면 새로운 페이지를 적재한다.
     - 메모리가 꽉 찼다면, 사용된 지 가장 오래된(리스트의 맨 뒤) 페이지를 제거하고 그 자리에 새 페이지를 넣는다.

- **트레이드 오프**
  - 장점
    - FIFO보다 페이지 폴트 횟수가 현저히 적음
    - 벨레이디의 모순이 발생하지 않음
  - 단점 
    - 모든 메모리 접근 시마다 시간을 기록하거나 위치를 옮겨야 함
    - 참조 시각을 기록하기 위한 별도의 메모리 공간 필요
    
- **주의사항**

  - **구현의 복잡성**
    - 순수하게 모든 접근 시각을 기록하는 LRU를 하드웨어(CPU/MMU) 수준에서 완벽히 구현하는 것은 매우 비싸다.
    - 그래서 실무에서는 이를 흉내 낸 Clock Algorithm(또는 Second Chance) 같은 근사 알고리즘을 사용하곤 한다.
    
  - **시간 복잡도**
    - 페이지를 찾고 위치를 옮기는 작업이 O(1)에 이루어지지 않으면 메모리 접근 자체가 느려집니다. 이를 위해 보통 해시 테이블(Hash Table)과 이중 연결 리스트(Doubly Linked List)를 조합하여 설계한다.
    
  - **워크로드 특성 파악**
    - 만약 프로그램이 매우 큰 데이터를 한 번씩만 훑고 지나가는 방식(Sequential Scan)이라면, LRU는 오히려 독이 될 수 있다. 
    - 금방 다시 쓸 데이터가 한 번만 쓰고 버려지는 데이터에 밀려 나갈 수 있기 때문이다.

## 메모리 지역성

- **정의**
  - 메모리 지역성은 프로세스가 실행되는 동안 메모리의 특정 영역을 집중적으로 참조하는 경향을 말한다.
  - CPU는 빛의 속도로 일하지만, 데이터를 가져오는 메모리(RAM)는 상대적으로 매우 느립니다. 이를 극복하기 위해 CPU 내부에 아주 빠르고 비싼 캐시 메모리(Cache Memory)를 두는데, 이 캐시가 효과적으로 작동하려면 "다음에 CPU가 어떤 데이터를 찾을지"를 미리 예측해야 한다. 그 예측의 근거가 바로 지역성이다.
  
- **지역성의 세 가지 유형**
  - **시간 지역성 (Temporal Locality)**
    - 예를 들어, 루프(Loop) 문에서 사용하는 제어 변수(i)나 합계를 구하는 변수 등은 짧은 시간 동안 반복해서 읽고 쓰게 된다.
    
  - **공간 지역성 (Spatial Locality)**
    - 특정 메모리 주소가 참조되면, 그 인근의 주소들도 곧 참조될 가능성이 높다는 원리이다.
    - 가장 대표적인 예가 배열(Array)이다. 배열은 메모리에 순차적으로 저장되므로, 0번 칸을 읽었다면 1번, 2번 칸도 연달아 읽을 확률이 매우 높다.

  - **순차 지역성 (Sequential Locality)**
    - 데이터가 기억장치에 저장된 순서대로 인출되어 실행되는 경향을 의미이다. 명령어들이 순차적으로 실행되는 프로그램 코드 자체가 좋은 예시다.
    
- **동작 원리와 캐시 메모리**
  - 캐시 메모리는 이 지역성을 활용하여 성능을 높인다.
  - CPU가 데이터를 요청하면 먼저 캐시를 확인한다. 
  - 이때 데이터가 있으면 캐시 히트(Cache Hit)라 하며 즉시 처리한다. 
  - 만약 없다면 캐시 미스(Cache Miss)가 발생하여 느린 RAM까지 다녀와야 한다.
  - 중요한 점은 캐시가 데이터를 가져올 때 딱 필요한 1바이트만 가져오는 것이 아니라, 블록(Block/Line) 단위로 뭉텅이로 가져온다는 것이다. 
  - 공간 지역성 덕분에 주변 데이터도 곧 쓰일 것이기 때문이다.
  
- **트레이드 오프**
  - 장점
    - 캐시 히트율이 높아지면 전체 시스템 속도가 비약적으로 향상됨
    - 고성능 캐시 메모리는 매우 비싸고 용량이 작음
    - 지역성을 고려한 코드 작성은 개발 난이도를 높임

- **주의사항**
  - **데이터 구조의 선택**
    - 연속된 메모리 공간을 사용하는 배열은 공간 지역성이 매우 좋아 캐시 효율이 높다. 
    - 반면, 데이터가 메모리 여기저기 흩어져 있는 연결 리스트(Linked List)는 지역성이 나빠 캐시 미스가 잦고 성능이 떨어질 수 있다.
    
  - **다차원 배열 접근 방향**
    - C언어나 자바 같은 행 우선(Row-major) 언어에서 2차원 배열을 다룰 때, 가로 방향이 아닌 세로 방향으로 먼저 읽게 코드를 짜면 공간 지역성을 완전히 파괴하게 된다. 이는 성능을 수십 배 떨어뜨리는 주범이 된다.
    
  - **캐시 오염 (Cache Pollution)**
    - 지역성이 전혀 없는 거대한 데이터를 한 번 읽어버리면, 캐시에 있던 소중하고 자주 쓰이는 데이터들이 모두 쫓겨나게 된다. 
    - 이를 캐시 오염이라고 하며, 대규모 파일 처리 시 주의해야 한다.
    
## 페이지 스왑 알고리즘(LFU | Least Frequently Used)

- **정의**
  - LFU는 메모리에 적재된 페이지 중 참조 횟수가 가장 적은 페이지를 교체 대상으로 선정하는 알고리즘이다. 
  - 과거에 많이 사용된 페이지는 앞으로도 사용될 가능성이 높다는 가정에 기반한다.
  
- **용도**
  - 데이터의 인기도가 명확하게 갈리는 시스템에서 유용하다. 
  - 예를 들어, 웹 서버에서 자주 요청되는 정적 파일이나 데이터베이스에서 반복적으로 조회되는 핫 데이터(Hot Data)를 관리할 때 효과를 발휘한다.
  
- **동작 원리**
  - LFU는 각 페이지마다 참조 횟수를 기록하는 카운터를 할당하여 관리한다.
  1. 특정 페이지가 참조될 때마다 해당 페이지의 카운터를 1씩 증가시킨다.
  2. 페이지 폴트가 발생하여 교체가 필요할 때, 카운터 값이 가장 작은 페이지를 찾아 메모리에서 내보낸다.
  3. 만약 카운터 값이 동일한 페이지가 여러 개라면, 보통 FIFO 방식을 병행하여 그중 가장 먼저 들어온 페이지를 내보낸다.
  
- **트레이드 오프**
  - 장점
    - 장기적인 관점에서 페이지의 인기도를 정확히 반영함
    - 자주 쓰이는 핵심 데이터의 보존율 극대화
  - 단점 
    - 각 페이지의 참조 횟수를 계속 유지하고 비교해야 함
    - CPU와 메모리 자원의 추가적인 오버헤드 발생
    
- **주의사항**
  - LFU를 현업에서 바로 사용하기 어려운 치명적인 단점이 하나 있다. 
  - 바로 과거에만 많이 쓰이고 지금은 안 쓰이는 페이지 문제이다.
  
  - **상황**
    - 프로그램 초기화 단계에서 특정 페이지가 1,000번 참조되었다고 가정해 보겠다.
    - 초기화가 끝난 후에는 전혀 필요 없는 데이터임에도 불구하고, 카운터가 1,000이나 되기 때문에 메모리에서 절대 쫓겨나지 않고 자리를 차지하게 된다.
    
  - **해결책**
    - 이를 보완하기 위해 시간이 지남에 따라 카운터 값을 점차 감소시키는 노화(Aging) 기법을 섞어서 사용하기도 한다.
    
## 페이지 스왑 알고리즘(NUR | Not Used Recently, 최근 미사용)


- **정의**
  - NUR(또는 NRU)은 최근에 사용되지 않은 페이지를 우선적으로 교체 대상으로 삼는 알고리즘이다. 
  - 모든 페이지의 사용 기록을 정밀하게 추적하는 대신, 두 개의 비트(Bit)만을 사용하여 페이지의 상태를 단순화하여 관리한다.
  
- **용도**
  - 메모리 관리 부하를 줄이면서도 LRU(Least Recently Used)와 유사한 효율을 내고 싶을 때 사용한다. 
  - 현대 운영체제에서 매우 널리 쓰이는 실용적인 방식이다.
  
- **동작 원리**

  - NUR은 각 페이지마다 하드웨어가 지원하는 두 종류의 비트를 참조한다.
  
  1. **참조 비트(Reference Bit, R)**
     - 페이지가 읽히거나 쓰였을 때 1로 설정된다.
    
  2. **변형 비트(Modified Bit, M)**
     - 페이지의 내용이 수정(Write)되었을 때 1로 설정된다. (Dirty Bit라고도 부른다.)
  
- **트레이드 오프**
  - **장점**
    - 적은 메모리와 CPU 자원으로도 꽤 훌륭한 교체 대상을 찾아낸다. 
    - 하드웨어가 비트를 자동으로 업데이트해주므로 소프트웨어 오버헤드가 매우 적다.
    
  - **단점**
    - "최근"의 기준이 비트가 초기화되는 주기에 의존하므로, LRU처럼 정밀하게 "가장 오래전에 사용된 것"을 집어내지는 못한다.
    
- **주의사항**

  - **초기화 주기 설정**
    - R 비트를 너무 자주 0으로 만들면 모든 페이지가 "참조 안 됨"으로 보여 효율이 떨어지고, 너무 가끔 하면 "최근"이라는 의미가 퇴색된다. 
    - 이 적절한 주기를 찾는 것이 운영체제 튜닝의 묘미이다.
    
  - **Write-back 비용 고려**
    - NUR이 M 비트를 보는 이유는 비즈니스 비용 때문이다. 
    - 수정된 페이지(M=1)를 버리려면 하드디스크에 다시 쓰는(Write-back) 과정이 필요해 시간이 오래 걸린다. 
    - 따라서 가급적 수정되지 않은 페이지(M=0)를 먼저 버려 시간을 아끼는 전략을 취한다.
    
## 스레싱(Thrashing)

- **정의**
  - 스레싱은 프로세스가 실제 연산을 수행하는 시간보다 페이지를 교체(Page Swapping)하는 데 더 많은 시간을 소비하는 상태를 말한다.
  - 쉽게 말해, 일은 안 하고 서류 뭉치(페이지)를 캐비닛(디스크)에서 책상(RAM)으로 옮기느라 하루를 다 보내는 사무원과 같다. 
  - 시스템의 처리량(Throughput)은 급감하고, 사용자는 화면이 멈춘 듯한 끔찍한 경험을 하게 된다.
  
- **발생 원인 및 동작 원리 (악순환의 고리)**
  - 스레싱은 운영체제의 `이상적인 의도`가 빗나가면서 발생하는 역설적인 현상이다.
  
  1. **메모리 부족**
     - 실행 중인 프로세스가 많아지면서 각 프로세스에 할당된 메모리 조각(Frame)이 너무 적어진다.
    
  2. **페이지 폴트 폭증**
     - 각 프로세스는 당장 필요한 코드를 실행하려 하지만 메모리에 없으니 계속 페이지 폴트를 일으킨다.
    
  3. **CPU 휴식**
     - 페이지를 디스크에서 가져오는 동안(I/O 작업), CPU는 할 일이 없어 대기 상태에 빠진다.
    
  4. **운영체제의 오판**
     - 운영체제는 "어? CPU가 노네? 일이 부족한가 보네?"라고 착각하여 새로운 프로세스를 더 실행(다중 프로그래밍 정도 증가)한다.
    
  5. **시스템 붕괴**
     - 가뜩이나 부족한 메모리를 새로운 프로세스가 더 뺏어가면서 페이지 폴트는 더욱 빈번해지고, 결국 시스템은 페이지 교체 작업만 하다가 멈춰버린다.
  
- **트레이드 오프**

  - **다중 프로그래밍**
    - 장점 : CPU 자원을 쉴 새 없이 사용하여 효율 극대화
    - 단점 : 임계점을 넘는 순간 스레싱 발생
    
  - **시스템 안정성**
    - 장점 : 예측 가능한 응답 속도 및 서비스 가용성 유지
    - 단점 : 동시 처리량이 낮아져 서버 자원 낭비 우려
  
- **해결 및 예방 기법 (실무적 접근)**  
  - **워킹 셋 (Working Set) 모델**
    - 프로세스가 일정 시간 동안 자주 참조하는 페이지들의 집합(Working Set)을 미리 파악한다. 
    - 운영체제는 이 `최소한의 세트`를 통째로 메모리에 올릴 수 있을 때만 프로세스를 실행시킨다.
    
  - **페이지 폴트 빈도(PFF, Page Fault Frequency) 조절**
    - 페이지 폴트가 너무 자주 발생하면 "이 프로세스는 메모리가 부족하구나"라고 판단하여 프레임을 더 준다. 
    - 반대로 너무 안 발생하면 "메모리가 남네?"라고 판단하여 뺏어간다.
    
  - **근본적인 하드웨어 증설**
    - 결국 땅(RAM)이 좁아서 생기는 문제이다. 
    - 소프트웨어 최적화로 한계가 있다면 물리적인 RAM을 추가하는 것이 가장 확실한 비즈니스 솔루션이다.
    
- **주의사항**
  - **성능 모니터링**
    - CPU 사용률이 갑자기 떨어지면서 디스크 I/O가 100%에 근접한다면 십중팔구 스레싱이다. 
    - 이때는 프로세스를 강제 종료하거나 서버를 분산시켜야 한다.
    
  - **프로세스 우선순위** 
    - 중요한 비즈니스 로직을 가진 프로세스가 스레싱 때문에 희생되지 않도록 우선순위 기반의 메모리 할당 전략을 세워야 한다.
