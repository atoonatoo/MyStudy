
## 정의

- 레디스(Redis)는 `Remote Dictionary Server`의 약자로, 메모리 내에 데이터를 저장하고 관리하는 오픈 소스 기반의 비정상용(In-memory) 키-값(Key-Value) 구조 데이터 관리 시스템이다.

## Sentinel

- **정의**
  - 레디스 센티넬은 레디스의 고가용성(High Availability)을 실현하기 위한 분산 관리 시스템이다. 
  - 레디스 인스턴스와는 별개의 프로세스로 실행되며, 마스터와 복제본(Replica) 노드들을 상시 감시하고 장애 발생 시 자동으로 복구 절차를 수행하는 제어 계층의 역할을 담당한다.
  
- **사용하는 이유**
  - 서비스 운영 중 발생할 수 있는 단일 장애 지점(Single Point of Failure)을 제거하고 서비스 연속성을 확보하는 것이 주된 목적이다. 
  - 마스터 노드가 예기치 않게 종료되었을 때 관리자가 수동으로 복제본을 마스터로 승격시키고 설정을 변경하는 번거로움을 없애며, 장애 복구 시간을 단축하여 가동 중지 시간을 최소화하기 위해 사용한다.

- **핵심 역할 및 특징**
  - **모니터링**
    - 마스터 및 복제본 노드들이 정상적으로 작동하는지 지속적으로 확인한다.
  - **알림** 
    - 감시 중인 노드에 문제가 발생할 경우 API를 통해 관리자나 시스템에 통지한다.
  - **자동 장애 조치**
    - 마스터 노드 장애 시 복제본 노드 중 하나를 새로운 마스터로 승격시키고, 나머지 복제본들이 새 마스터를 복제하도록 구성을 변경한다.
  - **환경 설정 제공자**
    - 클라이언트가 현재 유효한 마스터 노드의 주소를 알 수 있도록 서비스 디스커버리 역할을 수행한다.
    
- **동작 원리**
  1. **주관적 다운(S-DOWN)**
    - 특정 센티넬 노드가 설정된 시간 동안 마스터로부터 응답을 받지 못할 경우, 해당 센티넬은 마스터를 주관적으로 다운되었다고 판단한다.
  2. **객관적 다운(O-DOWN)**
    - 주관적 다운을 감지한 센티넬이 다른 센티넬들에게 상태 확인을 요청한다. 설정된 쿼럼(Quorum) 이상의 센티넬이 마스터의 장애에 동의하면 객관적으로 다운된 것으로 간주한다.
  3. **리더 선출 및 장애 조치**
    - 장애가 확정되면 센티넬들 사이에서 장애 조치를 주도할 리더 센티넬을 선출한다. 선출된 리더는 복제본 노드들의 우선순위, 복제 오프셋(데이터 최신도), 런타임 ID 등을 평가하여 최적의 노드를 새로운 마스터로 선정한다.
  4. **구성 전파**
    - 새로운 마스터가 승격되면 나머지 복제본 노드들의 설정을 변경하여 새 마스터를 바라보게 하며, 기존의 마스터가 복구될 경우 이를 복제본 노드로 강제 편입시킨다.
  
- **트레이드 오프**
  - **데이터 일관성 vs 가용성**
    - 레디스는 비동기 복제 방식을 사용하므로, 장애 조치 시점에 마스터에서 복제본으로 전달되지 못한 데이터는 영구적으로 유실될 가능성이 존재한다.
  - **인프라 복잡도** 
    - 센티넬 자체의 고가용성을 위해 최소 3대 이상의 추가 프로세스 및 서버 자원이 필요하며, 관리 포인트가 늘어난다.
  - **확장성 제한** 
    - 센티넬은 고가용성 솔루션일 뿐 샤딩(데이터 분산 저장)을 지원하지 않는다. 
    - 따라서 데이터 용량이 단일 서버의 메모리 한계를 초과하는 경우 레디스 클러스터로 전환해야 한다.
    
- **주의 사항**
  - **최소 3대 구성**
    - 쿼럼 합의와 센티넬 자체의 장애 대비를 위해 반드시 홀수 대수(최소 3대)로 구성해야 한다.
    - 2대로 구성할 경우 한 대만 장애가 발생해도 과반수 동의를 얻지 못해 장애 조치가 불가능해진다.
  - **클라이언트 지원 여부**
    - 애플리케이션의 레디스 클라이언트 라이브러리가 센티넬 프로토콜을 공식 지원하는지 확인해야 한다. 센티넬을 통해 마스터 주소를 동적으로 획득하는 로직이 필수적이다.
    
  - **네트워크 파티션(Split Brain)**
    - 네트워크 단절로 인해 마스터가 고립되었을 때 여러 마스터가 존재하는 상황이 발생할 수 있다. 
    - 이를 방지하기 위해 min-replicas-to-write 및 min-replicas-max-lag 옵션을 적절히 설정하여 데이터 정합성을 보호해야 한다.
  
  - **물리적 분리**
    - 가용성을 극대화하기 위해 센티넬 프로세스와 레디스 노드를 서로 다른 물리 서버나 랙에 분산 배치하는 것이 권장된다.

## Redis Cluster

- **정의**
  - 레디스 클러스터는 데이터를 여러 레디스 노드에 자동으로 분산하여 저장(Sharding)하고, 동시에 고가용성을 제공하는 분산 데이터베이스 시스템이다.
  - 별도의 관리 프로세스(Sentinel) 없이 노드들이 서로 직접 통신하며 상태를 감시하고 장애를 복구하는 분산 아키텍처를 가진다.

- **사용하는 이유**
  - **수평적 확장(Scale-out)**: 단일 서버의 메모리 용량을 초과하는 대규모 데이터를 처리하기 위해 여러 서버의 자원을 통합하여 사용한다.
  - **성능 분산**: 쓰기 및 읽기 부하를 여러 마스터 노드에 분산시켜 전체 시스템의 처리량(Throughput)을 극대화한다.
  - **고가용성 유지**: 클러스터 내 일부 마스터 노드에 장애가 발생하더라도, 복제본 노드가 즉시 승격되어 서비스 중단 없이 운영을 지속하기 위함이다.

- **핵심 역할 및 특징**
  - **데이터 샤딩(Sharding)**
    - 데이터를 여러 마스터 노드에 분할 저장하여 데이터 용량 및 처리 성능을 확장한다.
  - **자율적인 장애 조치(Failover)**
    - 센티넬 없이 노드 간 거십 프로토콜(Gossip Protocol)을 통해 서로를 감시하며, 장애 발생 시 투표를 통해 복구 절차를 수행한다.
  - **다중 마스터 구조**
    - 여러 개의 마스터 노드가 동시에 존재하며, 각 마스터는 자신만의 데이터 영역(Hash Slot)을 소유하고 독립적으로 쓰기 요청을 처리한다.

- **동작 원리**
  1. **해시 슬롯(Hash Slot)**
    - 클러스터는 전체 데이터를 16,384개의 해시 슬롯으로 나눈다. 각 키는 `CRC16(key) mod 16384` 연산을 통해 특정 슬롯에 할당되며, 마스터 노드들은 이 슬롯들을 나누어 관리한다.
  2. **거십 프로토콜(Gossip Protocol)**
    - 노드들은 '클러스터 버스'라는 별도의 통신 채널을 통해 서로의 상태 정보를 주고받으며 클러스터의 전체 맵(Map)을 동기화한다.
  3. **클라이언트 리다이렉션(Redirection)**
    - 클라이언트가 잘못된 노드에 데이터를 요청할 경우, 해당 노드는 올바른 슬롯을 가진 노드의 정보를 담은 'MOVED' 에러를 응답하여 클라이언트가 직접 재접속하도록 유도한다.
  4. **슬롯 마이그레이션**
    - 노드 추가 및 제거 시, 가동 중지 없이 관리 중인 해시 슬롯을 다른 노드로 이동시켜 데이터 재분배(Resharding)를 수행한다.

- **트레이드 오프**
  - **다중 키 연산 제한**
    - 여러 키를 동시에 처리하는 연산(MSET, SUNION 등)은 모든 키가 동일한 해시 슬롯에 속해 있을 때만 가능하다. 서로 다른 노드에 걸친 연산은 원칙적으로 불가능하다.
  - **네트워크 오버헤드**
    - 노드 수가 증가할수록 거십 프로토콜로 인한 노드 간 통신 부하가 증가하며, 이는 전체적인 네트워크 성능에 영향을 줄 수 있다.
  - **설정 및 운영 복잡도**
    - 단일 인스턴스나 센티넬 구조에 비해 구성이 복잡하며, 리다이렉션을 처리할 수 있는 전용 클라이언트 라이브러리가 반드시 필요하다.

- **주의 사항**
  - **최소 구성 요건**
    - 자율적인 장애 조치를 위한 과반수 투표 시스템을 위해 최소 3대의 마스터 노드가 필요하며, 안정성을 위해 각각의 복제본을 포함한 총 6대의 노드 구성이 권장된다.
  - **해시 태그(Hash Tags) 활용**
    - 다중 키 연산이 필요한 경우, 키 이름에 `{}`를 사용하여 특정 부분만 해싱함으로써 관련 데이터들이 동일한 슬롯에 저장되도록 강제해야 한다.
  - **클라이언트 라이브러리 선정**
    - 사용하는 언어의 라이브러리가 클러스터 모드(MOVED/ASK 응답 처리 등)를 완벽히 지원하는지 검토해야 한다.
  - **메모리 파편화 및 관리**
    - 각 노드의 메모리 사용량을 균등하게 유지하기 위해 데이터 편중(Skew) 현상이 발생하는지 지속적으로 모니터링해야 한다.

## Redis Clustering

- **정의**
  - 레디스 클러스터링은 여러 레디스 노드에 데이터를 자동으로 분산 저장하여 수평적 확장성(Scalability)과 고가용성을 동시에 제공하는 분산 아키텍처이다.
  - 별도의 관리 프로세스(Sentinel) 없이 노드들이 서로 통신하며 상태를 감시하고, 데이터를 샤딩(Sharding)하여 관리하는 방식이다.

- **사용하는 이유**
  - 단일 레디스 인스턴스의 메모리 용량 한계를 극복하기 위해 대규모 데이터를 여러 서버에 나누어 저장(데이터 샤딩)하기 위함이다.
  - 초고속 데이터 처리가 필요한 환경에서 쓰기와 읽기 부하를 여러 마스터 노드로 분산시켜 전체적인 처리량(Throughput)을 극대화하기 위해 사용한다.
  - 특정 노드에 장애가 발생하더라도 서비스 중단 없이 데이터 접근을 유지하는 고가용성을 확보하기 위함이다.

- **핵심 역할 및 특징**
  - **자동 데이터 샤딩**
    - 데이터를 여러 마스터 노드에 자동으로 분할 저장하며, 해시 슬롯(Hash Slot) 개념을 사용하여 구현한다.
  - **분산형 구조 (Decentralized)**
    - 중앙 제어 노드 없이 모든 노드가 서로 가십 프로토콜(Gossip Protocol)을 통해 상태 정보를 교환하고 장애를 감지한다.
  - **자체 장애 조치 (Self-Failover)**
    - 별도의 센티넬 없이 클러스터 내부의 마스터 노드들이 과반수 동의를 통해 장애 노드를 판단하고 복제본을 마스터로 승격시킨다.
  - **고성능 유지**
    - 클라이언트가 올바른 노드로 직접 접근하도록 유도하여 프록시 계층으로 인한 병목 현상을 방지한다.

- **동작 원리**
  1. **해시 슬롯(Hash Slot) 할당**
    - 클러스터는 총 16,384개의 해시 슬롯을 가지며, 이를 마스터 노드들에게 나누어 할당한다.
  2. **키-슬롯 매핑**
    - 모든 키는 `CRC16(key) mod 16384` 공식을 통해 특정 슬롯에 매핑되며, 해당 슬롯을 보유한 마스터 노드에 저장된다.
  3. **노드 간 통신 (Gossip Protocol)**
    - 각 노드는 전용 포트(기본 포트 + 10000)를 통해 서로의 상태와 슬롯 정보를 주기적으로 공유한다.
  4. **리다이렉션 (Redirection)**
    - 클라이언트가 잘못된 노드에 요청을 보낼 경우, 해당 노드는 올바른 슬롯을 가진 노드의 정보를 포함하여 'MOVED' 에러를 응답하며 클라이언트의 재접속을 유도한다.
  5. **장애 감지 및 승격**
    - 마스터 노드가 응답하지 않으면 나머지 마스터들이 투표를 실시한다. 과반수의 찬성을 얻으면 해당 마스터의 복제본 노드 중 하나를 새로운 마스터로 승격시킨다.

- **트레이드 오프**
  - **성능 vs 다중 키 작업 제약**
    - 개별 키 접근 속도는 매우 빠르나, 서로 다른 슬롯에 속한 키들을 한 번에 처리하는 다중 키 작업(MSET, SUNION 등)은 기본적으로 지원되지 않거나 성능 저하를 야기한다.
  - **데이터 가용성 vs 복제 지연**
    - 비동기 복제 방식을 취하므로 마스터 승격 시점에 미처 복제되지 못한 일부 데이터가 유실될 수 있다.
  - **구조적 유연성 vs 운영 복잡도**
    - 노드 추가 및 제거가 자유롭지만, 대규모 클러스터 운영 시 네트워크 설정 및 노드 관리의 복잡도가 크게 증가한다.

- **주의 사항**
  - **최소 노드 구성**
    - 고가용성을 보장하기 위해 최소 3대의 마스터 노드가 필요하며, 각 마스터당 최소 1대의 복제본을 두어 총 6대의 노드로 구성하는 것이 표준이다.
  - **클라이언트 라이브러리 지원**
    - 일반 레디스 클라이언트가 아닌 클러스터 모드를 지원하는 라이브러리를 사용해야 하며, 'MOVED' 및 'ASK' 리다이렉션 메시지를 처리할 수 있어야 한다.
  - **멀티 키 작업 시 해시 태그(Hash Tags) 사용**
    - 서로 다른 키를 동일한 슬롯에 강제로 할당해야 할 경우, 키 이름에 `{}`를 사용하여 동일한 해시 값을 갖도록 설계해야 한다.
  - **메모리 파편화 및 오버헤드**
    - 클러스터 버스(Cluster Bus) 통신으로 인한 네트워크 트래픽 증가와 각 노드가 클러스터 맵을 유지해야 하는 메모리 비용을 고려해야 한다.
