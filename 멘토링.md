


- **비관적 락 (Pessimistic Lock - SELECT FOR UPDATE)**
    
    - 실무 미사용 이유: 데이터베이스(DB)는 확장이 어려운데(Scaling issue), 락을 잡으면 DB에 과부하가 직접적으로 전달됨.
        
    - 성능 저하: 락의 범위가 크고 시간이 길어질수록 DB 부하가 기하급수적으로 증가하여 실무 수준의 동시성 처리에 부적합함.
        
- **낙관적 락 (Optimistic Lock - JPA Version)**
    
    - 실무 미사용 이유: 데이터 버저닝 관리 비용이 비싸며, 충돌 시 재시도 로직(Retry) 구현이 까다로움.
        
    - CPU 부하: 충돌 대응을 위해 `while(true)` 루프와 `Thread.sleep()`을 사용하는 방식은 트래픽이 몰릴 경우 CPU 부하를 폭발시켜 시스템 전체를 위험하게 함.
        
- **분산 락 (Distributed Lock - Redis)**
    
    - 실무 활용도: 실제 서비스의 99% 이상이 Redis를 활용한 분산 락 시스템을 사용함.
        
    - 장점 1 (속도): epoll 방식을 사용하여 처리 속도가 매우 빠름.
        
    - 장점 2 (저부하): 인메모리 기반으로 DB에 직접적인 부하를 주지 않음.
        
    - 장점 3 (단순성): 기본적으로 싱글 스레드(Single Thread)로 요청을 처리하므로 별도의 복잡한 동시성 제어 고민을 덜어줌.
        
- **Redis의 단점과 극복 방안**
    
    - 단점 (취약성): C언어로 작성되어 가비지 컬렉션(GC)이 없으며, 부하 시 OOM(Out Of Memory)으로 인해 프로세스가 죽을 수 있음.
        
    - 휘발성: 인메모리 특성상 데이터가 날아갈 위험이 존재함.
        
    - 극복 방안: Redis 클러스터링(Sentinel, Clustered)을 통해 여러 대를 묶어 가용성을 확보하고, 소프트웨어 기술(가상 메모리 등)로 하드웨어 한계를 극복함.
        
- **JPA 영속성 컨텍스트(Persistence Context) 및 실무 팁**
    
    - Dirty Checking: 엔티티가 Managed 상태라면 `repository.save()`를 명시하지 않아도 변경 사항이 감지되어 업데이트 쿼리가 날아감.
        
    - 엔티티 상태: Managed(관리), Detached(분리), Removed(삭제) 상태에 대한 명확한 이해 필요.
        
    - 용어 정리: 모호한 '도메인 객체'라는 용어보다는 기술적으로 명확한 'Entity'라는 용어 사용을 권장함.
        
- **향후 학습 과제**
    
    - Redis 심화: 클러스터링 구조와 가상 메모리(Magic)의 동작 원리 파악.
        
    - 동시성 제어 실무 적용: Redis 분산 락의 구체적인 구현 패턴 학습.
