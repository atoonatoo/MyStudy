## 2. 오늘의 CS 지식 세션 (Day 1)

클라이언트님, 오늘 저희가 함께 살펴볼 첫 번째 핵심 키워드는 **로드 밸런싱(Load Balancing)**입니다. 대규모 트래픽을 처리하는 서비스의 안정성을 결정짓는 가장 기초적이면서도 강력한 아키텍처 요소입니다.

### 오늘의 키워드: 로드 밸런싱 (Load Balancing)

### 비즈니스 임팩트

클라이언트님의 서비스에 사용자가 갑자기 몰리는 상황을 가정해 보겠습니다. 서버 한 대가 감당할 수 있는 한계를 넘어서면 서비스는 중단되고, 이는 곧 매출 손실과 고객 신뢰도 하락으로 이어집니다. 로드 밸런싱은 여러 대의 서버에 트래픽을 고르게 분산하여 시스템의 가용성을 극대화하고, 서비스 중단 없는 쾌적한 환경을 고객에게 제공하기 위해 반드시 필요합니다.

### 기술적 핵심 메커니즘

로드 밸런서는 클라이언트와 서버 그룹 사이에서 '교통 정리' 역할을 수행합니다. 주요 동작 방식은 다음과 같습니다.

- L4 로드 밸런싱: 전송 계층(Transport Layer)에서 IP 주소와 포트 번호를 기준으로 트래픽을 나눕니다. 데이터의 내용을 확인하지 않으므로 속도가 매우 빠르고 효율적입니다.
    
- L7 로드 밸런싱: 응용 계층(Application Layer)에서 HTTP 헤더, 쿠키, URL 등 메시지의 내용을 직접 확인하고 분산합니다. 특정 URL 패턴에 따라 서로 다른 서버로 보낼 수 있는 정교한 제어가 가능합니다.
    
- 주요 알고리즘: 단순히 순서대로 배분하는 라운드 로빈(Round Robin), 서버의 현재 연결 수가 적은 곳으로 보내는 최소 연결(Least Connection), 서버의 성능에 따라 가중치를 두는 가중치 기반 배분 등이 있습니다.
    

### 실무 사례 및 주의사항

저희가 예전에 운영하던 글로벌 서비스에서도 특정 이벤트 기간에 트래픽이 평소의 100배 이상 몰린 적이 있습니다. 이때 L7 로드 밸런서를 통해 정적 이미지 요청과 동적 API 요청을 각각의 전용 서버군으로 분리하여 처리함으로써 단 한 번의 장애 없이 이벤트를 마칠 수 있었습니다.

다만 주의할 점은 '세션 관리'입니다. 클라이언트가 첫 번째 요청을 A 서버에 보냈는데, 로드 밸런서가 두 번째 요청을 B 서버로 보낸다면 로그인 정보가 유지되지 않을 수 있습니다. 이를 해결하기 위해 Sticky Session 기능을 사용하거나, Redis 같은 외부 저장소에 세션을 공유하는 설계를 병행해야 합니다.

#### Optimistic Lock
- 낙관적 락(Optimistic Lock)
    - 데이터 충돌이 자주 발생하지 않을 것이라고 낙관적으로 가정하고 접근하는 방식.
    - 데이터베이스 자체에 물리적 락을 거는 대신, 버전관리를 통해 에플리케이션 레벨에서 데이터 정합성을 유지.
      
- 트랜잭션 내 self-invocation(자가 호출) 문제
    - 정의
        - AOP(Aspect-Oriented Programming)의 동작 원리인 프록시 방식 떄문에 발생하는 문제.
        - 같은 클래스 내에서 `@Transactional`이 붙은 메서드를 직접 호출하면 트랜잭션이 적용되지 않는 현상.
    - 원인
        - 스프링은 `@Transactional`이 붙은 Bean을 생성할 때, 원본 객체 대신 이를 감싸는 프록시 객체를 생성하여 컨테이너에 등록한다.
          
        - 프록시 객체는 무엇인가?
            - 원본 객체를 감싸고 있는 가짜 객체.
            - `OptimisticStockService`가 호출하면, 실제로는 스프링이 만든 프록시 객체가 호출.
              
        - 프록시 객체를 생성하여 컨테이너에 등록하는 이유?
            - 제어권 가로채기
                - `OptimisticLockFacade`가 `OptimisticStockService`를 주입받을 때, 스프링 원본이 아닌 프록시 객체를 전달.
                  
            - 자동화된 인프라 로직
                - 이렇게하면 사용자가 직접 `transaction.begin()`이나 `transaction.commit()`을 작성하지 않아도, 프록시가 컨테이너에서 적절한 타이밍에 이 로직들을 끼워 넣을 수 있다.
                  
        - 왜 이런 방식을 사용하는가?
            - `관심사의 분리(Separation of Concerns)`
                - 코드의 순수성 유지
                    - 서비스 코드에는 오직 재고를 줄인다는 핵심 로직만 남는다.
                - 재사용성과 유지보수
                    - 트랜잭션 관리 방식이 바뀌더라도 수 백개의 서비스 코드를 고칠 필요없이, 프록시 설정(스프링 설정)만 바꾸면 된다.
                - 선언적 프로그래밍
                    - 개발자는 `이 메서드는 트랜잭션이 필요해`라고 선언(`@Transactional`)만 하면 된다. 나머지는 프록시가 알이서 처리해주는 마법같은 편리함을 누리는 것.
          
- Facade 패턴
    - 여러 개의 클래스나 복잡한 서브시스템 사이 상호작용을 대신 처리해주는 통합 인터페이스를 제공.
    - 복잡성 은닉
        - 호출하는 쪽(클라이언트)은 내부에 재시도 로직이 어떻게 돌아가는지 알 필요없다.
    - 결합도 감소
        - 시스템 구조가 바뀌어도 퍼사드만 유지된다면 이를 사용하는 외부 코드는 수정할 필요 없다.
    - 사용하는 이유
        - 관심사 분리
        - 트랜잭션 경계 설정.
            - Self-invocation 문제를 해결하는데 매우 좋은 방법.
            - 퍼사드가 외부에서 서비스를 호출함으로써 프론시 객체가 정상적으로 트랜잭션을 시작.
  
- `@Version`을 활용한 낙관적 락
    - 데이터를 읽을 때, 현재의 버전 정보를 함께 읽는다.
    - 수정 시점에 내가 처음에 읽었던 버전이 db에 그대로 있는 확인.
    - 만약 그 사이 다른 사용자가 데이터를 수정해서 버전이 올라갔다면 `ObjectOptimisticLockingFailureException`이 발생해 수정에 실패.

### Concurrency

- **동시성이란?**
    - 여러 개의 작업이 마치 동시에 실행되는 것처럼 보이게하는 논리적인 개념이다.
      
    - **동시성 제어를 해야하는 이유?**
        - 여러개의 프로세스나 스레드가 공유 자원에 접근했을 때, 데이터의 일관성과 무결성을 보장하기 위해서 이다.
          
- **Thread-safe 해야하는 이유?**
    - 멀티 스레드 환경에서 공유 자원에 대한 경쟁 상태를 방지하여 데이터의 일관성을 유지하기 위함이다.
      
    - **Thread-safe를 제공해주는 기능이 java에는 무엇이 있는가?**
          
        - **언어 기본 키워드**
            - **synchronized(상호 배제)**
                - 정의
                    - synchronized는 한마디로 자물쇠이다.
                    - 여러 개의 스레드가 하나의 공유 자원에 접근 할 때, 한 번에 단 하나의 스레드만 실행할 수 있도록 보장해주는 상호 배제 메커니즘이다.
                - 목적
                    - 데이터의 원자성과 가시성을 보장하여 멀티 스레드 환경에서도 데아터가 오염되지 않는 스레드 세이프한 상태를 만들기 위함이다. 
                - 동작원리
                    - java의 모든 객체는 monitor라는 숨겨진 자물쇠를 가지고 있다.
                    - 획득(Lock Acquisition) : 스레드가 `synchronized` 영역에 접근하려고 하면, 해당 객체의 모니터 락을 요청한다.
                    - 점유 : 락이 비어있다면 스레드는 락을 얻고 코드를 실행한다.
                    - 대기(entry set) : 이미 다른 스레드가 락을 쥐고 있다면, 나중에 온 스레드는 entry set에서 자기 차례가 올 때까지 blocked 상태로 기다린다.
                    - 반납 : 코드가 끝나거나 예외가 발생하면 락을 자동으로 반납하고, 대기 중인 스레드 중 하나가 다시 락을 획득한다.
                - 사용법
                    - 메서드 수준
                        - `public synchronized void add()`
                        - 메서드 전체를 잠근다.
                        - 구현이 간단하지만, 메서드 내용이 길면 성능 저하가 크다.
                    - 블록 수준
                        - `synchronized(this) { ... }`
                        - 필요한 특정 코드 영역만 잠근다.
                        - 락 객체를 지정할 수 있어 훨씬 세밀한 제어 가능.
                    - 인스턴스 락
                        - `this`나 인스턴스 메서드에 사용.
                        - 해당 객체(인스턴스)당 하나의 스레드만 접근 가능.
                    - 클래스 락
                        - `static synchronized`, `ClassName.class`에 사용.
                        - 해당 클래스의 모든 인스턴스를 통틀어 단 하나의 스레드만 접근 가능.
                - 트레이드 오프
                    - 성능 저하
                        - 락을 얻고 해제하는 과정 자체에 CPU 비용이 들며, 스레드가 차단될 때, 컨텍스트 스위칭이 발생하여 시스템 자원을 소모한다.
                    - 병렬성 감소
                        - 아무리 코어가 많아도 sychronized 구간은 결국 한 줄로 서서 지나가야하는 병목 구간이 된다.
                - 주의 사항
                    - 데드락(교착 상태)
                        - 스레드 A는 락1을 쥐고 2를 기다리는데 스레드 B는 락2를 쥐고 1을 기다리는 상황이다.
                        - 서로 영원히 기다리게되는 문제 발생.
                    - 락의 범위 최소화
                        - 꼭 필요한 공유 자원 접근 부분만 블록으로 감싸야한다.
                        - 범위를 넓게 잡을 수록 애플리케이션은 느려진다.
                    - 가급적 `java.util.cunccurent` 패키지 고려
                        - 현대 Java에서는 `sychronized` 보다 더 유연하고 성능이 좋은 `ReentrantLock`이나 원자적 변수 `AtomicInteger` 등을 사용하는 경우가 많다.
                          
            - **volatile(가시성 보장)**
                - 변수의 값을 CPU 캐시가 아닌 메인 메모리에서 직접 읽고 쓰도록 강제한다.
                - 한 스에드가 변경한 값을 다른 스레드가 즉시 볼 수 있게 하지만, count++ 같은 복합 연산의 원자성은 보장하지 못한다.
            
        - **원자적 변수**
            - java.util.concurrent.atomic 패키지에 포함되어 있으며 성능이 매우 뛰어나다.
            - CAS(Compare-And-Swap) 알고리즘을 사용하여 락을 걸지 않고도 (Lock-free) 원자적 연산을 수행.
              
            - **CAS(Compare-And-Swap)** 
                - CAS는 멀티스레드 환경에서 데이터의 일관성을 유지하기 위해 사용하는 Non-blocking 동기화 방식이다.
                - 전통적인 방식이 lock을 걸어 다른 스레드의 접근을 완전히 차단한다면, CAS는 내가 알고 있는 값이 실제 값과 같다면 교체한다는 논리로 동작하는 낙관적 방식의 알고리즘이다.
                - CAS 연산은 크게 3개의 인자를 필요로 한다.
                - V (Memory Location)
                    - 업데이트하려는 메모리 주소(실제 현재 값)
                - A (Expected Old Value)
                    - 스레드가 예상하는 기존 값
                - B (New Value)
                    - 변경하려는 새로운 값
                - 동작 로직
                    1. 메모리의 현재값(V)이 내가 예상한 값(A)과 같은 지 비교
                    2. 값이 같다면 (V == A) 메모리 값을 새로운 값(B)으로 교체한다. (성공)
                    3.  값이 다르다면 (V !== A) 다른 스레드가 그 사이 값을 변경한 것으로 간주하여 아무 작업도 하지 않는다.
                - CAS를 사용하는 이유
                    - 컨텍스트 스위칭 비용이 없어 속도가 빠르다.
                    - 다만, 무한 루프로 인한 CPU 많은 소모가능성이 존재.
                - CAS의 한계 : ABA문제
                    1. 스레드 1이 값 A를 읽음.
                    2. 스레드 2가 값을 A -> B -> A로 변경.
                    3. 스레드 1이 다시 확인했을 때, 값은 여전히 A이므로, 변경이 없었다고 착각하고 연산을 수행.
                    - 이 과정에서 중간에 데이터가 변했음에도 불구하고 감지하지 못하는 문제가 발생할 수 있음.
                    - 이를 해결하기 위해 Version이나 Stampe를 함께 체크하는 방식을 사용.

### 가상메모리(Magic)
- 학습 방식
    - 아래의 URL 블로그에 작성한 게시물의 내용을 전부 공부한다.
    - 질문이나 자세한 이해는 제미나이를 통해 공부한다.
    - 정리한 내용을 옵시디언 노트에 작성한다.
    - 마무리가 되면 주기적으로 반복하여 읽는다.
    - 제미나이나 GPT를 통해서, 음성으로 잘 이해했는지 질문과답변을 오가며 검증한다.
    - [블로그 1](https://parkground.tistory.com/790)
    
- 마감 기한
    - **월요일까지**
    
- 정의
    - 가상 메모리는 프로세스 전체가 메모리에 올라와 있지 않아도 실행이 가능하도록 하는 메모리 관리 기법이다.
    - 물리적 메모리(RAM)의 한계를 극복하고, 각 프로세스에게 `너는 이 컴퓨터의 메모리를 통째로 쓰고 있다`는 환상을 심어준다.
- 용도
    - 실제 RAM보다 큰 프로그램 실행
    - 멀티태스킹 최적화
    - 프로세스 간 메모리 격리
- 동작 원리
    - 프로그램이 사용하는 주소(논리 주소)와 실제 RAM의 주소(물리 주소)를 분리하여 관리한다.

- MMU(Memory Management Unit)
    - CPU가 바라보는 주소는 가상의 번호표일뿐 실제 RAM의 위치로 바꿔주는 통역사가 필요하다.
    - MMU는 CPU 옆에 붙어서 논리 주소를 물리 주소로 즉시 변화해주는 하드웨어 장치.
      
- 페이징(Paging)
    - 가상 메모리를 관리하기 위해 메모리를 일정한 크기로 나눈 단위를 페이지(Page)라고 한다.
    - 이에 대응 하는 물리 메모리의 단위를 프레임(Frame)이라고 한다. 보통 `4KB` 크기를 사용.
      
- 페이지 테이블(Page Table)
    - 어떤 페이지가 어떤 프레임에 있는지 적혀있는 지도.
    - 프로세스 마다 고유의 페이지 테이블을 가진다.
    
- 요구 페이징(Demand Paging)
    - 현대 운영체제는 프로그램을 실행할 때 전체를 RAM에 올리지 않는다.
    - 당장 필요한 부분만 올리는데 여기서 중요한 사건이 발생한다.
    
    - 페이지 폴트(Page Fault)
        - CPU가 요청한 페이지가 현재 RAM에 없을 때 발생하는 일종의 중단(Interrupt) 현상.
        1. CPU가 페이지 테이블을 확인했는데 유효 비트가 0(없음)을 발견.
        2. 운영체제가 제어권을 넘겨 받아 하드디스크(스왑 영역)에서 해당 페이지를 찾는다.
        3. 빈 프레임을 찾아 데이터를 로드하고 페이지 테이블을 업데이트한다.
        4. 중단되었던 명령어를 다시 실행한다.
    
- 트레이드 오프
    - 메모리 활용
        - 이점
            - 실제 RAM보다 훨씬 큰 프로그램을 실행 가능
        - 비용
             - 주소 번역 과정에서 발생하는 연산 오버헤드
    - 멀티태스킹
        - 이점
            - 여러 앱을 동시에 띄워도 메모리 충돌 없음
        - 비용
            - 페이지 테이블을 저장하기 위한 추가 메모리 소모
    - 보안성
        - 이점
            - 프로세스 간 영역을 침범할 수 없음 (격리)
        - 비용
            - 빈번한 페이지 폴트 발생 시 급격한 성능 저하
    
- 스레싱(Thrashing)
    - 프로세스가 실행되는 시간보다 페이지를 교체(Swap In/Out)하는데 더 많은 시간을 소비하는 현상.
    - 메모리가 너무 부족하면 CPU가 일은 안 하고 하드디스크의 데이터만 주고받가가 시스템이 멈춘 것처럼 보인다.
      
- 페이지 교체 알고리즘(Page Replacement)
    - 빈 프레임이 없을 때 어떤 페이지를 쫒아낼지 결정하는 전략.
    - 가장 유명한 LRU(Least Recently Used)는 가장 오랜동안 사용하지 않은 것을 버리는 방식인데, 이 알고리즘의 효율에 따라 서비스 응답 속도가 천차만별이 된다.

- 가상 메모리 기술 면접 질문 리스트
    - 가상 메모리와 페이징 시스템에 대해 구체적으로 설명.
    - 요구 페이징과 페이지 폴트에 대해 구체적으로 설명.
    - MMU와 TLB에 대해 구체적으로 설명.
