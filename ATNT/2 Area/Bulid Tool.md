##### 제목 : 
    
- Maven / Gradle
    
    ```java
    - Maven : 
    	Ant의 대안 빌드 관리 도구
    	프로젝트 진해시 사용하는 수많은 라이브러리들을 관리하는 도구
    	네트워크를 통해 라이브러리들과 연관된 라이브러리까지 연동되어 관리가 된다.
    	작업 의존성 그래프 기반, 다중 모듈 빌드를 병렬 실행가능
    
    	* 엔트의 약점이었던 빌드 스크립트 문제를 해결하기 위해 
    		빌드 생명주기, 프로젝트 객체 모델이라는 새로운 개념 도입
    	
    	* Maven의 특징 : 
    		1) 프로젝트를 모델링 : 
    			 다양한 목적을 가지고 있는 플러그인은 POM으로 정의된다. 
    			 프로젝트에 의존되는 라이브러리들은 pom.xml에서 의존 관계를 정의한다.
    		2) POM.xml :
    			 Maven의 근간이 되는 개념으로 현재 프로젝트의 구성과 프로젝트에서 사용하고 
    			 있는 외부 플러그인 그리고 각 플러그인들 간의 의존 관계를 
    			 pom.xml에 정의 할 수 있다.
    
    	* POM(Project object model) | 프로젝트 객체 모델 : 
    		Maven을 사용하기 위해선 pom을 사용해야한다.
    		Project object model을 담고 있는 파일.
    		pom.xml에서 주요하게 다루는 기능 
    		1) 프로젝트 정보 : 프로젝트 이름, 라이센스 등
    		2) 빌드 설정 : 소스, 리소스, 라이프사이클 별 실행한 플러그인 등 빌드와 관련된 설정 
    		3) 빌드 환경 : 사용자 환경 별로 달라질 수 있는 프로파일 정보
    		4) pom 연관 정보 : 의존 프로젝트(모듈), 상위 프로젝트, 포함하고 있는 하위 모듈 
    
    	* XML 
    		Maven은 XML을 기반으로 의존성 (Dependency) 및 디렉토리 구조 관리 등의 기능을 제공하며
    		한동안 많이 사용됐다.
    		그러나 라이브러리가 추가되거나 각각의 라이브러리가 서로 다른 버전의 라이브러리를 
    		참조하는 종속성을 가지고 있을 경우 관리가 어려워지는 문제가 있다. 
    		XML의 특징인 구조화는 장점이지만 문서의 자체가 비대해지는 단점이 있다.
    		& 이에 대한 대안으로 Gradle이 각광받고 있다. 
    			Gradle은 JVM 기반의 빌드 도구이다. Maven과 호환된다. 그 외에도 설정이 간편하고
    			프로젝트 별로 유연하게 설정할 수 있는 장점이 있다. 
    		
    - Gradle : 
    	지금까지의 개발된 빌드 도구들의 장점과 에코시스템을 활용하고 약점은 제거하는 것을 목표로 만듬.
    	* 에코시스템 : 프레임워크, 라이브러리, IDE를 총체적으로 일컫는 말
    		예를 들어 자바 에코시스템으로는 스프링이나 이클립스, 그리고 자바 기반 라이브러리가 있다.
    
    - Gradle VS Maven 비교
    	1) Maven : 
    	  * 빌드를 쉽게
    	  * pom.xml을 이용한 정형화된 빌드 시스템
    	  * 뛰어난 프로젝트 정보 제공
    	  * 개발 가이드 라인 제공
    	  * 새로운 기능을 쉽게 설치하고 업데이트할 수 있음	
    		
    	2) Gradle : 
    	  * 유연한 범용 빌드 도구
    	  * Maven을 사용 할 수 있는 변한 가능 컨벤션 프레임워크
    	  * 멀티프로젝트에 하기 좋음
    	  * Apache lvy에 기반한 강력한 의존성 관리
    	  * Maven과 lvy 레포지토리 완전 지원
    	  * 원격저장소, pom, lvy 파일 없이 연결되는 의존성 관리 지원
    	  * 그루비 문법 사용
    	  * 빌드를 설명하는 풍부한 도메인 모델
    
    - Gradle이 Maven보다 좋은 점 
    	* 시기적으로 Gradle이 늦게 나왔기에 사용성, 성능 등 비교적 스펙이 뛰어나다.
    	* Maven
    	  1) Build라는 동적인 요소를 XML로 정의하기에는 어려운 부분이 많다.
    	  2) 설정 내용이 길어지고 가독성이 떨어짐, 의존관계가 복잡한 프로젝트에 설정하기 부적절
    	* Gradle
    	  1) Groovy를 사용하기 때문에, 동적인 빌드는 Groovy 스크립트로 플러그인을 호출하거나 
    		   직접 코드를 짜면 된다.
    	  2) Configuration Injection 방식을 사용해서 공통 모듈을 상속해서 단점을 커버함.
    	  3) 성정 주입 시 프로젝트의 조건을 체크할 수 있어서 프로젝트별로 주입되는 설정을
    		   다르게 할 수 있다.
    
    - Maven의 장단점
    	* 장점 : 
    		1) 단순성 : 메이븐은 XML 기반의 설정 파일을 사용하며, 간단한 구조를 빠르게 프로
    			 젝트를 설정할 수 있어, 초기 설정이 상대적으로 간단하다.
    		2) 표준화된 빌드 생명주기 : 메이븐은 미리 정의된 빌드 생명주기를 제공하며, 
    			 표준 빌드 명령을 사용하여 일관된 방식으로 빌드할 수 있다.
    		3) 중앙 저장소 : Maven Central Repository를 통해 의존성을 쉽게 관리할 수 있어,
    			 필요한 라이브러리들을 쉽게 가져올 수 있다.
    	- 단점 : 
    		1) 유연성 부족 : Maven은 설정이 간단하고, 표준화된 빌드 생명주기를 제공하지만, 
    			 특정한 요구사항에서 유연성이 부족할 수 있다.
    		2) XML 설정 파일 복잡성 : 대규모 프로젝트의 경우 XML 설정 파일이 복잡해질 수 
    			 있고, 이로 인해 가독성이 떨어지고 유지보수가 떨어질 수 있다.
    
    - Gradle의 장단점
    	* 장점 : 
    		1) 유연성 : Groovy을 사용하여 빌드 스크립트를 작성하므로 Maven보다 훨씬 더
    			 유연하고 복잡한 빌드 요구사항을 다룰 수 있다.
    		2) 성능 : Gradle은 빌드 속도가 빠르고 호율적으로 의존성을 관리한다. 
    			 Gradle은 빌드 시에 필요한 의존성만 다운로드하므로 불필요한 데이터 전송을 줄일수있다.
    		3) 스크립트 언어 활용 : Groovy나 Kotlin과 같은 동적 언어를 사용하기에 간결하고
    			 가독성이 좋은 빌드 스크립트를 작성할 수 있다.
    	* 단점 : 
    		1) 학습 곡선 : Maven에 비해 높은 학습 곡선을 가지고 있기에 처음 사용하는 사용자에게는 
    			 복잡해 보일 수 있다.
    		2) 자바 이외의 언어에 대한 지원 부족 : 
    			 Maven은 자바를 초점으로 하지만, 그래들은 다양한 언어와 플랫폼을 지원하긴 하지만 
    			 자바 이외의 환경에서는 지원이 부족할 수 있다.
    
    - 결론 : 
    	Maven은 간단하고 표준화된 빌드를 원하는 경우에 유용, 
    	Gradle은 유연성이 높고 복잡한 빌드 요구사항을 다루어야할 때 유용.
    	프로젝트의 특성과 개발자의 선호도에 따라 다를 수 있다.
    ```
    
- Build Tool / Dependency Managing Tool
    
    ```java
    - Build Tool, 
    	* "정형화된 작업을 자동화하기 위한 소프트웨어"
    	* "소스 코드에서 실행 가능한 어플리케이션 생성을 자동화하는 프로그램이다."
    	
    	* 정형화된 작업에 사람이 관여하지 않아도 되니 실수가 줄어듦.
    	* 빌드 스크립트는 항상 관리되므로 문서처럼 낡은정보가 될 위험이 줄어듦.
    
    - Build Tool을 사용하는 이유 : 
    	빌드 도구는 개인 프로젝트, 소규모 프로젝트, 특히 대규모 프로젝트의 
    	코드 작성에 많은 도움을 주는 프로그램이다.
    	정형화된 작업을 자동화하여 프로그래밍의 시간을 절약하며 직접 작성하는 코드의 실수를 방지한다.
    
    - 정형화된 작업 : 
    	* 컴파일(Compile) : 소스 코드를 기계어로 변환하여 실행 가능한 코드로 만든다.
    		이는 주로 컴파일 언어에서 사용되며, 예를 들어 java의 경우 java 소스 코드를 바이트 코드로
    		변환하는 작업이 여기에 해당된다.
    
    	* 테스트(Test) : 단위 테스트, 통합 테스트, 시스템 테스트 등을 실행하여 소프트웨어의 품질을 검증한다.
    
    	* 패키징(Package) : 빌드된 결과물을 배포 가능한 형태로 패키징한다.
    		이는 실행 가능한 파일, 라이브러리, 아티펙트 등을 생성하는 단계를 포함한다.
    
    	* 의존성 관리(Dependency Management) : 프로젝트가 필요로 하는 외부 라이브러리나 모듈을 관리하고 
    		필요한 경우 이를 다운로드하거나 업데이트한다. 
    
    	* 문서 생성(Documentation) : 코드 문서, API 문서 등을 생성하고 관리한다.
    		이는 주로 자동화된 도구를 사용하여 API 문서를 생성하는 작업이다.
    	
    	* 정적 분석(Static Analysis) : 소스 코드의 정적 분석을 수행하여 코드 품질이나 잠재적인 버그를 
    		식별하는 작업을 말한다.
    	
    	* 배포(Deployment) : 빌드된 소프트웨어를 서버에 배포하거나 패키지를 저장소에 업로드하는 등의 작업을 수행
    		
    	* 환경 구성(Configuration) : 프로젝트의 환경 설정, 설정 파일 등을 구성하고 관리한다.
    		
    
    - Dependency Managing Tool(의존성 관리) : 
    	* "외부 라이브러리, 모듈, 패키지 등과 같은 의존성을 효과적으로 관리하고 가져오는 것이 주 목적이다."
    
    	* 프로젝트에 필요로하는 의존성들을 선언하고 이들을 가져오고, 사용하기 위한 자동화된 기술
    
    	* 개발자들은 내가 직접 작성한 순수한 코드로만 프로젝트를 만드는 것은 매우 힘든일이다.
    	  대부분의 모든 개발자들은 그때그때 필요한 프로젝트들을 여기저기 가져오며 사용한다. 
    		이런저런 프로젝트에 필요로하는 의존성을 선언하고 이들을 사용하기 위한 자동화 기술이다. 
    
    - 의존성 관리 도구 주요 기능 : 
    	* 의존성 정의(Dependency Definition) : 프로젝트에 사용할 외부 라이브러리나 모듈을 명시적으로 정의하는 단계
    		프로젝트 구성 파일에 필요한 의존성 정보를 추가하여 프로젝트가 어떤 외부 라이브러리에 의존하는지를 명시한다.
    
    	* 버전 관리(Version Management) : 각 의존성에 대해 어떤 버전을 사용할 지를 결정하는 것
    		버전 관리를 통해 호환성을 유지하고 안정적인 프로젝트를 구성할 수 있다. 
    
    	* 자동으로 의존성 해결(Automatic Dependency Resolution) : 정의된 의존성 정보를 기반으로 의존성 관리 도구가
    		필요한 라이브러리를 자동으로 가져와 프로젝트를 통하는 과정이다.
    		이를 통해 중복 다운로드를 방지하고 효율적인 의존성 해결이 가능하다.
    ```


- Build Tool
    빌드 도구(Build Tool) : 
    	* 프로젝트내의 소스코드로부터 실행 가능한 어플리케이션을 자동으로 제작해주는
    		프로그램 
    	* 왜 사용하는가? : 
    		소규모 프로젝트의 경우 의존성 라이브러리, API 등 직접 설치하고 소스 코드를
    		빌드하고 실행하는 것에는 문제가 되지않는다. 하지만 프로젝트 규모가 커지고
    		소스 코드가 방대해지면 각각의 라이브러리를 설치하고 사용하는 것은 프로젝트의
    		수행함에 있어 불필요한 비중을 차지하게 만든다. 따라서 개발자가 이러한
    		불필요한 업무를 하지 않고 자동으로 의존성을 관리하고 소스를 빌드할 수 있는
    		도구를 사용하는 것이다.
    	* 빌드 도구의 수행 목록
    		1. 의존성 있는 라이브러 패키지, 파일 등 다운로드하
    		2. 바이너리 코드로 컴파일해서 패키징하기
    		3. 프로그램 테스트
    		4. 배포하
    	* 빌드 도구 : Maven, Gradle, Gulp, Webpack
